Few-shot vs fine-tuning decision framework with cost calculator and interactive decision tree

It's not binary. There are 4 options on a spectrum from "zero infrastructure" to "full model ownership."
Zero infrastructure Full model ownership
1. Zero-shot / System prompt
Deploy: today
Instructions only, no examples. Cost: just the tokens in your prompt.
✓ Zero setup, zero training data
✓ Change behavior by editing a string
✓ Works for standard tasks
✗ Inconsistent on novel formats
✗ Can't teach custom domain logic
✗ Prompt must fit in every call
2. Few-shot prompting
Deploy: hours
2–10 examples prepended to every API call. Examples teach format, tone, and domain logic.
✓ No training, instant iteration
✓ Can swap examples at runtime
✓ Great for low-to-medium volume
✗ Examples cost tokens every call
✗ High volume = very expensive
✗ Context window fills up faster
3. Fine-tuning
Deploy: days–weeks
Train a model on hundreds–thousands of examples. Behavior is baked into weights — no examples needed at inference.
✓ Shorter prompts → lower cost at scale
✓ Consistent format and tone
✓ Can teach custom knowledge/style
✗ Needs 100s–1000s of labeled examples
✗ Retraining cost when task changes
✗ Slower to iterate
4. Train from scratch / RLHF
Deploy: months
Full pre-training or reinforcement learning from human feedback. Used by AI labs, not application engineers.
✓ Complete model control    ✗ Requires millions of examples, GPU clusters, ML research team
As an AI engineer (not an AI researcher), you'll spend 90% of your time in options 1–3. Option 4 is for teams like Anthropic, OpenAI, and Google.

Few-shot wins here
Prototyping & early product — you're still figuring out what the task even is. Changing a few-shot example takes seconds; retraining takes days.
Low-to-medium volume — under ~50k calls/day, the token cost of examples is cheaper than the engineering + training cost of fine-tuning.
Multiple task variants — if you have 10 slightly different customer segments needing slightly different outputs, you can swap few-shot examples per segment without maintaining 10 fine-tuned models.
No labeled data yet — you can hand-write 5 good examples in an afternoon. Collecting 500 labeled examples for fine-tuning takes weeks.
Tasks that change — A/B testing prompts, seasonal adjustments, regulatory changes. Few-shot is a config change; fine-tuning is a retraining job.
Fine-tuning wins here
High volume, stable task — 500k+ calls/day on a task that won't change. At that scale, eliminating 500 tokens of examples per call saves tens of thousands of dollars monthly.
Strict output format — code generation for a specific framework, medical record templates, legal document structures. Fine-tuning bakes the format in so reliably you rarely need output validation.
Proprietary tone/style — teaching the model to write like your brand voice, in your internal jargon, with your company-specific terminology. Impossible to do reliably with prompts alone.
Latency-sensitive — fewer input tokens = faster time-to-first-token. For real-time applications, removing 500 tokens of examples meaningfully improves p95 latency.
Data privacy — if your examples contain sensitive customer data, you can't put them in every API call. Fine-tuning keeps examples in the training run, not runtime prompts.
The hybrid pattern (what most production systems use)
Fine-tune for format + tone, then use a short system prompt for task-specific instructions. Best of both worlds.

The 4 interview questions you will get asked
Q: "When would you choose few-shot over fine-tuning?"
"I default to few-shot when the task is still being defined, volume is under ~50k calls/day, or I need to iterate quickly. Fine-tuning requires labeled data and a retraining pipeline — that overhead isn't worth it until the task is stable and the volume justifies the savings."
Q: "What are the risks of fine-tuning?"
"Three main risks: (1) catastrophic forgetting — fine-tuning on a narrow task can degrade the model's general capabilities. (2) overfitting — with fewer than ~500 examples, the model memorizes rather than generalizes. (3) distribution shift — if production inputs look different from training examples, performance degrades silently. You need an eval pipeline that catches this."
Q: "How do you know when fine-tuning has worked?"
"You need a held-out eval set — typically 10–20% of your labeled data kept separate from training. You measure: (1) format compliance rate (does it always output valid JSON?), (2) task accuracy against ground truth, (3) edge case handling. Fine-tuning is successful when it matches or beats the few-shot baseline on all three, with shorter prompts."
Q: "What's LoRA and when would you use it?"
"LoRA (Low-Rank Adaptation) is a parameter-efficient fine-tuning technique. Instead of updating all model weights, it trains small adapter matrices that are added to specific layers. This means: 10–100x fewer trainable parameters, fits on a single GPU, and trains in hours not days. It's the standard approach for fine-tuning open-source models like Llama or Mistral when you're self-hosting. The trade-off vs full fine-tuning is slightly lower max quality, but for most production tasks the difference is negligible."
The answer that impresses senior interviewers
"In practice I don't think of it as few-shot OR fine-tuning — I think of it as a progression. I start with zero-shot to validate the task is solvable. I add few-shot examples to hit quality targets. Once the task is stable and volume grows, I collect the few-shot examples + model outputs as training data and fine-tune. That way the fine-tuning data is already validated because it came from a working few-shot system."
