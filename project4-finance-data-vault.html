Data Vault 2.0 Theory

The Complete Data Vault Mental Model

Before writing a single line of code, you must understand the three building blocks. Every table in a Data Vault is one of exactly three types.

🏛
Dan Linstedt's core insight (creator of Data Vault): "Separate what you know (business keys) from how things relate (links) from what you know about them (attributes). Business keys are stable. Relationships change. Attributes change the most." This separation is why Data Vault can accommodate schema changes, new sources, and regulatory audits — without breaking existing pipelines.
H
🟡 Hubs — The "What Exists"
One row per unique business entity. No history. No attributes.
▼

A Hub captures the existence of a business entity — nothing more. It stores the business key (account number, customer ID, trade ID), a hash key (surrogate key), when it was first seen, and which source system reported it. No attributes (no name, no balance) — those go in Satellites.

Rules:

  • Primary key is a hash of the business key (MD5 or SHA-256)
  • Never updated — new business keys get new rows, existing rows never change
  • No foreign keys — Hubs reference nothing
  • Always includes: load_date, record_source, hash_key, business_key
Hub — Example Structure
-- Hub_Account: every unique account we've ever seen
CREATE TABLE raw_vault.hub_account (
    account_hash_key  CHAR(32) NOT NULL,   -- MD5(account_id)
    load_date         TIMESTAMP NOT NULL,  -- When first loaded
    record_source     VARCHAR(100) NOT NULL,-- "GL_SYSTEM" / "TRADING_SYS"
    account_id        VARCHAR(50) NOT NULL, -- The actual business key
    PRIMARY KEY (account_hash_key)
);
-- NO name, NO balance, NO type — those go in SAT_ACCOUNT_DETAILS
L
🟣 Links — The "How Things Relate"
Relationships between Hubs. Every join lives here.
▼

A Link records a relationship between two or more Hubs — like a transaction that connects an account to a counterparty. Links are immutable like Hubs — the relationship itself is recorded, but no attributes about it.

Rules:

  • Contains foreign hash keys to parent Hubs
  • Primary key is a hash of all the business keys it links
  • Never updated — new relationships get new rows
  • No business attributes (transaction amount goes in a Satellite)
  • Always includes: link_hash_key, load_date, record_source, + FK hash keys
Link — Example Structure
-- Link_Transaction: a transaction connecting account to counterparty
CREATE TABLE raw_vault.link_transaction (
    transaction_hash_key  CHAR(32) NOT NULL, -- MD5(transaction_id)
    load_date             TIMESTAMP NOT NULL,
    record_source         VARCHAR(100) NOT NULL,
    account_hash_key      CHAR(32) NOT NULL,  -- FK → Hub_Account
    counterparty_hash_key CHAR(32) NOT NULL,  -- FK → Hub_Counterparty
    instrument_hash_key   CHAR(32) NOT NULL,  -- FK → Hub_Instrument
    transaction_id        VARCHAR(50) NOT NULL, -- Business key of the link
    PRIMARY KEY (transaction_hash_key)
);
-- NO amount, NO trade_date — those go in SAT_TRANSACTION_DETAILS
S
🟢 Satellites — The "What We Know About Them"
All descriptive attributes + full history. The workhorse of Data Vault.
▼

Satellites store all the attributes of a Hub or Link — and every time anything changes, a new row is inserted. This is full history by design, not SCD Type 2 — there's no closing old records, just appending new ones with a load_date.

Rules:

  • Attached to exactly one Hub or Link
  • Primary key = parent_hash_key + load_date (composite)
  • Insert-only — never update or delete
  • Include a hash_diff column: MD5 of all attributes. Only insert new row if hash_diff changed (deduplication)
  • May split by rate-of-change: SAT_ACCOUNT_STATIC (rare changes) + SAT_ACCOUNT_BALANCE (daily changes)
Satellite — Example Structure
-- Sat_Account_Details: attributes of an account (version history)
CREATE TABLE raw_vault.sat_account_details (
    account_hash_key  CHAR(32) NOT NULL,   -- FK → Hub_Account
    load_date         TIMESTAMP NOT NULL,  -- When THIS version was loaded
    load_end_date     TIMESTAMP,             -- NULL = current version
    record_source     VARCHAR(100) NOT NULL,
    hash_diff         CHAR(32) NOT NULL,   -- MD5 of all attributes (for dedup)
    -- The actual attributes:
    account_name      VARCHAR(200),
    account_type      VARCHAR(50),
    account_status    VARCHAR(20),
    credit_rating     VARCHAR(10),
    PRIMARY KEY (account_hash_key, load_date)
);
-- To get current version: WHERE load_end_date IS NULL
-- To get as-of date: WHERE load_date <= target_date
--                       AND (load_end_date IS NULL OR load_end_date > target_date)

The Three-Layer Architecture

Layer 1: Raw Vault ← Everything you loaded, exactly as it came in
  Hubs + Links + Satellites (no transformations, no business rules)
  Purpose: immutable audit record — "what did source systems tell us?"
  Never modified after loading

Layer 2: Business Vault ← Cleaned, standardized, business rules applied
  Computed Satellites + Bridge tables
  Purpose: "what do we know after applying business interpretation?"
  e.g., "amount in USD after FX conversion"

Layer 3: Information Marts ← Star/Snowflake schema for end users
  PIT (Point-In-Time) tables + dimensional views
  Purpose: analyst-friendly query performance
  "What was the account balance on March 15th?"
Why Data Vault for Finance/Audit

SOX Compliance + Data Vault = Perfect Match

📋
SOX (Sarbanes-Oxley Act, 2002): US law requiring public companies to maintain accurate financial records with full audit trails. Section 302: CEOs must certify financial statements are accurate. Section 404: internal controls over financial reporting must be documented and tested. Violations = criminal charges, SEC investigation, delisting. This is why financial companies take data lineage so seriously.

With a traditional overwriting DW, that's impossible. When NOT to Use Data Vault

⚠️
Data Vault adds complexity — 3x more tables than star schema, slower raw query performance, steeper learning curve. Don't use it for: small teams without DV expertise, simple reporting use cases, startups that need speed over auditability, or systems where history truly doesn't matter. Use it for: regulated industries (banking, healthcare, insurance, government), large multi-source enterprise warehouses, anywhere full audit lineage is a legal or regulatory requirement.

Why MD5 hash of business key as PK? The hash is always 32 chars regardless of business key length. Joins on fixed-length CHAR(32) are faster than VARCHAR joins. Performance Layer

PIT Tables + Bridge Tables

The Raw Vault is auditable but slow to query — too many joins and time-range conditions. PIT and Bridge tables are the performance layer that makes the vault queryable at analyst speed.

PIT
Point-In-Time (PIT) Tables
Pre-computed "which satellite version was current at each snapshot date?"
▼

Without PIT, to get account state on March 15th you'd write: WHERE sat.load_date <= '2024-03-15' AND (sat.load_end_date IS NULL OR sat.load_end_date > '2024-03-15') for every satellite in every query. That's expensive. A PIT table pre-computes the answer for each snapshot date (daily or monthly).

SQL — PIT Table Structure
-- PIT_Account: for each account, on each snapshot date,
-- which version of each satellite was current?
CREATE TABLE business_vault.pit_account (
    account_hash_key        CHAR(32) NOT NULL,
    snapshot_date           DATE NOT NULL,

    -- For each satellite: the load_date of the current version as of snapshot_date
    sat_account_details_ldts TIMESTAMP,   -- NULL = no record existed yet
    sat_account_balance_ldts TIMESTAMP,

    PRIMARY KEY (account_hash_key, snapshot_date)
);

-- Now to get account state on March 15th:
SELECT h.account_id, d.account_name, b.balance_amount
FROM business_vault.pit_account p
JOIN raw_vault.hub_account h ON p.account_hash_key = h.account_hash_key
JOIN raw_vault.sat_account_details d
  ON p.account_hash_key = d.account_hash_key
 AND p.sat_account_details_ldts = d.load_date  ← simple equality!
JOIN raw_vault.sat_account_balance b
  ON p.account_hash_key = b.account_hash_key
 AND p.sat_account_balance_ldts = b.load_date   ← simple equality!
WHERE p.snapshot_date = '2024-03-15';
-- No complex date range conditions — PIT pre-computed the answer
BRG
Bridge Tables
Pre-computed join paths across multiple Links for fast cross-vault queries
▼

Some queries need to cross multiple Links: "show me all transactions for accounts managed by employees in the London office." Without a Bridge, this requires joining Hub_Account → Link_AccountEmployee → Hub_Employee → Sat_Employee_Details. A Bridge pre-computes this path.

SQL — Bridge Table Structure
-- Bridge_AccountTransactions: pre-computed path from account through transactions
CREATE TABLE business_vault.bridge_account_transactions (
    bridge_hash_key       CHAR(32) NOT NULL,
    snapshot_date         DATE NOT NULL,
    account_hash_key      CHAR(32) NOT NULL,
    transaction_hash_key  CHAR(32) NOT NULL,
    counterparty_hash_key CHAR(32) NOT NULL,
    instrument_hash_key   CHAR(32) NOT NULL,
    PRIMARY KEY (bridge_hash_key, snapshot_date)
);

-- Now a complex cross-vault query becomes a simple join:
SELECT a.account_id, t.notional_amount, c.legal_name
FROM business_vault.bridge_account_transactions b
JOIN raw_vault.hub_account ha   ON b.account_hash_key = ha.account_hash_key
JOIN raw_vault.sat_account_details a
    ON b.account_hash_key = a.account_hash_key
JOIN raw_vault.sat_transaction_details t
    ON b.transaction_hash_key = t.transaction_hash_key
JOIN raw_vault.sat_counterparty_details c
    ON b.counterparty_hash_key = c.counterparty_hash_key
WHERE b.snapshot_date = '2024-03-15';

Step 3 · Raw Vault Load

Loading the Raw Vault with PySpark

The Raw Vault loader applies the most important Data Vault rule: never transform, just load. Hash the business keys, add metadata, insert new records only.

🏛
The Golden Rule of Raw Vault loading: No business logic. No FX conversion. No name standardization. No deduplication beyond hash_diff. The raw vault is a legal record — "this is what source systems told us, exactly, with timestamps." Business interpretation happens in the Business Vault layer.
scripts/hash_utils.py — reusable hashing functions
"""
Data Vault hashing utilities.
Consistent hashing is CRITICAL — same business key from different sources
must produce the same hash for entity resolution to work.
"""
from pyspark.sql import functions as F
from pyspark.sql import DataFrame

def dv_hash_key(col_name: str) -> F.Column:
    """
    Compute Data Vault hash key from a business key column.
    Rules: UPPERCASE, trim whitespace, MD5.
    CONSISTENCY IS EVERYTHING — deviation breaks entity resolution.
    """
    return F.md5(F.upper(F.trim(F.col(col_name))))

def dv_hash_key_multi(*col_names) -> F.Column:
    """
    Hash multiple columns (for Link keys or multi-part business keys).
    Concatenate with || separator before hashing.
    """
    parts = [F.upper(F.trim(F.col(c))) for c in col_names]
    concat = parts[0]
    for p in parts[1:]:
        concat = F.concat(concat, F.lit("||"), p)
    return F.md5(concat)

def dv_hash_diff(*col_names) -> F.Column:
    """
    Hash all satellite attribute columns for change detection.
    Only insert a new satellite row if hash_diff differs from latest version.
    """
    concat_cols = [F.coalesce(F.col(c).cast("string"), F.lit("~NULL~"))
                   for c in col_names]
    concat_expr = concat_cols[0]
    for p in concat_cols[1:]:
        concat_expr = F.concat(concat_expr, F.lit("|"), p)
    return F.md5(concat_expr)

def add_dv_metadata(df: DataFrame, record_source: str) -> DataFrame:
    """Add standard Data Vault metadata columns to any DataFrame."""
    return (df
        .withColumn("load_date",     F.current_timestamp())
        .withColumn("record_source", F.lit(record_source))
    )

def insert_new_only(new_df: DataFrame, existing_df, key_col: str, out_path: str):
    """Insert-only pattern: only write rows with new hash keys."""
    if existing_df is not None:
        existing_keys = existing_df.select(key_col)
        new_only = new_df.join(existing_keys, on=key_col, how="left_anti")
    else:
        new_only = new_df
    cnt = new_only.count()
    if cnt > 0:
        new_only.coalesce(1).write.mode("append").parquet(out_path)
        log.info(f"  Inserted {cnt} new rows → {out_path}")
    else:
        log.info(f"  No new rows for {out_path}")
    return cnt

Step 4 · Business Vault

Business Vault — Apply Business Rules

The Business Vault sits on top of the Raw Vault and applies business interpretation: FX conversion, name standardization, derived metrics. Everything is computed, nothing destroys the raw record.

💡
Business Vault vs Raw Vault: Raw Vault = "what did source systems tell us?" Business Vault = "what does that mean in our business context?" FX conversion, USD normalization, credit rating standardization — these go in the Business Vault as Computed Satellites, not in the Raw Vault. This separation means if your FX rates were wrong, you can recalculate the Business Vault from the unchanged Raw Vault.

# Simplified FX rates (in production: load from market data feed)
FX_RATES_TO_USD = {
    "USD": 1.0000, "EUR": 1.0850, "GBP": 1.2700,
    "JPY": 0.0067, "CHF": 1.1200
}

def build_computed_sat_transaction_usd():
    """
    Computed Satellite: Transaction amounts normalized to USD.
    This is a BUSINESS RULE (FX conversion) applied on top of Raw Vault.
    If FX rates change, we can recompute this satellite from raw data.
    """
    log.info("Building CSAT_Transaction_USD...")
    sat = spark.read.parquet(f"{VAULT}/satellites/sat_transaction_details")

    # Build FX mapping expression
    fx_expr = F.lit(1.0)
    for currency, rate in FX_RATES_TO_USD.items():
        fx_expr = F.when(F.col("currency") == currency, F.lit(rate)).otherwise(fx_expr)

    csat = (sat
        .withColumn("fx_rate_to_usd",      fx_expr)
        .withColumn("notional_amount_usd",
            F.round(F.col("notional_amount").cast(DoubleType()) * fx_expr, 2))
        .withColumn("price_usd",
            F.round(F.col("price").cast(DoubleType()) * fx_expr, 4))
        .withColumn("bv_load_date",         F.current_timestamp())
        .withColumn("bv_record_source",     F.lit("BUSINESS_VAULT_FX"))
        .select("transaction_hash_key", "bv_load_date", "bv_record_source",
                "fx_rate_to_usd", "notional_amount_usd", "price_usd", "currency")
    )
    csat.coalesce(1).write.mode("overwrite").parquet(f"{BV}/computed_sats/csat_transaction_usd")
    log.info(f"  → {csat.count()} computed sat rows written")

def build_pit_account():
    """
    Build Point-In-Time table for accounts.
    For each account + each snapshot date: which satellite version was current?
    """
    log.info("Building PIT_Account...")
    hub      = spark.read.parquet(f"{VAULT}/hubs/hub_account")
    sat_det  = spark.read.parquet(f"{VAULT}/satellites/sat_account_details") if __import__('os').path.exists(f"{VAULT}/satellites/sat_account_details") else None

    # Generate daily snapshot dates for 2024
    snapshots = spark.sql("""
        SELECT explode(sequence(to_date('2024-01-01'),
               to_date('2024-12-31'), interval 1 day)) AS snapshot_date
    """)

    # Cross join accounts with snapshot dates
    pit_base = hub.crossJoin(snapshots).select("account_hash_key","snapshot_date")

    if sat_det is not None:
        # For each account + snapshot: find the latest satellite load_date <= snapshot_date
        sat_snapshot = (sat_det
            .join(snapshots, sat_det.load_date.cast("date") <= snapshots.snapshot_date, "inner")
            .groupBy("account_hash_key", "snapshot_date")
            .agg(F.max("load_date").alias("sat_account_details_ldts"))
        )
        pit = pit_base.join(sat_snapshot, ["account_hash_key","snapshot_date"], "left")
    else:
        pit = pit_base.withColumn("sat_account_details_ldts", F.lit(None))

    pit.write.mode("overwrite").partitionBy("snapshot_date").parquet(f"{BV}/pit/pit_account")
    log.info(f"  → PIT_Account built")

def main():
    log.info("=== Business Vault Build Started ===")
    build_computed_sat_transaction_usd()
    build_pit_account()
    log.info("=== Business Vault Complete ===")

if __name__ == "__main__":
    main()

def build_fact_trading_activity():
    """
    Build the main fact table for trading activity.
    Joins Hub → Link → Satellite (latest version via is_current pattern).
    USD-normalized amounts from Business Vault computed satellite.
    """
    log.info("Building fact_trading_activity...")

    # Load vault components
    hub_acct  = spark.read.parquet(f"{VAULT}/hubs/hub_account")

            "year", "month", "mart_load_ts"

SQL — SOX Audit Trail: GL Journal Entry Lineage
-- Full audit trail: for every GL journal entry, show all versions
-- and when it was loaded (proving no retroactive manipulation)
WITH journal_versions AS (
    SELECT
        lj.journal_id,
        lj.account_hash_key,
        lj.gl_hash_key,
        sj.load_date                        AS version_loaded_at,
        sj.record_source,
        sj.posting_date,
        sj.fiscal_period,
        sj.debit_amount,
        sj.credit_amount,
        sj.description,
        sj.approved_by,
        sj.hash_diff,
        ROW_NUMBER() OVER (
            PARTITION BY lj.journal_id
            ORDER BY sj.load_date
        )                                   AS version_number,
        LAG(sj.hash_diff) OVER (
            PARTITION BY lj.journal_id
            ORDER BY sj.load_date
        )                                   AS previous_hash_diff
    FROM raw_vault.link_gl_journal_entry lj
    JOIN raw_vault.sat_gl_journal_details sj
        ON lj.journal_hash_key = sj.journal_hash_key
)
SELECT
    journal_id,
    version_number,
    version_loaded_at,
    record_source,
    posting_date,
    fiscal_period,
    debit_amount,
    credit_amount,
    approved_by,
    CASE
        WHEN version_number = 1             THEN 'ORIGINAL'
        WHEN previous_hash_diff != hash_diff THEN 'MODIFIED'
        ELSE                                      'UNCHANGED RELOAD'
    END                                     AS change_type
FROM journal_versions
ORDER BY journal_id, version_number;

SQL — Counterparty Exposure Report (Regulatory)
WITH active_exposures AS (
    SELECT
        lt.counterparty_hash_key,
        SUM(cs.notional_amount_usd)          AS total_exposure_usd,
        COUNT(DISTINCT lt.transaction_id)    AS trade_count,
        COUNT(DISTINCT lt.instrument_hash_key)AS instrument_count,
        MIN(std.trade_date)                  AS earliest_trade,
        MAX(std.settlement_date)             AS latest_settlement
    FROM raw_vault.link_transaction lt
    JOIN raw_vault.sat_transaction_details std
        ON lt.transaction_hash_key = std.transaction_hash_key
    JOIN business_vault.csat_transaction_usd cs
        ON lt.transaction_hash_key = cs.transaction_hash_key
    WHERE std.status != 'cancelled'
      AND std.settlement_date >= CURRENT_DATE   -- Active/unsettled trades only
    GROUP BY lt.counterparty_hash_key
),
counterparty_totals AS (
    SELECT SUM(total_exposure_usd) AS firm_total_exposure
    FROM active_exposures
)
SELECT
    scp.legal_name,
    scp.lei_code,
    scp.credit_rating,
    scp.country_of_incorporation,
    ROUND(ae.total_exposure_usd, 0)       AS exposure_usd,
    ae.trade_count,
    ae.instrument_count,
    ROUND(ae.total_exposure_usd
        / ct.firm_total_exposure * 100, 2) AS pct_of_total_exposure,
    DENSE_RANK() OVER (
        ORDER BY ae.total_exposure_usd DESC
    )                                      AS exposure_rank,
    -- Regulatory flag: concentration > 10% of total = reportable
    CASE
        WHEN ae.total_exposure_usd / ct.firm_total_exposure > 0.10
        THEN 'CONCENTRATION RISK — Report to Regulator'
        ELSE 'Within Limits'
    END                                    AS regulatory_status
FROM active_exposures ae
JOIN raw_vault.hub_counterparty hc
    ON ae.counterparty_hash_key = hc.counterparty_hash_key
JOIN raw_vault.sat_counterparty_details scp
    ON hc.counterparty_hash_key = scp.counterparty_hash_key
CROSS JOIN counterparty_totals ct
ORDER BY exposure_usd DESC;

Q1Data Vault Explain Data Vault 2.0 in plain English. What problem does it solve that star schema can't?
Model Answer

Data Vault 2.0 is a data modeling methodology designed for regulated, multi-source enterprise environments where full audit history and data lineage are legal requirements. A star schema is optimized for query performance — dimensions and facts are structured for analysts. But it overwrites history (Type 1) or requires careful SCD management (Type 2), and it can't easily handle data from multiple source systems with different schemas for the same entity. Data Vault solves three problems star schema can't. First, complete history: it's insert-only — nothing is ever updated or deleted, so you can reconstruct any past state. Second, multi-source integration: business keys are hashed consistently, so the same customer appearing in CRM and billing gets the same hash key and resolves to one entity regardless of source. Third, audit lineage: every row has load_date and record_source, so you can trace any number on any report back to the exact source file and timestamp it came from. For SOX compliance, that lineage is a legal requirement.

Q2Hub What is a Hub and what are the strict rules for building one?
Model Answer

A Hub is a table that captures the existence of a unique business entity — one row per unique business key, ever. Hub_Account has one row per account ID. That's it. No attributes about the account (no name, no balance, no type) — those go in Satellites attached to the Hub. The strict rules: first, the primary key must be a hash of the business key — MD5 or SHA-256. The hash must be computed consistently — uppercase, trim whitespace, same encoding — because the same account from two different source systems must produce the same hash for entity resolution to work. Second, Hubs are insert-only — once an account appears, that row never changes. A new load only adds new business keys. Third, Hubs contain only: hash_key, load_date, record_source, and business_key. Nothing else. No foreign keys — Hubs reference nothing. The conceptual purpose: Hubs answer "what business entities have we ever seen?" without saying anything about what we know about them.

Q3Satellite What is hash_diff in a Satellite and why is it critical?
Model Answer

Hash_diff is an MD5 hash of all the attribute columns in a Satellite row. Its purpose is change detection and deduplication. Every time a source system sends data, we compute the hash_diff for the incoming record and compare it to the latest existing hash_diff for that entity. If they're the same, nothing changed — we don't insert a new row. If they differ, something changed — insert a new row with the new values and current load_date. Without hash_diff, a daily full load from a source system would insert millions of duplicate rows with no real changes — the satellite would balloon with identical versions. Hash_diff keeps satellites lean: new rows only appear when attributes actually changed. This is critical for performance, storage, and maintaining clean history. One important implementation detail: you must handle NULLs consistently when computing hash_diff — I replace NULL with a sentinel value like '~NULL~' before hashing, because NULL != NULL in SQL and inconsistent NULL handling creates false "change detected" rows.

Q4Data Vault What is the difference between a Raw Vault and a Business Vault?
Model Answer

The Raw Vault is the immutable legal record — it stores exactly what source systems told us, with no transformations, no business rules applied. A Raw Vault Satellite for a transaction has the raw notional amount and raw currency code, exactly as they came from the trading system. The Raw Vault answers "what did source systems tell us?" The Business Vault applies business interpretation on top of the Raw Vault. It adds Computed Satellites — derived tables built by applying business rules: FX conversion to normalize all amounts to USD, name standardization from multiple formats, credit rating mapping from different agency scales. The Business Vault answers "what does that data mean in our business context?" The separation is powerful for auditability: if our FX rates were wrong last quarter, we can fix the rates in the Business Vault and recompute — the Raw Vault remains unchanged, preserving the original source data. Auditors can see both: what the source said, and what business rules were applied to arrive at the reported figure.

Q5Data Vault What is a Point-in-Time (PIT) table and why is it needed?
Model Answer

Without PIT tables, querying a Data Vault at a specific historical date requires complex date-range conditions on every satellite: WHERE load_date <= target_date AND (load_end_date IS NULL OR load_end_date > target_date). This is slow and error-prone, especially when joining multiple satellites. A PIT table pre-computes the answer for each snapshot date. For each account on each snapshot date — say, daily for 2024 — it stores the exact load_date of the satellite version that was current on that date. This converts complex date-range joins into simple equality joins, which are orders of magnitude faster. It also handles NULL gracefully: if an account didn't exist on a snapshot date, the PIT row has NULL for the satellite load_date, and the join naturally returns no row. PIT tables are rebuilt nightly as new satellite versions arrive. They're the bridge between the Raw Vault (optimized for auditability) and the Information Mart (optimized for query performance). Without PIT, analysts would hit the Raw Vault directly with terrible query performance.

Q6Link How do you handle a many-to-many relationship that changes over time in Data Vault?
Model Answer

A Link records the existence of a relationship — immutably. If a trader is assigned to account A, that's a row in Link_AccountEmployee. The Link itself never changes — it just says "this relationship existed." But what if the trader is reassigned? You might think you need to delete the old Link row. You don't. Instead, you add a Satellite on the Link — Sat_AccountEmployee_Status — that tracks whether the relationship is active. The Satellite has: link_hash_key, load_date, is_active (boolean). When the trader is reassigned: the old Link row remains unchanged (permanent record). A new Satellite row is inserted with is_active=False. A new Link row is created for the new assignment, with a Satellite row where is_active=True. This approach preserves the complete history: when was the relationship established, when did it end, who was the account managed by on any given date. The Link captures "this relationship existed at some point" and the Satellite captures "was it active on any given date?"

Q7SOX / Audit What is SOX compliance and what does it require from a data engineering perspective?
Model Answer

SOX — Sarbanes-Oxley Act — is a 2002 US law requiring public companies to maintain accurate financial records with documented internal controls. It was passed after Enron and WorldCom accounting scandals. From a data engineering perspective, SOX has five key requirements. First, complete audit trail: any number in a financial report must be traceable back to its source transaction — which system, which file, which timestamp. Data Vault's record_source and load_date on every row satisfy this. Second, point-in-time reproducibility: auditors can ask "show me the exact report you filed on March 15th" — the system must reproduce it exactly. PIT tables enable this. Third, no retroactive modification: historical data cannot be overwritten. Data Vault's insert-only architecture satisfies this. Fourth, change history: every change to financial data must be captured with when it changed and who authorized it. Satellites record every version. Fifth, cross-system reconciliation: data from multiple systems must reconcile. Hub entity resolution connects the same account across GL, trading, and loan systems. Violation of SOX is a criminal offense — executives can face personal liability, which is why financial firms take data lineage this seriously.

Q8Data Vault How do you handle late-arriving data in a Data Vault?
Model Answer

Data Vault handles late-arriving data more elegantly than star schema because of its insert-only design. If a transaction from January arrives in March, we simply insert it into the Link and Satellite with the March load_date — but the actual business date (trade_date) is stored in the Satellite attributes, not the load_date. Load_date = when we learned about it. Business date = when it happened. These are always stored separately in Data Vault, which is a design requirement. For PIT tables, we need to rebuild the affected snapshots — the January snapshot needs to be recalculated because new data arrived that should have been reflected. We run a PIT rebuild for any snapshot_date where the new satellite data falls in range. For the Information Mart, the new transaction will appear in the next mart rebuild. For auditors, this is actually preferable to star schema: the audit trail shows "this transaction was reported to us on March 15th, but it occurred on January 5th" — both dates preserved, no manipulation, clear lineage. Late-arriving data with documented arrival timestamps is an acceptable audit pattern; silent overwrites are not.

Q9Architecture When would you choose EMR over Glue for a PySpark job?
Model Answer

Both run PySpark — the choice is operational. Glue is serverless: no cluster to manage, you pay per DPU-second, it auto-scales, and it integrates natively with S3 and Glue Data Catalog. It's the right default for most ETL jobs. I'd choose EMR when: first, I need fine-grained Spark configuration tuning — specific shuffle partition counts, custom memory settings, Kryo serialization — Glue exposes a subset of Spark settings while EMR gives full control. Second, I need spot instances — EMR can run core nodes on spot instances at 70% cost savings for long-running jobs. For a nightly Data Vault load of $2B portfolio data, that's significant. Third, I need a specific library or Spark version Glue doesn't support — EMR lets you install anything via bootstrap actions. Fourth, for very long-running jobs (8+ hours) where EMR's predictable cluster cost is cheaper than Glue's per-DPU pricing. For this project, I chose EMR because the Data Vault load involves complex PySpark joins across multiple vault layers and I needed tuned Spark configurations for the hash join operations.

Q10Data Vault How does multi-source entity resolution work in Data Vault?
Model Answer

Multi-source entity resolution is one of the strongest capabilities of Data Vault. Consider an account "ACC-12345" that appears in the GL system as "ACC-12345" and in the trading system as "12345" — slightly different formats. Without standardization, these would be treated as two different entities. In Data Vault, I apply consistent hash key computation before loading Hubs: uppercase, trim whitespace, remove special characters, then MD5. With the right normalization rules, "ACC-12345" and " 12345 " both map to the same hash key. In Hub_Account, there's one row. The record_source column shows it was first seen from GL_SYSTEM. When the trading system later sends "12345", the normalized hash matches — no new Hub row is inserted, but the existing row's record_source could be updated or a second Satellite entry added noting the second source. For entities that genuinely need cross-system resolution (same customer in CRM and billing with different IDs), a Reference Table maps the different business keys to a master key before hashing. This is called an integration key — a canonical business key that identifies the entity regardless of which system is reporting it.

Q11Satellite Why would you split a satellite into two tables (e.g., static details vs daily balance)?
Model Answer

Satellites should be split by rate of change. Account name and account type change maybe once a year. Account balance changes daily. If they're in the same satellite, every daily balance load creates a new satellite row — even though name and type didn't change. After a year, the satellite has 365 versions for each account, and 364 of them have identical name and type. Querying the latest name now requires filtering through all those unnecessary rows. Splitting into Sat_Account_Details (name, type, status, credit rating — slow-changing) and Sat_Account_Balance (balance, currency, balance_date — fast-changing) means: Sat_Account_Details only gets a new row when name or type actually changes — maybe a few rows per year per account. Sat_Account_Balance gets a new row daily — hundreds of rows per year per account. Querying the current name is fast (few rows). Querying the full balance history is accurate (one row per real change). The rule: if two groups of attributes change at different frequencies, they deserve separate satellites. More satellites, but each is smaller and queries are faster.

Q13Architecture What is a Bridge table in Data Vault and when do you need one?
Model Answer

A Bridge table is a pre-computed join path that spans multiple Links in the Data Vault. Without it, queries that cross multiple Links require complex multi-step joins: Hub_Account → Link_AccountEmployee → Hub_Employee → Link_EmployeeDesk → Hub_Desk. That's four joins just to connect an account to a trading desk. With a Bridge_AccountTradingDesk table, the path is pre-computed daily — one join from the fact query to the bridge, then one join to each endpoint. Bridge tables are snapshot-based, like PIT tables — rebuilt daily. You need a Bridge when: multiple business questions require crossing more than two Links, the cross-vault join path is used by many queries, or performance on the raw multi-join query is unacceptable. The analogy: PIT tables optimize single-Hub queries (give me everything about Account X on date Y). Bridge tables optimize cross-Hub queries (give me all accounts + their employees + their instruments for date Y). Together, PIT and Bridge tables are what transform a raw Data Vault — which is optimized for auditability, not query performance — into something analysts can actually use.

Q14Production How would you orchestrate all three vault layers to run in the correct order?
Model Answer

Data Vault has strict dependency ordering: Raw Vault must finish before Business Vault can run, which must finish before Information Mart. Within the Raw Vault, Hubs must load before Links (Links reference Hub hash keys), and Links must load before Satellites that are attached to Links. For orchestration, I'd use Apache Airflow with a DAG that enforces this order: Task 1 (parallel): load all Hub tables simultaneously — they're independent of each other. Task 2 (parallel, after Task 1): load all Link tables — they all depend on Hubs being present. Task 3 (parallel, after Task 2): load all Satellite tables — Sat on Hub loads after Hub, Sat on Link loads after Link. Task 4 (after Task 3 complete): rebuild PIT tables — they depend on Satellites. Task 5 (after Task 4): rebuild Bridge tables. Task 6 (after Task 4–5): rebuild Information Mart. Each task is idempotent — it checks what's already in the vault and only inserts new records. If any task fails, Airflow retries it and downstream tasks wait. The whole DAG runs nightly at midnight on a 2-hour SLA, matching the regulatory T+1 reporting requirement. I'd add data quality checks between stages — if Hubs load zero new records but we expect hundreds, something's wrong with the source feed.

Data Vault vs Star Schema vs 3NF — Full Comparison
▼

Third Normal Form (3NF) / Operational: Normalized for write operations, minimizes redundancy. Used in source systems (CRM, ERP). Not for analytics — too many joins, query performance poor.

Star Schema: Denormalized for read performance. Fast queries, simple SQL, works with BI tools. Problem: overwrites history by default, multi-source integration is manual and brittle, difficult to audit. Best for: simple, single-source analytical use cases.

Data Vault 2.0: Optimized for auditability, historization, and agile multi-source integration. All history preserved forever. Best for: regulated industries, multi-source enterprise DWs, anywhere full audit lineage is required. Cost: more tables, slower raw queries (mitigated by PIT/Bridge), steeper learning curve.

Hybrid (common in practice): Raw Vault for auditability + Information Mart (star schema) for query performance. This is what we built in Layer 3.

🔑
Hash Key Computation — Why Consistency is Everything
▼

The entire entity resolution capability of Data Vault depends on hash keys being computed identically for the same business entity regardless of source. If GL sends "ACC-123" and Trading sends "acc-123", without consistent normalization they produce different MD5 hashes and appear as two different entities in Hub_Account.

Normalization rules to apply before hashing:

  • Uppercase all characters
  • Trim leading and trailing whitespace
  • Standardize date formats (YYYY-MM-DD)
  • Remove known prefixes/suffixes if sources use different ones
  • Define NULL handling (use '~NULL~' as sentinel)
  • For multi-part keys, define separator consistently ('||')

Document these normalization rules in a data dictionary and apply them in the hash_utils module so every loading script uses the same logic. Inconsistent hashing is the #1 cause of Data Vault implementation failures.

AWS EMR Architecture Essentials
▼

EMR cluster structure: Master node (runs YARN ResourceManager + Spark driver) + Core nodes (run HDFS + Spark executors) + Task nodes (optional, Spark executors only, perfect for spot instances since they don't hold HDFS data).

S3 as storage: In modern EMR, S3 replaces HDFS as the data lake. Spark reads directly from S3, processes in memory/local SSD on executors, writes back to S3. This makes clusters ephemeral — start for a job, terminate when done, no persistent storage cost.

Spot vs On-Demand: Core nodes should be On-Demand (reliable, hold HDFS replicas if used). Task nodes can be Spot (up to 90% savings, can be interrupted — but EMR gracefully handles task node loss by rescheduling tasks on remaining nodes).

EMR Serverless (new option): Fully serverless EMR — no cluster management, auto-scales, pay per vCPU-hour. Similar to Glue but with full Spark config control. Worth knowing for interviews as it bridges EMR and Glue.

Financial Concepts Every Finance DE Must Know
▼

General Ledger (GL): The master financial record of a company — all debits and credits. Journal entries are the atomic unit. Double-entry: every debit has an equal credit. The GL is what produces the balance sheet and income statement.

Notional amount vs market value: Notional is the face value of a contract (what it's "for"). Market value is what it's worth today. A $1M bond bought at par has $1M notional and $1M market value. If rates move, market value changes but notional stays $1M.

T+2 settlement: Trade date + 2 business days = when the securities and cash actually exchange. A trade on Monday settles Wednesday. Important for end-of-period reporting — trades from Dec 31 may not settle until Jan 2.

Mark-to-Market (MTM): Revaluing assets at current market prices daily. Unrealized gain/loss. Used in trading books.

LEI (Legal Entity Identifier): A 20-character code uniquely identifying counterparties in financial transactions — the global standard for entity identification, required by multiple regulators for trade reporting.
