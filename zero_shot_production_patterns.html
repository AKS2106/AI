Zero-shot production prompt patterns for summarization, translation, extraction, classification, sentiment analysis, and rewriting

Weak version — what most beginners write
Summarize this article: {article}
Problems: undefined length, no audience, no format — every response will be different.
Production version
You are a content summarization engine for a B2B SaaS newsletter. Summarize the article below following these rules: - Length: exactly 3 sentences - Sentence 1: the core news/finding (what happened) - Sentence 2: why it matters to software engineers - Sentence 3: one concrete takeaway or next action Tone: neutral, factual, no hype words like "revolutionary" or "game-changing" Output: plain text only, no bullet points, no headers <article> {article_text} </article>
Why each part earns its place
exactly 3 sentences — "brief" means 2 sentences to some models, 8 to others. Be numeric.
Defining each sentence's job forces structure without requiring JSON.
Naming the audience ("software engineers") shifts vocabulary and what counts as "important."
<article> delimiters separate your instruction from the user's content — prevents injection.
Banning hype words by example is more reliable than saying "be objective."
Variants for different use cases
Hierarchical summary (long docs)
Return a JSON object: { "headline": "one sentence, <15 words", "summary": "2–3 sentences", "key_points": ["string", ...], "action_items": ["string", ...] }
Audience-adaptive summary
Summarize for a {audience} who has {expertise_level} knowledge of {domain}. Avoid jargon above their level. Max reading time: {minutes} minutes.
Naive version
Translate to French: {text}
Ignores tone, formality, domain vocabulary, and locale differences (French France vs French Canada).
Production version
You are a professional localization engine. Translate the text below with these requirements: - Target language: French (France, not Canadian French) - Formality: formal "vous" form, not "tu" - Domain: legal/contract language — preserve technical legal terms, do not simplify them - Brand names, product names, and proper nouns: keep in original language - If a phrase has no direct equivalent, translate the meaning, not the words Return ONLY the translated text. Do not include the original, explanations, or notes. <source_text> {text} </source_text>
The dimensions you must control in translation
Locale — French (France) vs (Canada), Spanish (Spain) vs (Mexico). Different vocabulary and spelling.
Formality register — formal/informal, honorifics. Critical for business comms.
Domain — legal, medical, technical. Terms must match industry convention, not dictionary definition.
What to preserve — brand names, product names, UI strings, code snippets.
Untranslatable phrases — tell the model what to do when direct translation fails.
Output format — same markdown/HTML structure as input? Preserve line breaks?
Detecting source language first (production pattern)
Step 1 — detect: Return the ISO 639-1 language code of the text below. Return only the code (e.g. "en", "fr", "de"). Text: {text} Step 2 — translate (only if Step 1 ≠ "en"): [translation prompt above]
Interview tip: for high-volume translation, you can skip detection by asking for it inline: "If the text is already in English, return it unchanged. Otherwise, translate to English."
The most important zero-shot pattern in production
You are a structured data extraction engine. Extract the following fields from the contract text below. FIELDS: - party_a: full legal name of the first party - party_b: full legal name of the second party - effective_date: ISO 8601 format (YYYY-MM-DD), or null if not stated - termination_date: ISO 8601 format, or null if not stated - governing_law: jurisdiction/state, or null - payment_terms: brief description, or null - auto_renewal: true/false/null RULES: - Extract only what is explicitly stated — do not infer or assume - Dates mentioned as "the first of next month" → null (ambiguous) - Return ONLY a JSON object matching the schema above, no other text <contract> {contract_text} </contract>
The 3-part extraction formula
Define every field's type and null condition — "or null if not stated" prevents the model from guessing when a field is missing.
Specify format for ambiguous types — dates especially. ISO 8601 is unambiguous. "June 2024" is not.
State the extraction policy — "extract only what is explicit" stops hallucination of missing fields. Without this, models fill in plausible-sounding values.
Named entity extraction (less structured)
Extract all named entities from the text below. Return a JSON object with these arrays: { "people": [], // full names of individuals "organizations": [], // company/org names "locations": [], // cities, countries, addresses "dates": [], // any date or time reference "monetary": [] // any amounts with currency } Include each entity only once (deduplicate). If an array has no entities, return []. <text>{text}</text>
Single-label classification
You are a content moderation classifier. Classify the user message below into EXACTLY ONE of these categories: CATEGORIES: - safe: normal conversation, questions, requests - spam: promotional content, repetitive nonsense, bot-like patterns - hate: attacks on people based on identity - threat: explicit or implied threats of harm - pii: contains personal data (emails, phone numbers, SSNs, addresses) Rules: - If multiple categories apply, choose the highest severity (threat > hate > pii > spam > safe) - Return a JSON object: {"category": "...", "confidence": 0.0–1.0, "reason": "one sentence"} Message: <msg>{user_message}</msg>
Multi-label classification
You are a product feedback tagger. A piece of feedback can belong to multiple categories. Return ALL that apply. TAGS: [performance, pricing, ui, onboarding, support, feature_request, bug_report, praise] Return JSON: {"tags": ["tag1", "tag2"], "primary_tag": "most dominant tag"} Only use tags from the list above. If none fit, return {"tags": [], "primary_tag": null} Feedback: <fb>{feedback}</fb>
The confidence score pattern (interview must-know)
Always ask for a confidence score in classification tasks. When confidence is below a threshold (e.g. 0.7), route to human review instead of taking automated action. This is the standard pattern for human-in-the-loop AI systems — a key topic in senior AI engineer interviews.
Handling unknown / out-of-scope inputs
// Always give the model an escape hatch: - If the input does not clearly fit any category, return {"category": "unclear", "confidence": 0.0, "reason": "..."} // Without this, the model will force-fit ambiguous input into a wrong category
Beyond positive/negative — production-grade sentiment
You are a customer experience analyst. Analyze the sentiment of the review below across multiple dimensions. Return a JSON object with this exact structure: { "overall": "positive" | "negative" | "neutral" | "mixed", "score": -1.0 to 1.0, // -1 = strongly negative, 0 = neutral, 1 = strongly positive "dimensions": { "product_quality": "positive" | "negative" | "neutral" | "not_mentioned", "customer_service": "positive" | "negative" | "neutral" | "not_mentioned", "value_for_money": "positive" | "negative" | "neutral" | "not_mentioned", "ease_of_use": "positive" | "negative" | "neutral" | "not_mentioned" }, "key_phrases": { "positive": ["phrase1", "phrase2"], // direct quotes from review "negative": ["phrase1", "phrase2"] }, "churn_risk": "high" | "medium" | "low" } Review: <review>{review_text}</review>
Why multi-dimensional sentiment beats simple positive/negative
Example review:
"The product itself is excellent but the setup was a nightmare and support took 3 days to respond."
Simple model says: mixed
Multi-dim model says:
product_quality: positive
ease_of_use: negative
customer_service: negative
churn_risk: medium
The churn_risk field is a derived signal — the model reasons about whether this person is likely to cancel. This is what makes sentiment analysis actually useful for product teams, not just a vanity metric.
Tone rewriting — same content, different voice
You are a professional business writing editor. Rewrite the message below to match the target tone. Do NOT change the factual content, add new information, or remove any key points. Target tone: formal and diplomatic Context: this is a client-facing email from a senior manager Original message: <original> {message} </original> Rules: - Preserve all facts, dates, and numbers exactly - Remove casual language, contractions, and slang - If the original is unclear, choose the most charitable interpretation - Maximum length change: ±20% of original word count
Reading level rewriting (Flesch–Kincaid targeting)
Rewrite the text below for a {grade_level} reading level. Target audience: {audience_description} Rules: - Replace technical jargon with plain-language equivalents - Break sentences longer than 20 words into shorter ones - Preserve all factual information — do not omit or simplify facts - Keep proper nouns and brand names unchanged <text>{technical_text}</text>
The content-preservation constraint (most important rule in rewriting)
Always explicitly forbid adding or removing content. Without this, models will "helpfully" pad, trim, or clarify the original — silently changing meaning. In legal, medical, and financial contexts this is dangerous. The phrase "do NOT change the factual content" is load-bearing.
Diff-aware rewriting (advanced)
Rewrite the text below. After the rewrite, return a bullet list of every change you made and why. Format: REWRITE: [rewritten text] CHANGES: - [original phrase] → [new phrase]: [reason] - ...
The changes list forces the model to be accountable for every edit. It also makes reviewing AI rewrites much faster — you scan the diff instead of reading both versions.
