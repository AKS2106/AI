1️⃣ LLM Fundamentals — How language models actually work Week 1-2 ▼
ConceptWhat You Need to Know
What is an LLM?Transformer-based neural networks trained to predict the next token. GPT-4, Claude, Llama3 are all LLMs. They generate text probabilistically — not deterministically.
TokensLLMs process tokens, not words. "customer" = 1 token. Costs are per token. ~750 words ≈ 1000 tokens. Use the tiktoken library to count.
Temperature0 = same answer every time (use for billing/facts). 1 = creative/random. Use 0.3 for support responses. Interviewers ALWAYS ask this.
Context WindowMax tokens an LLM can "see" at once. GPT-3.5 = 16K, GPT-4 = 128K, Claude = 200K. Critical for RAG design — limits how much context you can inject.
HallucinationLLMs generate plausible-sounding but wrong answers. RAG solves this by grounding in real documents. Always explain your mitigation strategy in interviews.
System vs User MessageSystem: your instructions for the AI (role, behavior, constraints). User: the actual question. This is the chat completion format.
python — your first LLM call
from groq import Groq
import os
from dotenv import load_dotenv

load_dotenv()
client = Groq(api_key=os.getenv("GROQ_API_KEY"))

response = client.chat.completions.create(
    model="llama3-70b-8192",
    messages=[
        {"role": "system", "content": "You are a helpful customer support AI."},
        {"role": "user", "content": "My order #12345 is delayed. What should I do?"}
    ],
    temperature=0.3,
    max_tokens=500
)
print(response.choices[0].message.content)
print(f"Tokens used: {response.usage.total_tokens}")
2️⃣ Prompt Engineering — The highest-ROI skill in AI Week 2-3 ▼

A well-crafted prompt can 10x the quality of LLM output without changing the model. This is the single most impactful skill early in your AI engineering journey.

TechniqueWhen to Use
Zero-shotAsk directly with no examples. Best for simple, well-defined tasks the LLM already understands.
Few-shotProvide 2-3 examples before the question. Use when output format matters (JSON, specific labels).
Chain-of-ThoughtAdd "Think step by step." Dramatically improves accuracy on complex reasoning, math, multi-step logic.
ReActAlternate between Reasoning and Acting (using tools). The basis for ALL AI agents.
Role Prompting"You are a senior support agent with 10 years experience..." — adds domain expertise and consistent tone.
Output Formatting"Respond ONLY in valid JSON with keys: category, priority, confidence" — for structured downstream processing.

3️⃣ Embeddings — Text as numbers that capture meaning Week 4-5 ▼

Embeddings are lists of numbers (~1536 floats) that represent the meaning of text. Similar texts → similar numbers. This is the mathematical foundation of every RAG system.

python — free local embeddings on Mac M5
from sentence_transformers import SentenceTransformer
from sklearn.metrics.pairwise import cosine_similarity

# Runs 100% locally on your Mac M5 — FREE forever
model = SentenceTransformer("all-MiniLM-L6-v2")

sentences = [
    "Customer is angry about billing issue",
    "User is upset about incorrect charge on invoice",
    "How to reset my password",
]

embeddings = model.encode(sentences)
print(f"Shape: {embeddings.shape}")  # (3, 384)

sim1 = cosine_similarity([embeddings[0]], [embeddings[1]])[0][0]
sim2 = cosine_similarity([embeddings[0]], [embeddings[2]])[0][0]

print(f"Billing vs Invoice: {sim1:.3f}")  # ~0.85 (similar!)
print(f"Billing vs Password: {sim2:.3f}") # ~0.20 (different)
💡
Key Interview Point
OpenAI text-embedding-3-small costs $0.0001/1K tokens. sentence-transformers/all-MiniLM-L6-v2 is free, runs locally, and is 90% as good for most tasks. Always mention this tradeoff.

python — ChromaDB basics
import chromadb
from sentence_transformers import SentenceTransformer

client = chromadb.PersistentClient(path="./my_db")
collection = client.get_or_create_collection("support_tickets")
model = SentenceTransformer("all-MiniLM-L6-v2")

# Add documents
docs = ["Billing issue with invoice", "Password reset help", "Order tracking problem"]
embeddings = model.encode(docs).tolist()

collection.add(
    documents=docs,
    embeddings=embeddings,
    ids=["doc1", "doc2", "doc3"],
    metadatas=[{"category": "billing"}, {"category": "tech"}, {"category": "shipping"}]
)

# Query
query = "I was charged twice"
query_embedding = model.encode([query]).tolist()
results = collection.query(query_embeddings=query_embedding, n_results=2)
print(results['documents'])  # Returns most similar docs
5️⃣ RAG Systems — The most important enterprise AI pattern Week 6-8 ▼

RAG solves hallucination by finding relevant documents and injecting them into the LLM context. This is how Amazon, Salesforce, and Microsoft build their AI assistants.

PhaseStepsTools
INDEXING (offline)Load docs → Chunk → Embed → Store in VectorDBLangChain loaders, ChromaDB, sentence-transformers
RETRIEVAL (per query)Embed query → Similarity search → Get top-K chunksChromaDB.query(), cosine similarity
GENERATION (per query)Build prompt with context → Call LLM → Return answerLangChain, Groq, prompt templates
EVALUATION (periodic)Check faithfulness, relevance, answer qualityRAGAS library
python — minimal RAG pipeline
from langchain_community.vectorstores import Chroma
from langchain_community.embeddings import HuggingFaceEmbeddings
from langchain_groq import ChatGroq
from langchain.chains import RetrievalQA
from langchain.text_splitter import RecursiveCharacterTextSplitter

# 1. Load and chunk your documents
splitter = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=50)
chunks = splitter.split_text(your_document_text)

# 2. Embed and store (free, local)
embeddings = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")
vectordb = Chroma.from_texts(chunks, embeddings, persist_directory="./rag_db")

# 3. Build RAG chain
llm = ChatGroq(model="llama3-70b-8192", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=vectordb.as_retriever(search_kwargs={"k": 5}),
    return_source_documents=True
)

# 4. Query
result = qa_chain.invoke({"query": "What is the refund policy?"})
print(result["result"])
print(result["source_documents"][0].page_content)
6️⃣ Advanced RAG — Hybrid Search, Re-ranking, HyDE, Parent-Child Week 9-11 ▼

Basic RAG fails in production. Advanced RAG adds layers that make it actually reliable. These separate junior from senior AI engineers.

TechniqueWhat Problem It Solves
Hybrid SearchSemantic alone misses exact codes like "SKU-4821". BM25 keyword + semantic = both. Use 70% semantic + 30% BM25 weighting.
Re-rankingFirst pass gets top-20 cheaply, cross-encoder re-ranks to top-5 accurately. Cohere Rerank API is free tier. Dramatically improves precision.
Query ExpansionLLM generates 3-5 alternative phrasings, search all of them. Solves vocabulary mismatch: user says "broken", docs say "malfunction".
HyDEGenerate a hypothetical answer first, embed THAT, then search. Works better than embedding the question because answers match document style.
Parent-Child ChunkingSmall chunks for precise retrieval, parent chunk returned for full context. Best of both worlds.
7️⃣ AI Agents — LLMs that take actions in the world Week 12-14 ▼

An AI Agent is an LLM that can take ACTIONS — calling APIs, searching the web, running code. The LLM decides which tool to use and when.

ComponentExplanation
The Brain (LLM)Reads current state + available tools + goal → outputs either a tool call or a final answer.
ToolsFunctions the agent calls: search_web(), query_database(), send_email(). Each has a name and description (the LLM reads these!).
ReAct LoopReason → Act → Observe → Reason → ... Repeat until final answer.
Stopping ConditionCRITICAL: always set max_iterations (10-15). Without this, agents loop forever.
Tool Description QualityThe LLM picks tools based on descriptions. "Gets order info" is bad. "Retrieves real-time order status, tracking, and ETA for a given order ID" is good.
8️⃣ LangGraph — Building controllable agentic workflows Week 13-15 ▼

LangGraph models your AI workflow as a directed graph. Nodes = processing steps. Edges = transitions. This gives you fine-grained control over complex multi-step AI processes.

ConceptExplanation
StateA TypedDict holding all data flowing through the graph. Every node reads from and writes to state.
NodesPython functions: take state as input, do something, return updated state. Keep nodes small and single-purpose.
Conditional EdgesGo to node B or node C based on state value. This is how you build branching logic.
CheckpointingSave graph state to SQLite (local) or PostgreSQL (production). Enables pause/resume and human-in-the-loop.
Human-in-the-loopPause graph, wait for human approval before irreversible actions (refunds, emails). Enterprise REQUIREMENT.
python — minimal LangGraph agent
from langgraph.graph import StateGraph, END
from typing import TypedDict, Annotated
import operator

class AgentState(TypedDict):
    messages: Annotated[list, operator.add]
    next_action: str

def agent_node(state: AgentState):
    # LLM decides what to do next
    return {"next_action": "get_order_status"}

def tool_node(state: AgentState):
    # Execute the tool
    return {"messages": ["Order #123: Delayed by 2 days"]}

def should_continue(state):
    if state["next_action"] == "end":
        return END
    return "tool_node"

graph = StateGraph(AgentState)
graph.add_node("agent", agent_node)
graph.add_node("tool_node", tool_node)
graph.set_entry_point("agent")
graph.add_conditional_edges("agent", should_continue)
graph.add_edge("tool_node", "agent")
app = graph.compile()

Interview Questions

How did you handle different types of customer intents?▼
Two-stage system: first a zero-shot LLM classifier that returns structured JSON with the intent and confidence score, then a prompt template selector that picks the right template. For billing queries I use temperature 0.1 for precision, for general queries 0.7 for natural conversation. This mirrors how Salesforce manages Einstein GPT prompt templates per use case.
How would you improve the system if the LLM gives wrong answers?▼
Three layers: (1) Golden dataset eval on every deployment catches regressions, (2) User feedback (thumbs up/down) is logged and I analyze patterns weekly, (3) For repeated failure cases, I add them to the golden dataset and tune the relevant prompt template. This is a data-driven improvement loop, not guesswork.
What is your approach to prompt versioning?▼
Store prompts in a database table with version numbers and performance scores. When updating a prompt, run it against the golden dataset first — only deploy if the score equals or beats the previous version. This prevents regressions and creates an audit trail.

def process_refund_node(state):
    refund_amount = state["refund_amount"]

        # PAUSE the graph — wait for human approval

            "type": "approval_required"

        if not approval["approved"]:
            return {"result": "Refund denied by supervisor"}
    
    # Process the refund

Interview Prep

30+ real questions, system design practice, and what interviewers actually want to see.

Top Interview Questions by Category

LLM Fundamentals

What is temperature and when do you use 0 vs 1?Fundamentals▼
0 = deterministic output (use for billing, classification, facts where you need consistency). 1 = creative/random (use for creative writing, brainstorming). 0.3-0.5 for customer support responses — professional but not robotic. Key insight: temperature affects the probability distribution over the vocabulary, not the "creativity" directly.
How do you handle context window limits in production?Fundamentals▼
Four strategies: (1) RAG — only inject the relevant chunks, not the whole document, (2) Sliding window — for long conversations, keep only the last N turns plus a summary of earlier turns, (3) Summarization — compress earlier context with a cheap model before adding new turns, (4) Model selection — use Claude 200K or GPT-4-128K when the full context is genuinely needed.

RAG Systems

What causes RAG to fail and how do you fix each failure mode?RAG▼
5 failure modes: (1) Bad chunking — answer split across chunks, fix with better chunk strategy and overlap, (2) Wrong embedding model — domain mismatch, fix with domain-specific or fine-tuned model, (3) Insufficient top-K — right document not in retrieved set, fix by increasing K and adding re-ranking, (4) Prompt not enforcing grounding — LLM ignores context and hallucinates, fix with explicit instruction "ONLY use the provided context", (5) LLM ignoring context despite instruction — use stronger models (GPT-4) or chain-of-thought prompting to force citation.
How do you evaluate a RAG system? What metrics matter?RAG▼
I use RAGAS which gives 4 automated metrics: (1) Faithfulness — does the answer only use facts from the context? (anti-hallucination), (2) Answer Relevance — does the answer actually address the question?, (3) Context Precision — are the retrieved chunks relevant? (retrieval quality), (4) Context Recall — did we retrieve ALL the needed info? Beyond RAGAS, I always have a golden dataset of 30-50 manually verified Q&A pairs and track pass rate on every deployment. In production, user feedback (thumbs up/down) feeds a continuous improvement loop.

AI Agents & LangGraph

Why use LangGraph instead of LangChain's AgentExecutor?Agents▼
Three critical differences: (1) Persistent state — LangGraph checkpoints every step to SQLite/PostgreSQL, so if a step fails or needs human input, you can resume from exactly where you left off. AgentExecutor starts over. (2) Conditional branching — LangGraph lets you define exactly which node runs next based on state values. AgentExecutor is a black box loop. (3) Human-in-the-loop — LangGraph's interrupt() mechanism lets you pause the graph and wait for a human decision. This is a non-negotiable for production financial or legal applications. (4) Better observability — every node execution is logged with inputs and outputs.
How do you prevent an agent from looping forever?Agents▼
Multiple safeguards: (1) recursion_limit in LangGraph (I set 15 — enough for complex tasks), (2) A step_count field in state — if any single tool is called more than 3 times with the same arguments, force a conclusion, (3) Catch GraphRecursionError and return a graceful fallback response rather than an unhandled exception, (4) In production, monitor P99 execution time — if an agent takes more than 30 seconds, something is wrong.

Multi-Agent Systems

When would you use multiple agents instead of one large agent?Multi-Agent▼
Three reasons: (1) Specialization — a BillingAgent with 10 billing-specific tools and an expert billing system prompt dramatically outperforms a generalist with all 40 tools. LLMs get confused with too many tools. (2) Independent scalability — during a software release, TechAgent traffic spikes 10x. I can scale that one agent without scaling Billing or Shipping. (3) Independent failure modes — if TechAgent is degraded, Billing still works. One big agent has one point of failure for everything. (4) Different LLM requirements — the Orchestrator needs GPT-4o-mini's reliable tool-calling, but specialist agents can use Groq Llama3 for speed and cost.

System Design

Design a RAG system for 1000 enterprise customers (multi-tenant)System Design▼
Key challenge: data isolation. Approach: (1) Pinecone namespace per tenant — tenant_id as namespace ensures strict data separation at query time, (2) Metadata filter on every query: {"filter": {"tenant_id": "customer_123"}} — defense in depth, (3) Document encryption at rest in S3 with per-tenant KMS keys, (4) AWS IAM role per tenant for API access, (5) Per-tenant usage tracking for billing. Architecture: API Gateway → Lambda (injects tenant_id from auth token) → Pinecone with namespace → Bedrock LLM. Data plane fully isolated, control plane shared. Cost: ~$0.001/query + $70/month Pinecone per 1M vectors.
When would you use AWS Bedrock instead of OpenAI directly?AWS▼
Three enterprise scenarios: (1) Data compliance — Bedrock processes data within your AWS VPC and data never leaves your account. Critical for healthcare (HIPAA), finance (SOX), government. OpenAI sends data to their servers. (2) Unified AWS billing — one invoice for all cloud services including LLM calls, simplifies enterprise procurement. (3) IAM integration — use existing AWS IAM roles and policies for LLM access control. No separate API key management. Trade-offs: Bedrock has slightly higher latency and fewer cutting-edge models than direct OpenAI access. For most enterprises, the compliance and security benefits outweigh the model selection limitations.
