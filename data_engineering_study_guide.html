01
Batch vs Streaming

The most fundamental choice in data engineering. Every system design question starts here. The key question is: how stale is too stale?

■ Batch Processing
Process large chunks of data on a schedule — hourly, nightly, weekly. The data is collected, stored, then processed all at once.

When to use: results can be hours or days old. Large volumes processed efficiently. Easy to re-run on failure. Lower cost — spins up, works, shuts down.

Real example: Swiggy's nightly restaurant payout report. All orders from the day processed at 2am. PySpark on EMR reads S3, aggregates by restaurant, writes report.
PySpark on EMRAirflow scheduleS3 + Redshift
■ Stream Processing
Process each event the moment it arrives. No waiting, no schedules. Results in seconds.

When to use: sub-minute freshness needed. Events arrive continuously. User is waiting on screen right now. Fraud detection, live dashboards, ride tracking.

Real example: Swiggy's fraud check on every order placement. Event → Kinesis → Spark Streaming → fraud score → approve/block. All in under 2 seconds.
Kinesis / KafkaSpark StreamingDynamoDB
Middle ground: micro-batching. Running batch every 5 minutes instead of nightly gives most of the freshness of streaming without the full complexity. Always ask: do we truly need sub-second latency, or would 5-minute micro-batches solve the problem? This is the answer that wins interviews.

In PySpark: spark.read = batch. spark.readStream = streaming. The transformation logic is nearly identical — the framework handles the difference. This is worth mentioning in interviews.
02
Data Lake vs Warehouse vs Lakehouse

You don't choose one — you use all three, and data flows from lake through to warehouse. Each has a specific role.

🏞️ Data Lake
What: raw storage, any format, cheap.

Format: JSON, CSV, Parquet, images, logs — anything.
Schema: schema-on-read (applied at query time).
Cost: ~$23/TB/month on S3.
Query speed: slow without tuning.

Use for: raw archival, ML training data, compliance (never delete raw).
AWS S3
🏛️ Data Warehouse
What: structured, optimised for SQL queries.

Format: tables only, columnar storage.
Schema: schema-on-write (enforced at insert).
Cost: ~$250/TB/month.
Query speed: very fast.

Use for: executive dashboards, BI reports, analyst SQL queries.
RedshiftSnowflake
🏠 Lakehouse
What: lake's cost + warehouse's query performance.

Format: Parquet + Delta Lake on S3.
Schema: enforced + evolvable.
Cost: S3 rates + compute on query.
Query speed: good with Athena/Spectrum.

Use for: mixed teams, cost-sensitive, ACID transactions needed.
Delta LakeAthenaRedshift Spectrum
Key interview insight — Redshift Spectrum: lets analysts run SQL on S3 Parquet files directly without copying data into Redshift first. You get warehouse-speed SQL at lake-level storage cost. This is the practical "lakehouse" pattern most AWS teams use, even without Delta Lake.
Schema-on-read vs schema-on-write: Warehouse enforces schema at write time — bad data is rejected immediately. Lake applies schema at query time — flexible but risky if upstream data changes silently. Delta Lake on S3 gives you write-time enforcement with the flexibility to add columns without breaking existing queries.
One-sentence interview answer: "I'd use S3 as the lake for raw ingestion and ML data, PySpark to transform into structured Parquet, and Redshift Spectrum for analyst SQL access — loading into Redshift proper only for the highest-frequency queries where Spectrum latency isn't fast enough."
03
Lambda, Kappa & Medallion

Three architecture patterns. Each solves a different problem. Know all three and when to use each — interviewers always ask you to compare them.

LAMBDA ARCHITECTURE
Lambda — batch accuracy + streaming speed, two pipelines
The same data feeds two parallel pipelines. Batch layer processes full history accurately (but slowly). Speed layer processes recent data in real-time (fast but approximate). Serving layer merges both results.

Batch layer: PySpark on EMR, runs nightly, reads all S3 data, writes pre-computed views to Redshift.
Speed layer: Spark Streaming on Kinesis, always on, processes last few minutes, writes to DynamoDB with TTL.
Serving layer: queries merge Redshift (history) + DynamoDB (live) into one answer.

The problem: two codebases with the same logic. Batch and streaming implementations can drift and give different numbers for the same metric.

Use when: need both real-time freshness AND accurate historical data. Financial reporting + live dashboard on the same platform.
EMR + PySparkSpark StreamingKinesisRedshiftDynamoDB
KAPPA ARCHITECTURE
Kappa — one streaming pipeline, replay history through it
Removes the batch layer entirely. ONE streaming pipeline handles everything — live events AND historical reprocessing. Kinesis/Kafka retains events in an immutable log. When you need to reprocess history, deploy a v2 consumer starting from an earlier offset. Same code, different starting position.

Replay process: (1) deploy v2 consumer from past timestamp, (2) it races through history writing to new output table, (3) once caught up to "now", swap serving layer to v2, (4) decommission v1.

The problem: replay is expensive. Kinesis retention max 365 days. Large history requires long retention periods or S3 backup.

Use when: primarily real-time needs, small team, frequent logic changes, don't need years of replay.
Kinesis / KafkaSpark StreamingOne codebase
MEDALLION ARCHITECTURE
Medallion — Bronze → Silver → Gold, progressive refinement
Three progressive layers. Each layer is more refined than the last. One codebase, batch-first. The most commonly used pattern on AWS + Databricks today.

🥉 Bronze: raw data lands exactly as received. No transformations. Any format. Append-only. Never modify. This is your safety net — always reprocessable.

🥈 Silver: cleaned, validated, deduplicated, enriched. PySpark reads Bronze, applies quality rules, joins reference data, writes clean Parquet. Bad rows go to quarantine with a reason code.

🥇 Gold: business-ready aggregates. GroupBy, sum, rollup. What dashboards and ML models consume. Rebuilt from Silver on each run.

Use when: batch analytics, data quality is critical, AWS + PySpark team, want simplicity over Lambda's complexity.

04
Data Quality & Validation

Bad data is the most common reason pipelines fail silently. The key insight: never drop bad data — quarantine it with a reason code so you can investigate and reprocess.

1. Completeness
Are required fields present? Check null rates on critical columns. Alert if null rate exceeds threshold (e.g. >5% on order_id).
isNotNull(), dropna()
2. Validity
Do values match expected format/range? Check: amount > 0, email regex, status in allowed list.
filter(), rlike(), between(), isin()
3. Uniqueness
Are primary keys actually unique? Detect duplicates, keep latest by updated_at using Window function.
row_number().over(Window)
4. Consistency
Do related fields agree? If status="delivered" then delivery_date must exist. Refund ≤ original amount.
when().otherwise()
5. Freshness — the most commonly forgotten dimension
Did the data arrive when expected? A pipeline can pass all null checks and still silently process yesterday's data if an upstream system failed. Always check max(event_timestamp) before processing. If more than N hours old, block the pipeline — don't process stale data into your Gold layer.
max(timestamp)datediff()CloudWatch alarm
Never reject at Bronze. Never drop silently. Always land raw data first, validate second. If your validation rule was wrong, you can reprocess from Bronze. If you rejected at ingest, the data is gone forever. Bad rows go to s3://bucket/silver/quarantine/ with a dq_reason column.
Full validation pipeline — PySpark
# Tag each row with dq_status and dq_reason validated = bronze_df.withColumn("dq_reason", when(col("order_id").isNull(), "null_order_id") .when(col("amount") <= 0, "invalid_amount") .when(~col("status").isin(["placed","confirmed","delivered"]), "bad_status") .when((col("status")=="delivered") & col("delivery_date").isNull(), "delivered_no_date") .otherwise(None) ).withColumn("dq_status", when(col("dq_reason").isNotNull(), "fail").otherwise("pass")) silver = validated.filter(col("dq_status") == "pass") quarantine = validated.filter(col("dq_status") == "fail") # Publish quality metrics — block Gold if pass rate too low pass_rate = silver.count() / bronze_df.count() * 100 if pass_rate < 95: raise Exception(f"Quality check failed: {pass_rate:.1f}% — halting Gold job")
Blocking checks
Halt the pipeline on failure. Use for: freshness too old, completeness null rate >10%, zero rows received. Bad data must not reach Gold.
Non-blocking checks
Log + alert but continue. Use for: soft business rules where partial data is better than no data. Flag the issue, let the pipeline proceed.
05
Spark Performance Tuning

The golden rule: diagnose before you tune. 90% of Spark problems are code problems, not resource problems. More machines won't fix data skew.

Diagnose first: open Spark UI (port 4040 on driver, or EMR Application History). Find the slow stage. Check: (1) task duration distribution — is one task taking 100× longer? That's skew. (2) shuffle bytes — are they huge? That's expensive joins. (3) spill to disk? Need more executor memory.
1
Data Skew — one task takes 100× longer than others
One key has far more rows than others (e.g. 70% of rows have country_code="IN"). All assigned to one task. That task runs for 45 minutes while 199 others finish in 10 seconds.

Fix 1 (try first): Enable AQE — spark.sql.adaptive.skewJoin.enabled=true — splits skewed partitions automatically.
Fix 2: Broadcast join — if the smaller table is under 100MB, broadcast(small_df) — no shuffle at all.
Fix 3 (nuclear): Salting — append random suffix (0-9) to skewed key, explode other table with same range, join on salted key. Splits one hot partition into 10.
2
Excessive Shuffle — massive data movement across network
groupBy, join, repartition all trigger shuffles. A shuffle on 500GB moves 500GB across the network between stages.

Fix 1: Broadcast joins for any table under 100MB — eliminates entire shuffle stage.
Fix 2: Pre-aggregate before joining — reduce data volume before the shuffle.
Fix 3: Increase spark.sql.autoBroadcastJoinThreshold to 100MB.
# Bad: SortMergeJoin shuffles both tables orders.join(products, on="product_id") # Good: BroadcastHashJoin — products sent to all executors, no shuffle orders.join(broadcast(products), on="product_id")
3
Wrong Partition Count — idle cores or scheduling overhead
Too few partitions: 4 partitions on a 16-core cluster leaves 12 cores idle. Too many partitions: 50,000 tiny partitions means more scheduling overhead than actual work.

Target: 128–256MB per partition. Formula: total data size (MB) ÷ 128 = good partition count.
repartition(n): full shuffle, balanced. Use to increase partitions or rebalance skewed data.
coalesce(n): no shuffle, cheap. Use only to reduce partitions (e.g. before writing to S3 to avoid small files).
4
Reading Too Much Data — full table scan when you need 1 day
Job reads 10TB from S3 but only processes 50GB. Caused by: no partition filter (Spark scans every folder), selecting all columns when you need 3.

Fix 1: Always filter on partition column early — filter(col("date") == "2024-01-15") before any joins.
Fix 2: Select only needed columns immediately — Parquet only reads selected columns, zero I/O for others.
Fix 3: Verify with df.explain(True) — look for "PushedFilters" in FileScan output.
Key Spark config settings for EMR
spark = SparkSession.builder .config("spark.sql.adaptive.enabled", "true") # AQE — enable always .config("spark.sql.adaptive.coalescePartitions.enabled", "true") # auto-tune partitions .config("spark.sql.adaptive.skewJoin.enabled", "true") # auto-fix skew .config("spark.sql.shuffle.partitions", "400") # tune to data size .config("spark.sql.autoBroadcastJoinThreshold", "52428800") # 50MB broadcast .config("spark.executor.memory", "16g") .config("spark.executor.cores", "4") # 4-5 max for S3 .config("spark.dynamicAllocation.enabled", "true") # scale up/down .getOrCreate()
AQE is the single most impactful config change. Adaptive Query Execution (Spark 3.0+, EMR 6.x) automatically coalesces small shuffle partitions, handles skew joins, and switches join strategies at runtime. Enabling it alone can halve job duration on skewed workloads.

Event-Driven Pipelines — Kafka & Kinesis

In traditional pipelines, you pull data on a schedule. In event-driven design, data pushes itself to you the moment something happens. The producer doesn't know or care who's listening.

Core principle: the order service doesn't call fraud, analytics, or notifications directly. It publishes ONE event to Kafka. All consumers subscribe independently. Adding a loyalty points service requires zero changes to the order service. This is loose coupling.
Kafka (MSK on AWS)
Topics: named streams (orders, payments, inventory).
Partitions: unit of parallelism — 12 partitions = 12 parallel consumers.
Partition key: use order_id as key → all events for same order go to same partition → ordering guaranteed per order.
Consumer groups: each group has independent offset. Fraud service and analytics service both read the same topic without interfering.
Retention: configurable, days to forever.
<5ms latencymulti-consumer groupsself-managed
Kinesis (AWS native)
Shards: equivalent to partitions — 1MB/s per shard.
Fan-out: up to 5 consumers per shard, 2MB/s each with enhanced fan-out.
Retention: default 24hrs, max 365 days (extra cost).
Integration: native Lambda triggers, Firehose to S3, Glue connectors — no config needed.
Management: fully managed, zero broker ops.
~70ms latencyAWS-nativezero ops
Manual offset commit — why it matters
consumer = KafkaConsumer( 'payments', group_id='fraud-detector-group', enable_auto_commit=False, # CRITICAL: commit manually after processing ) for message in consumer: try: score = get_fraud_score(message.value) write_to_dynamodb(message.value['txn_id'], score) consumer.commit() # commit ONLY after successful write # if we crash here, Kafka replays this message except Exception as e: send_to_dlq(message.value, str(e)) # dead letter queue consumer.commit() # avoid infinite retry loop
At-least-once delivery
Commit offset after processing. If crash before commit → Kafka replays. Event processed at least once, possibly twice. Your consumer must be idempotent — use event_id as DynamoDB dedup key. This is the standard choice.
Exactly-once delivery
Kafka transactions — complex, performance overhead. Use only for financial transactions where duplicates are unacceptable and idempotency is hard. enable.idempotence=true + transactional.id.

ML Pipeline: Training to Production

Training a model is 20% of the work. The other 80% is getting it to production safely, keeping it accurate, and letting the team iterate quickly.

1
Feature Store — offline + online
Offline store (S3): historical features for training. Point-in-time correct joins — features as they were at event time, not today's values.
Online store (DynamoDB): latest features for serving. Single GetItem call in <5ms. Updated by Spark Streaming after each event.
Why it matters: the same transformation code populates both stores. This prevents training-serving skew — the silent killer of ML models.
2
Model Training — SageMaker + MLflow
Critical: use time-based train/test split, never random. Random splitting on time-series data lets the model "see" future events during training (data leakage). Train on everything before date X, test on everything after.
MLflow logs: every training run — parameters, metrics, model artifacts, training dataset reference, Git commit. Enables full reproducibility and rollback.
with mlflow.start_run(): mlflow.log_param("training_rows", len(df)) mlflow.log_param("training_date", "2024-01-15") # Time-based split — NEVER random for time-series cutoff = int(len(df) * 0.8) X_train, X_test = X.iloc[:cutoff], X.iloc[cutoff:] model.fit(X_train, y_train) auc = roc_auc_score(y_test, model.predict_proba(X_test)[:,1]) mlflow.log_metric("auc_roc", auc) # Only register if better than current production model if auc > get_production_model_auc() + 0.003: mlflow.sklearn.log_model(model, "model") client.transition_model_version_stage(name, version, "Staging")
3
Deployment — never auto-deploy to production
Shadow mode: new model runs on 100% of traffic alongside old model, but results not shown to users. Compare outputs — if 99%+ agreement, safe to promote.
Canary: 5% of traffic to new model. Monitor CTR, error rate. If metrics equal or better after 24hrs, gradually increase: 5%→25%→50%→100%.
A/B test: split users (not just traffic) into stable cohorts. Measure business KPIs over 2–4 weeks. Use statistical significance testing before full deploy.
4
Monitoring + Drift Detection
Data drift: input feature distributions change. Average order value shifts, null rates spike. Detect with PSI (Population Stability Index) on each feature weekly. Alert if PSI > 0.2.
Concept drift: relationship between features and labels changes. Users who bought X no longer buy Y. Monitor prediction score distribution weekly.
Business metrics: the most important signal. Track CTR, conversion, revenue. A model with great AUC but falling CTR means offline metrics don't reflect real behaviour.
Retraining loop: drift alert → Airflow triggers SageMaker Pipeline → train → evaluate → Staging → human approval → Production.
Training-serving skew is the silent killer. If feature engineering code in training differs even slightly from serving — different null handling, different window aggregation — the model underperforms in production with no obvious error. A feature store with a single shared transformation definition prevents this entirely.

Case Study: Real-Time Fraud Detection

The hardest ML system design — sub-100ms latency, 99.9% uptime, model updates without downtime, and the cost of being wrong in either direction is high.

Core constraint: 100ms end-to-end. The payment must be approved or blocked before the user sees a response. This budget forces every design decision.
Latency budget — how 100ms is spent
Network + API Gateway
8ms

5ms
DynamoDB feature lookup
10ms
SageMaker ML scoring
40ms
Decision engine
5ms
Buffer remaining
32ms
Rule Engine (always first)
Hardcoded rules. Zero ML compute. <5ms.

• Velocity: >5 transactions in 1 hour
• Impossible travel: >500km in <60min
• Blocklisted card or merchant
• Amount >10× user's 30-day average
• New device + new merchant + high amount

Hard block on: impossible travel, blocklisted card. ML not consulted.
ML Model (XGBoost)
Catches subtle patterns rules miss. Features pre-computed in DynamoDB.

Score ≥ 0.85 → Block
Score 0.5–0.85 → Approve + flag for review
Score 0.3–0.5 + rules → Low priority review
Score < 0.3 → Approve silently (~95% of all txns)

Storage writes are async — non-blocking after response sent.
Key features fed to the model
Real-time (computed now):
Amount, merchant category, time of day, distance from last transaction, time since last transaction, is new merchant for this user
Historical (DynamoDB lookup):
Avg spend 7d/30d, transaction count last 1hr, distinct merchants 24hr, account age, previous fraud flags, KYC level, chargeback history
Fail open, not closed. If the ML endpoint goes down, fall back to rules-only and approve anything the rules don't catch. A blocked legitimate payment destroys customer trust instantly. A missed fraud transaction is a recoverable financial loss. Exception: impossible travel and blocklisted cards always block regardless of system state.
The class imbalance problem: only 0.1–0.5% of transactions are fraud. A model predicting "legitimate" for everything would be 99.9% accurate but useless. Never use accuracy. Use AUC-ROC, precision-recall, and F1. Oversample fraud cases or undersample legitimate ones during training to balance the dataset.

Case Study: Netflix-Style Recommendations

200M users, 15,000 titles, 200ms to load the homepage. The core insight: pre-compute candidates in batch, re-rank in real-time using the last 5 minutes of session context.

Why not score all 15,000 titles in real-time? Deep ranking model on 15,000 titles × 200M users × 50 requests/day = 150 trillion model calls/day. Impossible. Solution: two-stage funnel — fast retrieval narrows 15,000 → 500 offline, precise ranker scores 500 in real-time.
1
Candidate Generation — ANN on embeddings 15,000 → 500
Two-tower neural network produces a 128-dim embedding for every user and title. FAISS HNSW index stores all 15,000 title embeddings. ANN search finds the 500 most similar titles to the user's embedding in <10ms. Runs offline nightly, results stored per user in DynamoDB.
2
Ranking — precise scoring at request time 500 → 100
For each of the 500 candidates, XGBoost predicts P(user watches this title | user features + title features + context). Features: watch history, time of day, device, title genre/popularity, user-title affinity score. Includes session context — what the user watched 5 minutes ago.
3
Business Rules + Diversity 100 → 40
Remove watched titles (last 30 days). Max 3 titles per genre in top 10. Boost new releases and expiring licensed content. Apply parental controls and regional restrictions. In-memory, <5ms, no ML needed.
Embeddings — user and item
User embedding (128-dim): watch history, genres, ratings, time-of-day patterns. Updated daily via PySpark batch.

Item embedding (128-dim): genre, cast, director, NLP synopsis, viewing patterns across all users. Updated weekly.

Users and titles that are "similar" have vectors that are close in the 128-dim space. Finding candidates = nearest neighbour search.
Cold Start Problem
New user: show onboarding (pick 3 genres). Map to content-based embedding. Blend content (60%) + collaborative (40%). Improves over first 2 weeks.

New title: content-based embedding from metadata. Inherit collaborative embedding from 5 most similar existing titles. Show to 1% of matched users to collect signal (exploration budget).
Filter bubble prevention: diversity rules enforce genre breadth. "Something different" row has an explicit diversity budget. Novelty bonus in ranker gives small score boost to content unlike the user's recent 10 watches. Short-term CTR goes up with filter bubbles — but 30-day retention goes down. Always measure retention, not just CTR.
Interview answer on metrics: "I'd never use accuracy — the dataset is 99.9% legitimate. Use AUC-ROC for offline eval. For production, A/B test CTR, watch start rate, and 30-day retention on stable user cohorts over 2 weeks. The correlation between offline AUC improvement and online CTR improvement tells you whether your evaluation setup is trustworthy."

Case Study: Semantic Product Search

Traditional search matches keywords. Semantic search matches meaning. A user searching "comfortable shoes for standing all day" should find "orthopedic sneakers with memory foam" — zero word overlap.

Keyword search fails on
• "sofa" ≠ "couch" (synonyms)
• "gift for 5-year-old" → 0 results (intent)
• "runing shoes" (typos)
• "comfortable work shoes" (descriptive)

Good for: "Samsung Galaxy S24 128GB Black", "SKU-48291-XL-RED" (exact codes)
Semantic search handles all of these
• "sofa" → couch, settee, loveseat ✓
• "gift for 5-year-old" → toys, books, LEGO ✓
• "runing shoes" → running shoes ✓
• "comfortable work shoes" → anti-fatigue clogs ✓

Fails on: product codes and SKUs (no semantics in a code)
Critical rule — same model for queries and products. If you embed products with model A and queries with model B, the vectors live in different spaces — cosine similarity is meaningless. Always use the same model for both. When you upgrade the model, re-embed all 10M products before going live.
Query pipeline — 100ms end-to-end
1
Query understanding — detect intent <5ms
Product code? → weight keyword. Natural language? → weight semantic. Extract embedded filters ("red dress under 2000").
2
Embed query <20ms
sentence-transformers/all-MiniLM → 768-dim vector. Same model, same preprocessing as products.
3
ANN search in FAISS <10ms
HNSW index, 10M product vectors in memory. Returns 200 nearest products by cosine similarity.
4
Cross-encoder re-rank <30ms
Sees query + product together. Much more accurate than bi-encoder dot product. Scores all 200 candidates.
5
Personalisation + rules → top 20 <5ms
Boost user's past categories, in-stock, high-rated. Diversity rules: max 4 per brand. Return with facets.
Hybrid search — Reciprocal Rank Fusion
# Merge keyword (BM25) + semantic (ANN) results # alpha controls weighting: 0.8 = mostly semantic, 0.2 = mostly keyword def rrf(keyword_results, semantic_results, k=60, alpha=0.5): scores = {} for rank, p in enumerate(keyword_results, start=1): scores[p['id']] = scores.get(p['id'], 0) + (1-alpha) * (1/(k+rank)) for rank, p in enumerate(semantic_results, start=1): scores[p['id']] = scores.get(p['id'], 0) + alpha * (1/(k+rank)) return sorted(scores, key=scores.get, reverse=True) # Detect query type for dynamic alpha def detect_alpha(query): if re.match(r'^[A-Z0-9\-]{5,}$', query.upper()): return 0.1 # product code elif len(query.split()) > 4: return 0.8 # natural language else: return 0.5 # balanced
Bi-encoder vs Cross-encoder: bi-encoder embeds query and document independently — fast because product embeddings are pre-computed. Cross-encoder sees query + document together — much more accurate but too slow for 10M products. Solution: bi-encoder for retrieval (10M → 200), cross-encoder for re-ranking (200 → 20). This is the same two-stage funnel as Netflix recommendations.
This IS the RAG architecture. Semantic search's vector index + ANN retrieval is identical to the retrieval component of every RAG (Retrieval-Augmented Generation) system. The only difference: search returns results directly to the user; RAG passes them to an LLM as context. Understanding semantic search means you understand how every modern LLM-based system retrieves information.

Interview Framework

A systematic method for any pipeline design question. Interviewers aren't testing whether your design is perfect — they're testing whether you think like an engineer.

1
Clarify requirements 2–3 min
Never skip this. Ask: batch or real-time? Data volume (GB/day or TB/day)? How fresh does output need to be? Who consumes it — analysts, ML, dashboards? What if the pipeline fails — is data loss acceptable? Expected growth in 6 months?

Say: "Before I start designing, can I ask a few questions? Is the primary use case real-time or batch? What's the approximate data volume? And how stale is too stale for the output?"
2
Estimate scale 2 min
Numbers anchor every decision. Calculate: events/second at peak (drives streaming vs batch), daily data in GB/TB (drives storage and compute), number of concurrent readers (drives serving layer).

Say: "1M users × 10 events/day = 10M events/day = ~115 events/second. At 5KB each, that's ~50GB/day. That tells me we can use batch — 50GB/day doesn't justify streaming unless the latency requirement demands it."
3
High-level design — source → process → serve 5–8 min
Use the three-layer framing out loud: ingestion (Kinesis + Firehose), processing (EMR PySpark), serving (Redshift Spectrum + Athena). This maps naturally to Medallion architecture which most interviewers will recognise.

Say: "I'd structure this as three layers. Ingestion — events via Kinesis land raw in S3. Processing — PySpark on EMR validates, cleans, aggregates. Serving — Redshift Spectrum queries Parquet directly so analysts get SQL without data duplication."
4
Deep dive — pick 2 components 5–8 min
Don't wait to be asked. Pick two components yourself and go deep — schema and partitioning strategy, data quality rules, fault tolerance, scaling story.

Say: "Let me go deeper on two areas. First, S3 partitioning — I'd partition by date then region, so 'sales in Mumbai last week' opens 7 folders not 3 years of data. Second, data quality — validation between Bronze and Silver with quarantine for bad rows and a CloudWatch alarm that blocks the Gold job if pass rate drops below 95%."
5
Trade-offs — name what you gave up 2–3 min
Naming trade-offs unprompted is the single biggest signal of seniority. Every choice has a cost — say what it is.

Say: "I chose Medallion over Lambda because it's one codebase for a small team with no hard real-time requirement. The downside is our Gold layer is up to 1 hour old. If the business later needs real-time metrics, I'd add Kinesis writing into Bronze — Silver and Gold jobs stay unchanged. I'd also highlight that Redshift Spectrum is slower than native Redshift for the most frequent queries — if dashboards start timing out, we'd load those specific tables into Redshift proper."
COMMON MISTAKES TO AVOID
❌ Don't do this
• Jump into architecture without clarifying
• Say "use streaming" without justifying why
• Design only the happy path
• Say "add more machines" when asked to scale
• Drop bad data with filter() silently
• Design in silence with no commentary
• Pick one architecture without acknowledging alternatives
• Forget orchestration and monitoring
✓ Do this instead
• Spend 2 min on requirements — it's free marks
• Justify every choice: "I chose X because Y"
• Proactively cover failure modes
• Check Spark UI first, diagnose, then fix
• Quarantine bad rows with dq_reason column
• Think out loud — reasoning is what's scored
• Name the alternatives you rejected and why
• Always mention Airflow scheduling + CloudWatch alerts
