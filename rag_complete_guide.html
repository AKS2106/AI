RAG complete guide — pipeline overview, chunking strategies, embeddings, semantic search, and interview prep

RAG has two distinct phases. You run the indexing phase once (or on update). The query phase runs on every user question.
Phase 1 — indexing (done once, offline)
1
Load documents
PDFs, web pages, databases, Notion, Confluence — anything with text. LangChain has 80+ loaders.
2
Chunk into pieces
Split each doc into overlapping segments (~500 tokens each). Too big = too much noise. Too small = loses context.
3
Embed each chunk
Run every chunk through an embedding model (e.g. text-embedding-3-small). Output: a vector of ~1536 floats per chunk.
4
Store in vector DB
Save (vector, original text, metadata) to Chroma, Pinecone, Weaviate, or pgvector. Indexed for fast similarity search.
Phase 2 — query (runs on every user question)
5
Embed the question
Run the user's question through the same embedding model. Now it's a vector in the same space as your chunks.
6
Retrieve top-k chunks
Find the k chunks whose vectors are closest (highest cosine similarity) to the question vector. Typically k=3–5.
7
Augment the prompt
Inject the retrieved chunks into the LLM prompt as context: "Answer using only this context: [chunks]. Question: [question]"
8
Generate answer
LLM reads the context and answers. It can cite sources. It can say "I don't know" when the context doesn't contain the answer.
The key insight: the LLM never needs to memorize your data. It just needs to read and summarize the relevant passage you hand it. This means your knowledge base can be updated at any time without retraining.
Chunking is the most underrated decision in a RAG system. Bad chunking breaks retrieval even with a perfect embedding model.

The 4 chunking strategies compared
Fixed-size with overlap — split every N characters, overlap by M. Simple, predictable. The overlap prevents losing context at boundaries. LangChain's default. Best for: homogeneous documents (articles, reports).
Sentence/paragraph aware — split on natural language boundaries (periods, newlines). Chunks stay semantically coherent. Best for: conversational text, Q&A docs, anything with natural structure.
Recursive character splitting — try paragraph breaks first, fall back to sentence breaks, then word breaks, then characters. LangChain's RecursiveCharacterTextSplitter. Best for: mixed content.
Semantic chunking — embed each sentence, group sentences with similar embeddings together. Highest quality, slowest. Best for: technical docs where topics change mid-paragraph.
Interview trap: "What chunk size should I use?" — there is no universal answer. The right size depends on your embedding model's max input, your document type, and how specific your questions are. Always run evals with 20–50 representative questions and measure retrieval accuracy at different chunk sizes.

Cosine similarity — how "closeness" is measured
Cosine similarity measures the angle between two vectors — not their distance. Two vectors pointing in the same direction = similarity 1.0. Opposite directions = –1.0. Perpendicular = 0.0.
import numpy as np def cosine_similarity(a, b): return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)) # Example: vec_query = embed("What is the return policy?") vec_chunk = embed("Our refund window is 30 days") score = cosine_similarity(vec_query, vec_chunk) # → 0.87 (high similarity — good retrieval candidate)
Choosing an embedding model
API-based (easier)
text-embedding-3-small — OpenAI, 1536-dim, $0.02/1M tokens. Best value.
text-embedding-3-large — OpenAI, 3072-dim, higher accuracy for complex domains.
voyage-3 — Anthropic-recommended, strong on retrieval benchmarks.
Self-hosted (more control)
BAAI/bge-m3 — multilingual, runs on CPU, free. Good default for open-source stacks.
nomic-embed-text — strong performance, Apache 2.0 license.
Always use the same model at index time AND query time. Mismatched models = garbage results.

Beyond basic vector search — advanced retrieval
Hybrid search — combine semantic (vector) + keyword (BM25) scores. Semantic finds conceptually similar chunks; BM25 finds exact term matches. Weighted sum of both scores wins on most benchmarks. Use rank_fusion in Weaviate or EnsembleRetriever in LangChain.
Reranking — retrieve top-20 cheaply, then run a cross-encoder reranker to pick the best 3–5. Cross-encoders compare query + chunk together (more accurate but slower). Cohere Rerank, BGE-reranker. A single reranking step often improves RAG quality more than any other optimization.
Metadata filtering — pre-filter by metadata before vector search. "Only search chunks from documents tagged 'billing', uploaded after 2024-01-01". Faster and more precise. Every chunk should store metadata: source, date, section, doc_type.
Query expansion / HyDE — before searching, ask the LLM to generate a hypothetical answer to the question. Embed that answer and search with it. Works because a hypothetical answer uses the same vocabulary as the relevant chunks, improving recall on abstract questions.
Interview answer for "how do you improve RAG quality?": (1) Better chunking — semantic chunking or parent-doc retriever. (2) Reranking — cross-encoder after vector retrieval. (3) Hybrid search — add BM25. (4) Metadata filtering — reduce the search space. (5) Query rewriting — fix ambiguous questions before embedding. In that order of ROI.
The 5 questions you will definitely get asked
Q: "When would you use RAG vs fine-tuning?"
"RAG when the knowledge changes frequently (product docs, internal wikis, news) or when you need source citations. Fine-tuning when you need consistent style/behavior, not new knowledge — fine-tuning teaches the model HOW to talk, not WHAT to know. In practice: use RAG for knowledge, fine-tuning for format."
Q: "What causes bad RAG retrieval and how do you fix it?"
"Three main causes: (1) Chunks are too large — dilutes the relevant signal. Fix: smaller chunks + parent doc retriever. (2) Query-chunk mismatch — the question uses different vocabulary than the answer. Fix: HyDE, query expansion, or hybrid search. (3) Wrong embedding model — used a generic model on specialized domain text. Fix: domain-specific embedder or fine-tune the embedder."
Q: "How do you evaluate a RAG pipeline?"
"Two separate evals: (1) Retrieval eval — given a question, did we retrieve the right chunks? Measure recall@k and MRR. (2) Generation eval — given the right chunks, did the LLM answer correctly? Use RAGAS framework: measures faithfulness (answer grounded in context), answer relevancy, and context precision/recall. The key insight: retrieval and generation failures have different fixes."
Q: "What vector databases have you used, and how do you choose?"
"For prototyping: Chroma — local, no server, zero config. For production at moderate scale: Weaviate or Qdrant — self-hostable, strong hybrid search, good filtering. For large scale with managed infra: Pinecone — fully managed, simple API. For teams already on Postgres: pgvector — no new infra, good enough for <1M vectors. The choice is mostly about operational complexity, not capability."
Q: "What is the 'lost in the middle' problem?"
"LLMs pay less attention to context in the middle of a long prompt — they prioritize beginning and end. So if you stuff 10 retrieved chunks into the prompt, the most relevant chunk might be chunk 5 and the LLM partially ignores it. Fix: (1) Retrieve fewer, higher-quality chunks (k=3 not k=10). (2) Put the most relevant chunk first and last. (3) Use reranking to ensure quality before insertion."
The answer that impresses senior interviewers
"The hardest part of RAG isn't the vector search — it's the evaluation loop. Anyone can wire up ChromaDB in an afternoon. What separates production RAG from a demo is: a labeled eval set of 100+ question/answer pairs, automated metrics (RAGAS or custom), and a CI pipeline that runs the eval on every change to chunking strategy, embedding model, or retrieval params. Without this, you're flying blind."
