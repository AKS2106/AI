Window Functions
The most tested advanced PySpark topic
Core idea: GROUP BY collapses rows. Window functions keep all rows and ADD a new column. PARTITION BY = reset per group. ORDER BY = define sequence.
FunctionWhat it doesWhen to usePySpark
ROW_NUMBER()Unique number, no tiesDeduplication — keep one row per grouprow_number().over(w)
RANK()Ties get same rank, gaps afterRarely used in practicerank().over(w)
DENSE_RANK()Ties same rank, no gapsNth highest, Top N per groupdense_rank().over(w)
LAG(col, n)Value from n rows beforeDay-over-day change, self-comparisonlag("col").over(w)
LEAD(col, n)Value from n rows afterGap detection, next valuelead("col").over(w)
AVG() OVERGroup average per rowCompare to dept averageavg("col").over(w)
SUM() OVERRunning totalCumulative calculationssum_("col").over(w)

Date-Based Query Patterns
All 8 date questions from your interview list solved
Pattern A Activity in time window — visited >3 times, purchased once in 30 days ▼
SQL
SELECT user_id, COUNT(*) AS visits FROM visits WHERE visit_date >= CURRENT_DATE - 7 GROUP BY user_id HAVING COUNT(*) > 3;
PySpark
df.filter(datediff(current_date(), col("visit_date")) <= 7) \ .groupBy("user_id").count().filter(col("count") > 3)
Pattern B Absence — not logged in 3 days, not contacted in 14 days ▼
SQL — LEFT JOIN + IS NULL
SELECT u.user_id FROM users u LEFT JOIN logins l ON u.user_id = l.user_id AND l.login_date >= CURRENT_DATE - 3 WHERE l.user_id IS NULL;
PySpark — left_anti join
recent = logins.filter(datediff(current_date(),"login_date") <= 3) \ .select("user_id").distinct() users.join(recent, "user_id", "left_anti")
Pattern C Consecutive days — logged in 3 consecutive days, orders on consecutive days ▼
Key trick: date - ROW_NUMBER = same value for consecutive dates. Group by that value and count.
WITH daily AS ( SELECT user_id, CAST(login_date AS DATE) AS day FROM logins GROUP BY 1, 2 -- dedup first ), with_rn AS ( SELECT user_id, day, DATE_SUB(day, ROW_NUMBER() OVER ( PARTITION BY user_id ORDER BY day )) AS streak_group FROM daily ) SELECT DISTINCT user_id FROM with_rn GROUP BY user_id, streak_group HAVING COUNT(*) >= 3; -- 3+ consecutive days

Key Answers Per Bucket
The most important question from each bucket — with the answer to say
B1How do you handle a corrupt file in S3?▼
df = spark.read.option("mode","PERMISSIVE") \ .option("columnNameOfCorruptRecord","_corrupt") \ .json("s3://bucket/raw/") valid_df = df.filter(col("_corrupt").isNull()).drop("_corrupt") bad_df = df.filter(col("_corrupt").isNotNull()) \ .withColumn("source_file", input_file_name()) \ .withColumn("flagged_at", current_timestamp()) valid_df.write.mode("append").parquet("s3://processed/") bad_df.write.mode("append").json("s3://quarantine/")
Say this
"PERMISSIVE mode captures bad rows in _corrupt_record. I split into valid and quarantine DataFrames — valid goes to processed, bad goes to quarantine with source filename and timestamp. I alert if bad rate exceeds 5%."
B2How do you handle skewed data?▼
# Fix 1: broadcast join (small table) result = df_large.join(broadcast(df_small), "country") # Fix 2: salting (both tables large) df_large = df_large.withColumn("salt", (rand()*10).cast("int").cast("string")) \ .withColumn("key", concat(col("country"),lit("_"),col("salt")))
Say this
"Skew means one partition has far more data. I detect it by checking partition sizes. Fix: broadcast join if one table is small. If both large, use salting — add a random number to the join key so the hot key spreads across partitions."
B3How do you implement incremental processing?▼
# Approach: watermark in DynamoDB last_ts = get_watermark("orders_pipeline") # from DynamoDB df = spark.read.parquet("s3://bucket/orders/") \ .filter(col("updated_at") > last_ts) # Process and write df.write.mode("overwrite").partitionBy("dt").parquet("s3://processed/") # Save new watermark AFTER successful write new_ts = df.agg(max_("updated_at")).collect()[0][0] save_watermark("orders_pipeline", new_ts)
Say this
"Three approaches: partition filter for date-partitioned S3, watermark stored in DynamoDB for tables with updated_at, or Glue Job Bookmarks where AWS manages state automatically. Always save the watermark AFTER a successful write — not before."
B5What is the difference between Glue DynamicFrame and DataFrame?▼
# DynamicFrame for the doors (I/O) dyf = glueContext.create_dynamic_frame.from_catalog( database="db", table_name="orders", transformation_ctx="orders" # required for bookmarks ) df = dyf.toDF() # convert IMMEDIATELY # DataFrame for everything inside (all transforms) df = df.filter(col("status")=="active") \ .withColumn("year", year("order_date")) # Convert back only for output dyf_out = DynamicFrame.fromDF(df, glueContext, "output") # ... write ... job.commit() # CRITICAL — advances bookmarks
Say this
"DynamicFrame for the doors, DataFrame for everything inside. DynamicFrame handles messy schemas at I/O. DataFrame has the full Spark API for transforms. Always add transformation_ctx and always end with job.commit() — without it bookmarks never advance."
B7Delta Lake vs Parquet — when to use each?▼
# Delta Lake = Parquet files + _delta_log/ folder # That log gives you: ACID, MERGE, time travel, schema evolution # MERGE — upsert pattern target.alias("t").merge( updates.alias("s"), "t.order_id = s.order_id" ).whenMatchedUpdateAll().whenNotMatchedInsertAll().execute() # Time travel — query past version df = spark.read.format("delta") \ .option("timestampAsOf","2024-01-14") \ .load("s3://bucket/delta/orders/")
Say this
"Parquet is a file format — fast reads, append-only. Delta Lake is Parquet files plus a _delta_log folder. That log gives ACID transactions, MERGE for upserts, and time travel. Plain Parquet for simple append-only pipelines. Delta when I need corrections, history, or schema evolution."
