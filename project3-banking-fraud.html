Business Problem

🚨
A mid-size bank processes 10,000+ card transactions per minute. Their current fraud detection runs as a nightly batch job — meaning fraudulent charges aren't caught until the next morning. By then, the criminal has already made 50 more transactions. The bank needs real-time fraud detection that flags suspicious transactions within 5 seconds so the card can be blocked before more damage is done. Fraud costs the bank $12M annually — catching it in real-time reduces losses by an estimated 60%.

The Core Challenge

🌊

Instead of reading a file and writing a file, you read from a stream, apply rules with windows (time-based groups of events), and write alerts in real-time. Stream Processing 101

Everything You Need to Know About Streaming

Master these concepts before writing a single line of code. Interviewers test theory heavily here.

①
Batch vs Stream vs Micro-batch
The three processing paradigms
▼

Batch: Collect all data → process → output. Latency: minutes to hours. Tools: PySpark, AWS Glue, dbt. Use for: daily reports, ETL pipelines, ML training.

True streaming: Process each event individually, record by record, as it arrives. Latency: milliseconds. Tools: Apache Flink, Kafka Streams. Use for: fraud detection requiring <100ms response, high-frequency trading.

Micro-batch (what we use): Buffer events for a short window (10 seconds), then process the whole mini-batch at once. Latency: seconds. Tools: Spark Structured Streaming. Use for: fraud detection where seconds are acceptable, near-real-time dashboards.

💡
Interview answer: "I used Spark Structured Streaming with 10-second micro-batches. True streaming (Flink) offers sub-second latency but adds operational complexity. For fraud detection where a 5-second alert is acceptable, micro-batch gives us simpler fault tolerance and lets us reuse our Spark expertise from the batch pipelines."
②
Windows: Tumbling, Sliding, Session
The core concept of stream aggregation
▼

Tumbling window: Fixed-size, non-overlapping windows. "Count transactions per customer per 5-minute block." 0:00–5:00 is one window, 5:00–10:00 is the next. Each event belongs to exactly one window.

Sliding window: Fixed-size windows that overlap. "Count transactions in any 5-minute rolling period." A window slides forward every 1 minute — so events can appear in multiple windows. More expensive but catches patterns that fall on window boundaries.

Session window: Variable-size windows bounded by inactivity. "Group all activity within a session (no 30-minute gap)." Duration varies per user. Used for web analytics, not typically fraud detection.

⚠️
For fraud detection we use sliding windows: A fraudster making 5 transactions at 4:58, 4:59, 5:00, 5:01, 5:02 would be split across two tumbling windows (missing the pattern) but caught in a sliding window. The extra cost is worth it for fraud.
③
Watermarks and Late Data
How streaming handles out-of-order events
▼

In real systems, events arrive out of order. A mobile transaction might take 3 seconds to reach Kafka due to network latency. A watermark tells Spark: "Wait up to X seconds for late-arriving events before closing a window."

PySpark — Watermark Example
# Tell Spark: wait up to 10 seconds for late events
# Events more than 10 seconds late will be dropped
df_with_watermark = transactions_stream.withWatermark("event_time", "10 seconds")

# Now window aggregation respects the watermark
windowed = (df_with_watermark
    .groupBy(
        F.window("event_time", "5 minutes", "1 minute"),  # 5-min window, slide 1 min
        "account_id"
    )
    .agg(
        F.count("transaction_id").alias("tx_count"),
        F.sum("amount").alias("total_amount")
    )
)
④
Checkpointing and Fault Tolerance
How streams survive failures
▼

Streaming jobs run 24/7. They will fail — the server reboots, network blips, OOM errors. Checkpointing saves the stream's progress to S3 periodically. When the job restarts, it reads the checkpoint and picks up exactly where it left off — no events processed twice, no events missed.

PySpark — Checkpoint Setup
query = (windowed.writeStream
    .outputMode("update")           # only write changed rows
    .format("parquet")
    .option("checkpointLocation", "s3://fraud-checkpoints/windowed/")
    .trigger(processingTime="10 seconds")  # micro-batch every 10s
    .start("s3://fraud-output/windowed/")
)

Output modes:

  • append — only write new rows (stateless ops only)
  • update — only write rows that changed (most common for aggregations)
  • complete — write the full result table every trigger (expensive, small tables only)
⑤
Exactly-Once vs At-Least-Once Semantics
▼

At-most-once: Events may be lost on failure. Never used for financial data.

At-least-once: Events will be processed, but may be processed more than once on failure. You need idempotent sinks to handle duplicates. Used when losing events is worse than duplicating them.

Exactly-once: Every event is processed exactly once even through failures. Requires both source and sink to support it. Kafka + Spark Structured Streaming support exactly-once end-to-end. This is what we use for banking — double-processing a fraud alert or missing one are both unacceptable.

Apache Kafka Deep Dive

Kafka — Everything for Interviews

Kafka is the most commonly asked streaming technology in Data Engineer interviews. Master the vocabulary and you'll stand out.

K1
What Kafka Is and Why It Exists
▼

Kafka is a distributed message queue — a durable, ordered log of events. Think of it as a highway between systems: the bank's transaction system publishes transactions to Kafka without waiting for fraud detection to process them. Multiple consumers (fraud detection, analytics, compliance) can read the same events at their own pace. This decoupling means: if fraud detection is slow or down, transactions keep flowing and the backlog is replayed when it comes back up.

💡
The key insight: Kafka is not a database, not a queue that deletes messages when read. It's a log — messages are retained for a configurable period (days/weeks) and any consumer can re-read from any point.
K2
Topics, Partitions, Offsets
The three vocabulary words you MUST know
▼

Topic: A named stream of events. Like a table in a database, but append-only. Example: "bank-transactions" is a topic.

Partition: A topic is split into N partitions. Each partition is an ordered, immutable log. Partitions are the unit of parallelism — more partitions = more consumers reading in parallel. A message goes to exactly one partition, determined by its key (we use account_id as key, so all transactions for one account go to the same partition → guaranteed ordering per account).

Offset: A sequential ID for each message within a partition. Consumer tracks its offset to know where it is. Replay from any offset by resetting the consumer offset.

K3
Producers, Consumers, Consumer Groups
▼

Producer: Writes events to a topic. In our project: the transaction simulator publishes to "bank-transactions". Sets the partition key (account_id) to ensure ordering.

Consumer: Reads events from a topic. Tracks its own offset. Can replay by resetting offset.

Consumer group: Multiple consumer instances sharing the work. If topic has 6 partitions and consumer group has 3 instances, each instance reads 2 partitions. This is horizontal scaling for consumers. Each partition is consumed by exactly one instance in the group — no double processing.

⚠️
Interview trap: If you add more consumer instances than partitions, the extra instances sit idle. Consumers ≤ partitions for efficient scaling.
K4
Replication Factor and Fault Tolerance
▼

Each partition has one leader and N-1 replicas. All reads and writes go through the leader. Replicas stay in sync. If the leader broker dies, a replica is elected as the new leader automatically. With replication factor 3, the cluster tolerates 2 broker failures without data loss.

For local dev: We use replication factor 1 (single broker). For production banking: replication factor 3, minimum ISR (in-sync replicas) of 2.

K5
Schema Registry and Avro
How Kafka handles schema evolution
▼

JSON Kafka messages have a problem: if you add a new field to the transaction schema, old consumers break because they don't know about the new field. Schema Registry + Avro solves this with schema evolution:

  • Avro is a binary serialization format with an explicit schema
  • Schema Registry stores all schema versions centrally
  • Producers register the schema before sending — each message includes a schema ID
  • Consumers fetch the schema by ID — they can handle old and new schemas
  • Evolution rules: adding a field with a default is BACKWARD compatible (old consumers skip it). Removing a field is FORWARD compatible. Both = FULL compatible.

For this project: We simulate schema evolution in Step 7 — we start with JSON, then add a new field, and show the consumer handles both versions.

Architecture · Real-Time + Historical

Dual-Path Pipeline Architecture

Two simultaneous paths: a hot path for real-time alerts (under 5 seconds) and a cold path for historical analysis in S3/Athena.

[Bank Core System]
        ↓ produces transactions
[Kafka Topic: bank-transactions] (6 partitions, key=account_id)
        ↓ consumed by
[Spark Structured Streaming] (micro-batch every 10 seconds)
        ↓
[Apply fraud detection rules]
  ├── CLEAN transactions → S3 (Parquet, partitioned) → Athena
  └── FLAGGED transactions → Kafka Topic: fraud-alerts
                            ↓
                    [AWS Lambda consumer]
                            ↓
                    [DynamoDB: fraud_alerts table] ← real-time lookup
                    [SNS → SMS/Email alert to fraud team]
                    [CloudWatch metric → dashboard]

Why Two Paths?

🔴 Hot Path (Real-time)
Flagged transactions → Lambda → DynamoDB + SNS alert. Goal: under 5 seconds. No complex transforms — just flag and alert. DynamoDB handles 100K+ reads/sec for card authorization checks.
🟢 Cold Path (Historical)
All transactions → S3 Parquet → Athena. Goal: full history for analysts. Used for: fraud pattern analysis, model training, regulatory reporting, trend dashboards.

Kafka Transaction Producer

Simulates the bank's transaction system publishing events to Kafka. Generates a mix of normal and fraudulent patterns so our detection rules have something to catch.

producers/transaction_producer.py
"""
Kafka Transaction Producer
Simulates bank transactions — mix of legitimate and fraudulent patterns.
Run: python3 producers/transaction_producer.py
Press Ctrl+C to stop.
"""
import json, uuid, random, time, hashlib
from datetime import datetime
from kafka import KafkaProducer
from faker import Faker

fake = Faker()
random.seed(42)

KAFKA_BOOTSTRAP = "localhost:9092"
TOPIC           = "bank-transactions"

# ── Synthetic data pools ─────────────────────────────────
ACCOUNTS     = [f"ACC-{str(uuid.uuid4())[:8].upper()}" for _ in range(200)]
MERCHANTS    = [f"MER-{str(uuid.uuid4())[:8].upper()}" for _ in range(500)]
CATEGORIES   = ["grocery","restaurant","gas","electronics","travel","retail","pharmacy"]
COUNTRIES    = ["US","US","US","US","CA","GB","MX","CN","RU"]
CHANNELS     = ["in-person","in-person","online","mobile","ATM"]

def hash_card(card_num: str) -> str:
    return hashlib.sha256(card_num.encode()).hexdigest()

def make_normal_transaction(account_id: str) -> dict:
    """Generate a typical everyday transaction."""
    return {
        "transaction_id":   str(uuid.uuid4()),
        "account_id":       account_id,
        "card_number_hash": hash_card(f"4111{account_id[:8]}"),
        "merchant_id":      random.choice(MERCHANTS),
        "merchant_category": random.choice(CATEGORIES),
        "amount":           round(random.uniform(5, 250), 2),
        "currency":         "USD",
        "transaction_type": "purchase",
        "channel":          random.choice(CHANNELS),
        "country":          "US",
        "latitude":         round(random.uniform(25, 48), 4),
        "longitude":        round(random.uniform(-125, -66), 4),
        "event_time":       datetime.utcnow().isoformat(),
        "kafka_ingest_time": datetime.utcnow().isoformat(),
    }

def make_velocity_fraud(account_id: str) -> list:
    """Simulate velocity fraud: 6 rapid transactions on one account."""
    now = datetime.utcnow()
    transactions = []
    for i in range(6):
        transactions.append({
            "transaction_id":   str(uuid.uuid4()),
            "account_id":       account_id,
            "card_number_hash": hash_card(f"4111{account_id[:8]}"),
            "merchant_id":      random.choice(MERCHANTS),
            "merchant_category": "electronics",
            "amount":           round(random.uniform(800, 2000), 2),
            "currency":         "USD",
            "transaction_type": "purchase",
            "channel":          "online",
            "country":          random.choice(["US","CN","RU"]),
            "latitude":         round(random.uniform(25, 48), 4),
            "longitude":        round(random.uniform(-125, -66), 4),
            "event_time":       now.isoformat(),
            "kafka_ingest_time": now.isoformat(),
        })
    return transactions

def make_card_testing_fraud(account_id: str) -> list:
    """Card testing: small micro-transactions to verify card works."""
    now = datetime.utcnow()
    return [{
        "transaction_id":   str(uuid.uuid4()),
        "account_id":       account_id,
        "card_number_hash": hash_card(f"4111{account_id[:8]}"),
        "merchant_id":      random.choice(MERCHANTS),
        "merchant_category": "retail",
        "amount":           round(random.uniform(0.50, 5.00), 2),  # Tiny amounts
        "currency":         "USD",
        "transaction_type": "purchase",
        "channel":          "online",
        "country":          "US",
        "latitude":         40.7128,
        "longitude":        -74.0060,
        "event_time":       now.isoformat(),
        "kafka_ingest_time": now.isoformat(),
    } for _ in range(4)]

def main():
    producer = KafkaProducer(
        bootstrap_servers=KAFKA_BOOTSTRAP,
        value_serializer=lambda v: json.dumps(v).encode("utf-8"),
        key_serializer=lambda k: k.encode("utf-8"),
    )
    print(f"Producer started. Sending to topic: {TOPIC}")
    print("Press Ctrl+C to stop.\n")

    tx_count = 0
    fraud_count = 0

    while True:
        account = random.choice(ACCOUNTS)

        # 95% normal, 5% fraudulent patterns
        roll = random.random()
        if roll < 0.03:
            # Velocity fraud burst
            txs = make_velocity_fraud(account)
            fraud_count += len(txs)
        elif roll < 0.05:
            # Card testing pattern
            txs = make_card_testing_fraud(account)
            fraud_count += len(txs)
        else:
            txs = [make_normal_transaction(account)]

        for tx in txs:
            producer.send(TOPIC, key=tx["account_id"], value=tx)
            tx_count += 1

        producer.flush()

        if tx_count % 100 == 0:
            print(f"Sent: {tx_count} total | {fraud_count} fraud patterns | "
                  f"Rate: ~{tx_count} tx so far")

        time.sleep(random.uniform(0.05, 0.2))  # Realistic rate

if __name__ == "__main__":
    main()

Step 4 · Spark Structured Streaming

The Core Streaming Job

Reads from Kafka, parses JSON, computes sliding window aggregations, applies fraud rules, and writes to two sinks: S3 (all transactions) and fraud-alerts topic (flagged only).

⚠️
This job needs the Kafka-Spark connector JAR. Before running, download it:
pip3 install pyspark[kafka] or add the package at spark-submit time with --packages org.apache.spark:spark-sql-kafka-0-10_2.12:3.5.1

Fraud Detection Spark Structured Streaming Job
Reads bank-transactions from Kafka → detects fraud → writes alerts

Run (local):
spark-submit \
  --packages org.apache.spark:spark-sql-kafka-0-10_2.12:3.5.1 \
  spark_jobs/fraud_detection_stream.py

    StructField("card_number_hash", StringType(),    True),
    StructField("merchant_id",      StringType(),    True),
    StructField("merchant_category",StringType(),    True),
    StructField("amount",           DoubleType(),    True),
    StructField("currency",         StringType(),    True),

def apply_fraud_rules(df):
    """
    Apply rule-based fraud scoring.
    Each rule adds to a composite fraud_score.
    Score >= 0.6 → alert triggered.
    """
    scored = (df
        # Rule 1: Velocity — too many transactions in 5 min
        .withColumn("rule_velocity",
            F.when(F.col("tx_count_5min") >= 5, 0.4).otherwise(0.0))

        # Rule 2: High value — single large transaction
        .withColumn("rule_high_amount",
            F.when(F.col("max_single_amount") > 5000, 0.3).otherwise(0.0))

        # Rule 3: Geo impossible — multiple countries in same window
        .withColumn("rule_geo_impossible",
            F.when(F.col("distinct_countries") >= 2, 0.5).otherwise(0.0))

        # Rule 4: Card testing — multiple tiny transactions
        .withColumn("rule_card_testing",
            F.when(F.col("small_tx_count") >= 3, 0.4).otherwise(0.0))

        # Rule 5: Rapid online — many online transactions rapidly
        .withColumn("rule_rapid_online",
            F.when(F.col("online_tx_count") >= 4, 0.35).otherwise(0.0))

        # Composite score (capped at 1.0)
        .withColumn("fraud_score", F.least(
            F.col("rule_velocity") + F.col("rule_high_amount") +
            F.col("rule_geo_impossible") + F.col("rule_card_testing") +
            F.col("rule_rapid_online"), F.lit(1.0)))

        # Build a list of which rules fired
        .withColumn("rules_triggered", F.concat_ws(",",
            F.when(F.col("rule_velocity")      > 0, F.lit("velocity")),
            F.when(F.col("rule_high_amount")   > 0, F.lit("high_amount")),
            F.when(F.col("rule_geo_impossible")> 0, F.lit("geo_impossible")),
            F.when(F.col("rule_card_testing")  > 0, F.lit("card_testing")),
            F.when(F.col("rule_rapid_online")  > 0, F.lit("rapid_online")),
        ))

        # Flag as fraud if score >= 0.6
        .withColumn("is_fraud_alert", F.col("fraud_score") >= 0.6)
        .withColumn("alert_time", F.current_timestamp())
    )
    return scored

"""
Fraud detection rules — standalone module.
Separating rules from streaming infrastructure means:
- Fraud team can update rules without touching Spark code
- Rules can be unit tested independently
- Rules can be loaded from config file / database in production
"""
from pyspark.sql import functions as F
from pyspark.sql import DataFrame
from dataclasses import dataclass
from typing import List

@dataclass
class FraudRule:
    name: str
    score: float
    description: str

# Rule definitions — could be loaded from database in production
FRAUD_RULES: List[FraudRule] = [
    FraudRule("velocity",       0.40, "5+ transactions in 5 minutes"),
    FraudRule("high_amount",    0.30, "Single transaction > $5,000"),
    FraudRule("geo_impossible", 0.50, "Transactions from 2+ countries in 5 min"),
    FraudRule("card_testing",   0.40, "3+ micro-transactions < $10"),
    FraudRule("rapid_online",   0.35, "4+ online transactions in 5 min"),
]

FRAUD_THRESHOLD = 0.60  # Score >= this triggers alert

def apply_rules(df: DataFrame) -> DataFrame:
    """Apply all fraud rules and return scored DataFrame."""
    rule_conditions = {
        "velocity":       F.col("tx_count_5min") >= 5,
        "high_amount":    F.col("max_single_amount") > 5000,
        "geo_impossible": F.col("distinct_countries") >= 2,
        "card_testing":   F.col("small_tx_count") >= 3,
        "rapid_online":   F.col("online_tx_count") >= 4,
    }

    score_expr = F.lit(0.0)
    for rule in FRAUD_RULES:
        col_name = f"rule_{rule.name}"
        df = df.withColumn(col_name,
            F.when(rule_conditions[rule.name], F.lit(rule.score)).otherwise(F.lit(0.0)))
        score_expr = score_expr + F.col(col_name)

    df = df.withColumn("fraud_score", F.least(score_expr, F.lit(1.0)))
    df = df.withColumn("is_fraud_alert", F.col("fraud_score") >= FRAUD_THRESHOLD)
    return df

def get_rule_descriptions() -> dict:
    """Return rule descriptions for documentation / API."""
    return {r.name: {"score": r.score, "desc": r.description} for r in FRAUD_RULES}

Step 7 · Schema Evolution

Handling Schema Changes in a Live Stream

What happens when the bank's transaction system adds a new field (e.g., device_fingerprint) to the Kafka messages? Your consumer must not break. This is schema evolution.

🚨
The problem without schema evolution: Stream job is running in production reading JSON with 14 fields. Bank's system upgrades overnight and starts sending 15 fields. If your Spark job has a hardcoded schema with 14 fields, it silently drops the new field — or worse, fails on strict validation. After millions of events, you realize fraud pattern data is missing a key signal.
7a
Strategy 1: permissive JSON parsing with from_json
Simplest — works for most cases
▼
PySpark — Permissive Schema with Optional Fields
from pyspark.sql.types import StructType, StructField, StringType

# Version 1 schema (original)
SCHEMA_V1 = StructType([
    StructField("transaction_id", StringType(), True),
    StructField("account_id",     StringType(), True),
    StructField("amount",         DoubleType(), True),
    # ... other fields
])

# Version 2 schema — added device_fingerprint field
# Setting nullable=True for new fields is BACKWARD COMPATIBLE
# Old messages that don't have device_fingerprint will get NULL
SCHEMA_V2 = StructType([
    StructField("transaction_id",     StringType(), True),
    StructField("account_id",         StringType(), True),
    StructField("amount",             DoubleType(), True),
    StructField("device_fingerprint", StringType(), True),  # NEW field, nullable
    # ... other fields
])

# from_json silently ignores extra fields in the JSON that aren't in the schema
# and fills missing fields with NULL — this is backward compatible
parsed = raw.select(F.from_json(F.col("value").cast("string"), SCHEMA_V2).alias("d"))

# Handle NULL device_fingerprint gracefully
enriched = parsed.withColumn("device_fingerprint",
    F.coalesce(F.col("d.device_fingerprint"), F.lit("unknown")))

The key rule for schema evolution: Adding a field with a default value is safe (old consumers get NULL, new consumers get the value). Removing or renaming a required field breaks old consumers. Schema Registry enforces these compatibility rules automatically — it rejects incompatible schema updates before any producer can publish a breaking change.

SQL — Merchant Risk Score
WITH merchant_metrics AS (
    SELECT
        t.merchant_id,
        t.merchant_category,
        COUNT(t.transaction_id)            AS total_tx,
        COUNT(DISTINCT t.account_id)       AS unique_accounts,
        AVG(t.amount)                       AS avg_tx_amount,
        SUM(t.amount)                       AS total_volume,
        COUNT(f.alert_id)                   AS fraud_tx_count,
        SUM(CASE WHEN f.alert_id IS NOT NULL THEN t.amount ELSE 0 END) AS fraud_volume
    FROM transactions_history t
    LEFT JOIN fraud_alerts_history f
        ON t.account_id = f.account_id
       AND f.status = 'CONFIRMED'
    WHERE t.year = 2024
    GROUP BY t.merchant_id, t.merchant_category
    HAVING COUNT(t.transaction_id) >= 10  -- Minimum transaction threshold
),
category_avg AS (
    SELECT
        merchant_category,
        AVG(avg_tx_amount) AS cat_avg_amount
    FROM merchant_metrics
    GROUP BY merchant_category
)
SELECT
    m.merchant_id, m.merchant_category,
    m.total_tx, m.unique_accounts,
    ROUND(m.avg_tx_amount, 2)           AS avg_tx,
    ROUND(c.cat_avg_amount, 2)          AS category_avg_tx,
    m.fraud_tx_count,
    ROUND(m.fraud_tx_count * 100.0
        / NULLIF(m.total_tx,0), 2)      AS fraud_rate_pct,
    ROUND((m.avg_tx_amount - c.cat_avg_amount)
        / NULLIF(c.cat_avg_amount,0) * 100, 1) AS pct_above_category_avg,
    NTILE(5) OVER (ORDER BY
        m.fraud_tx_count * 1.0 / NULLIF(m.total_tx,0) DESC
    ) AS risk_quintile   -- 1=highest risk, 5=lowest risk
FROM merchant_metrics m
JOIN category_avg c ON m.merchant_category = c.merchant_category
ORDER BY fraud_rate_pct DESC;

Q1Kafka What is Kafka and why would you use it instead of just writing directly to a database?
Model Answer

Kafka is a distributed, durable message queue — a persistent log of events that multiple consumers can read independently. Writing transactions directly to a database means every consumer (fraud detection, analytics, compliance, audit) creates direct load on that database. If fraud detection is slow, it slows down the transaction system. Kafka decouples producers from consumers: the bank's transaction system publishes to Kafka without waiting for any downstream system. Each consumer reads at its own pace, with its own offset. If fraud detection restarts after a failure, it replays the backlog from where it left off — no data lost. The database can't do that. Kafka also handles massive throughput spikes that would overwhelm a database — it's designed to handle millions of messages per second with microsecond latency on writes.

Q2Kafka Why did you use account_id as the Kafka partition key?
Model Answer

Two reasons. First, ordering: within a partition, Kafka guarantees message order. For fraud detection, I need to see all of account ACC-123's transactions in chronological order to detect patterns like velocity. If account ACC-123's transactions could land in different partitions, they might be processed out of order. Using account_id as the key ensures all transactions for one account always go to the same partition. Second, data locality for stateful processing: Spark's streaming job can maintain state per account (running count, recent amounts) efficiently when all events for an account arrive at the same partition and get processed by the same executor. Without this, Spark would have to shuffle state across the cluster to aggregate per-account windows, which is expensive. The tradeoff: if some accounts are much more active than others (a "hot partition"), that partition gets more messages than others. For banking, account activity is relatively evenly distributed, so this isn't a major concern.

Q3Streaming Explain the difference between a tumbling and sliding window. Which did you use and why?
Model Answer

A tumbling window is fixed-size and non-overlapping. With a 5-minute tumbling window: window 1 covers 0:00–5:00, window 2 covers 5:00–10:00. Each event belongs to exactly one window. A sliding window slides forward by a step size smaller than the window size. A 5-minute window sliding every 1 minute: window 1 covers 0:00–5:00, window 2 covers 1:00–6:00, and so on. An event at 4:59 appears in 5 different windows. I used sliding windows because fraud patterns don't respect fixed time boundaries. A fraudster making transactions at 4:58, 4:59, 5:00, 5:01, 5:02 would be split across two tumbling windows — neither window would see 5 transactions, so no alert fires. A 5-minute sliding window with 1-minute slides catches this pattern in multiple windows. The cost is higher: each event is processed N times (window_size / slide_interval), but for fraud detection the accuracy improvement is worth it.

Q4Streaming What is a watermark in stream processing and why is it necessary?
Model Answer

A watermark defines how long the system waits for late-arriving events before closing a window. In the real world, events don't arrive in perfect chronological order. A mobile transaction might take 3 seconds to reach Kafka due to network latency. A transaction at timestamp 12:05:00 might arrive at Kafka at 12:05:03. Without a watermark, Spark would either wait forever (never close any window) or close windows immediately and miss late events. I set a 10-second watermark: "process events that arrive up to 10 seconds after their event timestamp." Spark won't close a window for event_time 12:05:00 until it sees events with timestamps ≥ 12:05:10. Events arriving more than 10 seconds late are dropped. The tradeoff: longer watermarks = more accurate results but higher latency. Shorter watermarks = faster results but more dropped late events. For fraud detection where a 5-second alert target is acceptable, a 10-second watermark is a reasonable balance.

Q5Streaming How does checkpointing make your streaming job fault-tolerant?
Model Answer

Checkpointing periodically saves the streaming job's state to a durable store — in my case S3. The state includes: which Kafka offsets have been processed, the current window aggregation state (per-account transaction counts, amounts), and the query plan. If the Spark job crashes — server reboot, OOM error, network partition — it restarts and reads the checkpoint. It resumes processing exactly from the last checkpointed offset without reprocessing everything or missing anything. Without checkpointing, on failure you'd have to decide: replay from the beginning (huge backlog, potentially millions of duplicates) or start from the latest offset (lose everything since the crash). Checkpointing gives you exactly-once semantics: combined with Kafka's ability to reset offsets and Spark's idempotent writes, each event is processed exactly once even through failures. The checkpoint location must be persistent storage — not local disk, which disappears when the container restarts.

Q6Streaming Explain the three output modes in Spark Structured Streaming. Which did you use where?
Model Answer

Append mode only writes new rows to the sink — rows that were added in the current micro-batch and won't change. I used this for writing raw transactions to S3 — each transaction is immutable, so append is correct. Update mode writes only rows that changed in the current micro-batch. I used this for writing windowed fraud scores — as new transactions arrive and windows get updated, only the changed window rows go to the sink. Complete mode writes the entire result table every trigger — useful for small aggregated tables where you want the full current state. I avoided complete mode for the windowed aggregations because the result table grows over time and writing all of it every 10 seconds would be expensive. The rule of thumb: append for immutable events, update for stateful aggregations that evolve, complete only for small reference tables.

Q7Architecture Explain your hot path vs cold path design. Why two separate paths?
Model Answer

The hot path handles the real-time fraud alert flow: flagged transactions go to a fraud-alerts Kafka topic, Lambda consumes them within seconds, writes to DynamoDB, and fires SNS notifications to the fraud team. The design is optimized for latency — every component is fast and simple. DynamoDB can handle millions of reads per second so the fraud alert lookup is sub-millisecond. The cold path handles everything going to S3 as Parquet for historical analysis — all transactions, not just flagged ones. Analysts use Athena to query months of transaction history, build fraud models, identify new patterns, and create regulatory reports. The reason for two paths: the requirements are fundamentally different. Hot path: sub-5-second latency, high availability, simple data model. Cold path: long retention, complex analytical queries, rich historical context. Trying to serve both requirements from a single system would mean either compromising latency (if the analytics system is the source of truth) or missing complex analytical capabilities (if DynamoDB is the only store). Lambda architecture — hot and cold paths — is the standard industry pattern for exactly this tradeoff.

Q8Kafka What is a consumer group and how does it enable scaling?
Model Answer

A consumer group is a set of consumer instances that collectively consume a topic. Kafka distributes partitions across the group: if a topic has 6 partitions and the consumer group has 3 instances, each instance gets 2 partitions. No two instances in the same group consume the same partition — no duplicate processing. As message volume grows, you scale by adding consumer instances to the group up to the number of partitions. To handle 10x more messages, add 10x more consumer instances (and potentially add partitions to the topic). Each consumer group has its own independent offset — so my fraud detection consumer group and analytics consumer group both read the full stream independently, at their own pace. This is why multiple use cases can consume the same Kafka topic simultaneously without interfering. The key constraint: you can't have more consumer instances than partitions in a group — extra instances sit idle waiting for a partition assignment.

Q9Fraud Walk me through one of your fraud detection rules and the business logic behind it.
Model Answer

I'll walk through the "geo impossible" rule. A legitimate user can't be in two countries simultaneously. If within a 5-minute window, account ACC-123 has transactions in both the US and China, that's physically impossible unless the card was cloned. My Spark window aggregation counts distinct countries per account in each 5-minute sliding window using countDistinct("country"). If distinct_countries ≥ 2, the rule fires with a score of 0.5 — the highest of any single rule, because it's one of the most reliable fraud signals. Combined with velocity (0.4), the composite score would be 0.9, well above the 0.6 alert threshold. In production, I'd refine this rule to exclude online transactions from the location check — you can make an online purchase from a US merchant while physically in Europe. I'd also add a country_is_online flag to distinguish card-present from card-not-present transactions before applying the geographic check.

Q10Fraud How would you reduce false positives in your fraud detection system?
Model Answer

My current system has rule-based scoring — all rules are static thresholds. False positives come from legitimate behavior that happens to trigger rules: a traveler making multiple purchases in two countries is flagged by geo_impossible. Four approaches to reduce false positives. First, enrich context: add a "known travel flag" from the bank's CRM — if the customer called to say they're traveling to China, suppress geo flags. Second, historical baseline: instead of static thresholds, compare to the account's own history. A customer who normally makes 8 transactions per day won't trigger velocity alerts. Third, ML scoring layer: train a gradient boosting model on labeled fraud/non-fraud history to replace or augment the rule scores. ML models learn subtle patterns rules can't express. Fourth, feedback loop: the alert_handler should capture fraud analyst decisions (confirmed vs false_positive) back into the pipeline. Track precision per rule combination and automatically raise thresholds when precision drops. In the SQL analysis query, I track false_positive rates per rule combination specifically for this purpose.

Q11Streaming What is exactly-once semantics and how does Kafka + Spark achieve it?
Model Answer

Exactly-once means every message is processed exactly one time — even through failures, retries, and restarts. It's the hardest guarantee in distributed systems. Kafka + Spark Structured Streaming achieves it through three components. First, Kafka offset tracking: Spark checkpoints the exact Kafka offset it's processed. On restart, it reads exactly from that offset — no re-reads, no skips. Second, idempotent producers: Kafka producers can be configured with enable.idempotence=true, so retried messages are deduplicated by the broker using sequence numbers. Third, transactional writes: Spark writes output and commits the Kafka offset atomically in a transaction. Either both happen or neither does — no partially committed micro-batches. The result: even if a Spark executor crashes mid-batch, after recovery the same data is written at most once. This matters enormously for banking: double-processing a fraud alert could block a legitimate customer twice; missing an alert means real fraud goes undetected.

Q12Architecture What is schema evolution and how did you handle it?
Model Answer

Schema evolution is the challenge of changing the schema of messages in a live Kafka topic without breaking existing consumers. The bank's development team might add a device_fingerprint field to transaction events — existing consumers need to keep working without any downtime. I handle it at two levels. For JSON: I use from_json with a schema that marks new fields as nullable=True. Old messages that don't have device_fingerprint produce NULL for that field — the consumer handles it gracefully with coalesce(device_fingerprint, 'unknown'). No consumer downtime. For production: I'd use Schema Registry with Avro. The Schema Registry enforces compatibility rules — a producer can only publish a new schema if it's backward compatible (existing consumers can read it) or forward compatible (they can skip unknown fields). This prevents accidental breaking changes from reaching production. The key rule: adding a field with a default value is backward compatible. Removing or renaming a required field is a breaking change the Schema Registry would reject.

Q13Architecture Why DynamoDB for the fraud alerts and not Postgres or Redshift?
Model Answer

The fraud alerts table has a very specific access pattern: the card authorization system needs to look up whether an account has an open fraud alert within milliseconds during a transaction authorization. That's a simple key-value lookup — account_id → alert status. DynamoDB handles this in single-digit milliseconds at any scale. Postgres would work for small volumes but requires capacity planning, connection pooling, and would become a bottleneck at 10K+ authorizations per minute. Redshift is a data warehouse — it's optimized for analytical queries scanning millions of rows, not for high-concurrency single-row lookups. The cost of a Redshift query per lookup would be absurd. DynamoDB's design — partition key-based lookup, no joins required, auto-scaling — is purpose-built for this exact use case. The tradeoff is DynamoDB's limited query flexibility (no ad-hoc SQL), which is why the historical cold path uses S3 + Athena for analytics. Neither system tries to do both jobs.

Q14Production How would you monitor this streaming pipeline in production?
Model Answer

Four layers. First, Kafka metrics: consumer lag is the most important — if the fraud detection consumer is falling behind (lag growing), it means we're not detecting fraud fast enough. I'd alert if lag exceeds 10,000 messages. Producer throughput and broker disk usage are also critical. Second, Spark metrics: micro-batch processing time (should stay under the trigger interval — 10 seconds), batch duration trends, and the number of records processed per batch. If processing time creeps above the trigger interval, micro-batches queue up and lag grows. Third, fraud detection quality: alert rate per hour (sudden drop → rules might be broken; sudden spike → possible attack or false rule change), false positive rate tracked by analyst feedback, and average fraud_score distribution over time. Fourth, business metrics: total alerts per day, confirmed fraud rate, average time-to-alert. A CloudWatch dashboard with all these metrics lets on-call engineers see at a glance if something's wrong. I'd also implement a "dead man's switch" — an alert that fires if no fraud alerts are generated for 4+ hours, because that might mean the detection pipeline silently failed.

Q15Production When would you choose Apache Flink over Spark Structured Streaming?
Model Answer

I'd choose Flink when the latency requirement is genuinely below one second, when the state management needs are very complex, or when the team already has Flink expertise. Flink processes events truly record by record, not in micro-batches — it achieves single-digit millisecond latency where Spark's minimum is around 100ms (the micro-batch overhead). For a stock trading system where a 200ms delay costs real money, Flink is the right choice. Flink also has more sophisticated stateful processing: managed state with multiple backends, complex event processing (CEP) for detecting patterns across multiple event types, and better support for out-of-order event handling. Spark wins when: the team knows Spark (huge advantage — no new system to learn), latency of seconds is acceptable, or the same team manages batch and streaming pipelines (one system instead of two). For our fraud detection use case — 5-second alert target, team with Spark expertise from Projects 1 and 2, micro-batch every 10 seconds — Spark Structured Streaming is the right call. The added operational complexity of Flink isn't justified.

Key Concepts

Deep Dive Concepts for Week 6

🌊
Lambda Architecture vs Kappa Architecture
▼

Lambda Architecture (what we built): Two separate pipelines — batch layer (cold path: S3/Athena) and speed layer (hot path: Kafka/Spark/DynamoDB). Batch gives accurate historical analysis; speed layer gives low-latency real-time view. Complexity: two codebases to maintain, results must be reconciled.

Kappa Architecture: Single streaming pipeline handles both real-time and historical. Store everything in a long-retention Kafka topic (or object store). Re-process history by replaying the stream. Simpler to maintain (one codebase) but more expensive (streaming infrastructure 24/7 vs batch running nightly).

When to use Lambda: Complex historical queries that need SQL joins across years of data. When batch and stream teams are separate. When history is large and reprocessing is expensive.

When to use Kappa: When the stream can handle history replay. Simpler organizations. Modern streaming systems (Flink, Kafka Streams) make Kappa more practical than it used to be.

🔑
Kafka Partition Key Selection — Critical Decision
▼

The partition key determines two things: ordering guarantees and data locality. Rules for choosing:

  • Choose a key where ordering matters to your consumers. We use account_id because fraud detection needs to see transactions in order per account.
  • Choose a key with high cardinality and even distribution. Using country as a key would put 80% of US transactions in one partition — a "hot partition" that can't be parallelized.
  • Avoid NULL keys. NULL keys go to random partitions — losing ordering guarantees.
  • For global ordering (no key): Use a single-partition topic. Fully ordered but no parallelism. Only for very low volume.
🔄
Event Time vs Processing Time vs Ingestion Time
▼

Event time: When the event actually happened — the transaction's timestamp from the bank's system. Most accurate for business logic. Can arrive late. Used in our fraud rules.

Processing time: When Spark processes the event. Simple to implement, no late data issues. But results change based on when you run the query — not deterministic. Useless for "how many transactions happened at 3pm yesterday?"

Ingestion time: When Kafka received the event. More reliable than processing time (set by Kafka broker, not consumer), but still not the same as when the event happened.

Rule: Always use event time for business logic. Use processing time only for monitoring pipeline health (e.g., "how long is our end-to-end latency?").

📦
DynamoDB vs Cassandra vs Redis — When to Use Each
▼

DynamoDB: AWS-managed NoSQL. Single-digit ms latency, infinite scale, pay-per-request. Best for: serverless architectures, key-value and simple query patterns, when you don't want to manage infrastructure. Used here because it's serverless and integrates naturally with Lambda.

Apache Cassandra: Open-source distributed NoSQL. Excellent write throughput, linear scaling, multi-region replication. Best for: time-series data, write-heavy workloads, when you need to self-host. Used by Netflix, Discord for massive write-heavy use cases.

Redis: In-memory key-value store. Sub-millisecond latency. Best for: caching, session storage, real-time leaderboards, rate limiting. Data is in RAM — expensive for large datasets. Use for fraud detection if you need <1ms lookups (e.g., real-time card authorization blocking).
