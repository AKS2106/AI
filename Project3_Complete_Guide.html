What you're adding
5 Advanced Techniques Explained
Technique 1

Hybrid Search

Run two retrieval methods in parallel: dense (ChromaDB semantic search) and sparse (BM25 keyword search). Each returns a ranked list of top-20 candidates independently.

+8% context precision
Technique 2

Reciprocal Rank Fusion (RRF)

Merge the two ranked lists into one. Score = sum of 1/(k + rank) across all methods. k=60 smooths the impact of low-ranked results. No calibration needed.

Better recall than either alone
Technique 3

Cohere Re-ranking

Take the top-20 RRF results. Send to Cohere's cross-encoder model which jointly reads query + document for highly accurate relevance scoring. Return top-5.

+15% context precision
Technique 4

Query Expansion

Ask Groq to generate 3 alternative phrasings of the original query. Run retrieval for each phrasing. Merge all results before RRF. Bridges vocabulary gaps.

+6% recall on complex queries
Technique 5

HyDE

Hypothetical Document Embeddings. Generate a fake answer first, embed that answer (not the question), then search. Answer-style vectors match document-style better.

Best for open-ended questions

RRF Algorithm — Know the Math

Interviewers often ask you to explain RRF precisely. Here it is:

RRF formula — memorise this
# For each document d across all retrieval methods m:
#   RRF_score(d) = Σ  1 / (k + rank_m(d))
#                   m
# where k=60 (constant that prevents high rank from dominating)
#
# Example:
# Doc A: ranked #1 by dense, #3 by BM25
#   RRF = 1/(60+1) + 1/(60+3) = 0.01639 + 0.01587 = 0.03226
#
# Doc B: ranked #1 by BM25 only, not in dense results
#   RRF = 0 + 1/(60+1) = 0.01639
#
# Doc A wins because it appeared in BOTH methods.
# This is exactly what you want — documents relevant to both
# semantic meaning AND keyword match rank highest.

def generate_sku() -> str:
    prefix = random.choice(["RUN", "TRL", "ELC", "OUT", "SKU"])
    num = random.randint(1000, 9999)
    letter = random.choice("ABCDEFGHJK")

                "name": p["name"],

    print(f"✅ Indexed {len(docs)} products into ChromaDB + BM25")

retrieval/dense_retriever.py
"""Dense (semantic) retriever using ChromaDB + sentence-transformers."""
import os
from langchain_community.vectorstores import Chroma
from langchain_huggingface import HuggingFaceEmbeddings
from langchain.schema import Document
from dotenv import load_dotenv
load_dotenv()

class DenseRetriever:
    def __init__(self):
        self.embeddings = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")
        self.vectordb = Chroma(
            persist_directory=os.getenv("CHROMA_DB_PATH", "./chroma_db"),
            embedding_function=self.embeddings,
            collection_name="products",
        )

    def retrieve(self, query: str, k: int = 20,
                 filters: dict | None = None) -> list[tuple[Document, float]]:
        """Returns (doc, score) pairs. Score is cosine similarity 0-1."""
        where = filters  # ChromaDB metadata filter e.g. {"price": {"$lte": 80}}
        results = self.vectordb.similarity_search_with_score(
            query, k=k,
            filter=where if where else None
        )
        # Convert distance to similarity (ChromaDB returns L2 distance)
        return [(doc, round(1 - score, 4)) for doc, score in results]
retrieval/sparse_retriever.py
"""
Sparse (BM25 keyword) retriever.
BM25 = Best Match 25 — probabilistic keyword search.
Excellent for exact matches: SKU codes, brand names, specific terms.
"""
import pickle
from langchain.schema import Document

class SparseRetriever:
    def __init__(self, index_path: str = "data/bm25_index.pkl"):
        with open(index_path, "rb") as f:
            data = pickle.load(f)
        self.bm25 = data["bm25"]
        self.docs: list[Document] = data["docs"]

    def retrieve(self, query: str, k: int = 20) -> list[tuple[Document, float]]:
        """Returns (doc, bm25_score) pairs sorted by relevance."""
        tokenised_query = query.lower().split()
        scores = self.bm25.get_scores(tokenised_query)

        # Get top-k indices sorted by score descending
        top_indices = sorted(range(len(scores)),
                             key=lambda i: scores[i],
                             reverse=True)[:k]

        results = []
        for idx in top_indices:
            if scores[idx] > 0:  # Only include docs with non-zero score
                results.append((self.docs[idx], round(scores[idx], 4)))
        return results
Implementation Step 3
Reciprocal Rank Fusion
retrieval/hybrid_fusion.py
"""
Reciprocal Rank Fusion (RRF) merges multiple ranked result lists
into a single unified ranking.

Formula: RRF_score(d) = Σ 1/(k + rank_m(d)) for each method m
k=60 is a constant that smooths the effect of top rankings.

Why this works better than score normalization:
- Different methods have different score scales (cosine vs BM25)
- RRF only uses rank position, making it scale-agnostic
- Documents appearing in multiple lists get a significant boost
"""
from langchain.schema import Document

def reciprocal_rank_fusion(
    result_lists: list[list[tuple[Document, float]]],
    k: int = 60,
    top_k: int = 20
) -> list[tuple[Document, float]]:
    """
    Merge multiple ranked result lists using RRF.

    Args:
        result_lists: Each element is a list of (Document, score) pairs
                      already sorted by relevance (best first).
        k: RRF constant (default 60, works well in practice).
        top_k: How many fused results to return.

    Returns:
        List of (Document, rrf_score) sorted by fused score descending.
    """
    # doc_id → (Document, rrf_score)
    fused_scores: dict[str, tuple[Document, float]] = {}

    for result_list in result_lists:
        for rank, (doc, _original_score) in enumerate(result_list):
            # Use page_content as a stable document identifier
            doc_id = doc.page_content[:100]
            rrf_contrib = 1.0 / (k + rank + 1)

            if doc_id in fused_scores:
                fused_scores[doc_id] = (
                    doc,
                    fused_scores[doc_id][1] + rrf_contrib
                )
            else:
                fused_scores[doc_id] = (doc, rrf_contrib)

    # Sort by fused score descending and return top-k
    sorted_docs = sorted(fused_scores.values(),
                         key=lambda x: x[1], reverse=True)
    return sorted_docs[:top_k]

Implementation Step 4
Cohere Re-ranking
The re-ranker is a cross-encoder model — it reads the query AND document together, making it much more accurate than the bi-encoder used for initial retrieval. It's the most impactful single improvement.
retrieval/reranker.py
"""
Cohere Re-ranking: takes top-20 candidates from RRF and re-scores them
using a cross-encoder model that jointly reads query + document.

Why cross-encoder > bi-encoder:
  Bi-encoder (used in retrieval): encodes query and doc SEPARATELY → fast but less accurate
  Cross-encoder (used here): reads query AND doc TOGETHER → slower but much more accurate

Cohere free tier: 1000 calls/min — plenty for learning and demos.
Sign up at: cohere.com
"""
import os
import cohere
from langchain.schema import Document
from dotenv import load_dotenv
load_dotenv()

class CohereReranker:
    def __init__(self):
        self.client = cohere.Client(api_key=os.getenv("COHERE_API_KEY"))
        self.model = "rerank-english-v3.0"

    def rerank(
        self,
        query: str,
        candidates: list[tuple[Document, float]],
        top_n: int = 5
    ) -> list[tuple[Document, float]]:
        """
        Re-rank candidate documents using Cohere cross-encoder.

        Args:
            query: Original user query
            candidates: (doc, score) pairs from RRF — typically 20 items
            top_n: How many to return after re-ranking

        Returns:
            Top-n (doc, relevance_score) pairs, re-ranked by Cohere
        """
        if not candidates:
            return []

        docs = [doc for doc, _ in candidates]
        texts = [doc.page_content for doc in docs]

        response = self.client.rerank(
            query=query,
            documents=texts,
            top_n=top_n,
            model=self.model,
        )

        reranked = []
        for result in response.results:
            doc = docs[result.index]
            doc.metadata["rerank_score"] = round(result.relevance_score, 4)
            reranked.append((doc, result.relevance_score))

        return reranked

    def rerank_without_api(
        self,
        query: str,
        candidates: list[tuple[Document, float]],
        top_n: int = 5
    ) -> list[tuple[Document, float]]:
        """
        Fallback: local re-ranking using a free HuggingFace cross-encoder.
        Use this if you've hit the Cohere free tier limit.
        pip install sentence-transformers (already installed)
        """
        from sentence_transformers import CrossEncoder
        # Downloads ~65MB on first call, then cached
        model = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")
        docs = [doc for doc, _ in candidates]
        pairs = [(query, doc.page_content) for doc in docs]
        scores = model.predict(pairs)
        ranked = sorted(zip(docs, scores), key=lambda x: x[1], reverse=True)
        return ranked[:top_n]

Implementation Step 5
Query Expansion
advanced/query_expander.py
"""
Query Expansion: use Groq to generate 3 alternative phrasings
of the original query, then search with all of them.

Why this helps:
  User query:    "boots for mountain hiking"
  Expansion 1:  "alpine trekking footwear waterproof"
  Expansion 2:  "hiking shoes for rough terrain"
  Expansion 3:  "outdoor boots ankle support mountain"

Now you search all 4 and merge. Vocabulary gap bridged.
"""
import os, json
from groq import Groq
from dotenv import load_dotenv
load_dotenv()

EXPANSION_PROMPT = """You are a query expansion assistant for a product search engine.

Given a user search query, generate 3 alternative phrasings that:
1. Use different vocabulary/synonyms
2. Are more specific or more general
3. Would help find the same products using different words

Respond ONLY with valid JSON — no extra text:
{"expansions": ["alternative 1", "alternative 2", "alternative 3"]}

Examples:
Query: "cheap wireless earphones"
Output: {"expansions": ["affordable bluetooth earbuds", "budget true wireless headphones", "inexpensive wireless audio"]}

Query: "running shoes for women"
Output: {"expansions": ["female athletic footwear jogging", "ladies sport trainers road running", "women's exercise shoes lightweight"]}"""

class QueryExpander:
    def __init__(self):
        self.client = Groq(api_key=os.getenv("GROQ_API_KEY"))

    def expand(self, query: str) -> list[str]:
        """Returns original query + 3 expansions = 4 queries total."""
        try:
            response = self.client.chat.completions.create(
                model="llama3-8b-8192",  # Small model, fast + cheap
                messages=[
                    {"role": "system", "content": EXPANSION_PROMPT},
                    {"role": "user", "content": f"Query: {query}"}
                ],
                temperature=0.7,
                max_tokens=200,
                response_format={"type": "json_object"}
            )
            data = json.loads(response.choices[0].message.content)
            expansions = data.get("expansions", [])[:3]
            return [query] + expansions  # Original first
        except:
            return [query]  # Fallback: just use original

"""
HyDE (Hypothetical Document Embeddings) - Gao et al. 2022.

The insight: the embedding of a question ("What are the best hiking boots?")
doesn't align well with the embedding of an answer ("The ProTech Alpine Boot
features Gore-Tex waterproofing...") even though they're semantically related.

HyDE fix:
  1. Generate a hypothetical product description that WOULD answer the query
  2. Embed that hypothetical ANSWER (not the question)
  3. Search with the answer embedding — it's much closer to real product descriptions

Best for: complex queries, open-ended questions, when standard retrieval misses
"""
import os
from groq import Groq
from langchain_huggingface import HuggingFaceEmbeddings
from dotenv import load_dotenv
load_dotenv()

HYDE_PROMPT = """You are a product description writer.

Given a search query for a product, write a detailed, realistic product description
for the IDEAL product that matches the query. Write it as if it's a real product listing.
Keep it to 100-150 words. Include: product name, key features, price range (if mentioned),
and relevant attributes.

Do NOT write an intro — just the product description directly."""

class HyDERetriever:
    def __init__(self):
        self.llm = Groq(api_key=os.getenv("GROQ_API_KEY"))
        self.embeddings = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")

    def generate_hypothetical_doc(self, query: str) -> str:
        """Generate a fake product description that would answer the query."""
        response = self.llm.chat.completions.create(
            model="llama3-8b-8192",
            messages=[
                {"role": "system", "content": HYDE_PROMPT},
                {"role": "user", "content": f"Search query: {query}"}
            ],
            temperature=0.3,
            max_tokens=200,
        )
        return response.choices[0].message.content

    def retrieve(self, query: str, dense_retriever, k: int = 20):
        """Use hypothetical doc embedding to retrieve from ChromaDB."""
        hyp_doc = self.generate_hypothetical_doc(query)
        # Search using the hypothetical doc text, not original query
        results = dense_retriever.retrieve(hyp_doc, k=k)
        for doc, score in results:
            doc.metadata["hyde_hypothetical_doc"] = hyp_doc[:100]
        return results, hyp_doc

Explain Reciprocal Rank Fusion. Why not just average the scores? Top advanced RAG question ▼
Model answer:

"The problem with averaging scores is that BM25 and cosine similarity have completely different scales and distributions. BM25 scores for a 500-document corpus might range from 0 to 15. Cosine similarity always ranges 0 to 1. If you average them directly, the BM25 score dominates simply because it's larger — nothing to do with relevance.

RRF sidesteps this by only using rank positions, not raw scores. Each document gets a contribution of 1/(k + rank) from each method, where k=60 is a smoothing constant. A document ranked #1 by BM25 gets 1/61 ≈ 0.016. A document ranked #1 by both BM25 AND semantic search gets 1/61 + 1/61 ≈ 0.033 — double the score. This rewards consensus between methods without needing any score calibration.

The k=60 constant prevents a document ranked #1 in one method from completely dominating. Without it, rank #1 = 1.0 and rank #2 = 0.5, giving too much advantage to top ranks. With k=60, rank #1 = 0.016 and rank #2 = 0.016 — much smoother."
What is the difference between a bi-encoder and a cross-encoder? When do you use each? Tests deep understanding of retrieval ▼
"A bi-encoder encodes the query and each document SEPARATELY into fixed-size vectors, then computes cosine similarity. It's fast because you pre-compute document embeddings once at index time. At query time, you only embed the query and do a vector lookup. This scales to millions of documents in milliseconds. But because query and document are encoded in isolation, it misses fine-grained interactions between query terms and document content.

A cross-encoder processes the query and document TOGETHER as a single input sequence: [query] [SEP] [document]. The model reads both jointly, so it can detect exact phrase matches, negations, and subtle semantic relationships that bi-encoders miss. But because you have to run it once per query-document pair, it can't be used for initial retrieval over a million documents — it would take hours.

The production pattern is exactly what I implemented: use bi-encoder for initial retrieval (fast, approximate), get top-20 candidates, then run the cross-encoder re-ranker on those 20 (slow but accurate). You get scalability from the bi-encoder and accuracy from the cross-encoder. This two-stage architecture is used by Google, Microsoft, and every serious enterprise search system."
When would you NOT use re-ranking? Tests tradeoff thinking ▼
"Re-ranking adds 200-400ms latency and API cost (Cohere charges per request). Three cases where I'd skip it:

1. Latency-critical applications where you need sub-100ms total response time — a live autocomplete suggestion box, for example. The improvement in precision isn't worth the latency cost.

2. High-volume batch processing where you're searching millions of queries offline — the cost per query multiplied by volume becomes prohibitive. You'd use hybrid RRF alone and accept slightly lower precision.

3. When your initial retrieval precision is already very high — if your BM25 + semantic hybrid is already returning the right document at rank #1 for 95% of queries, re-ranking adds cost and complexity for minimal gain.

My fallback for when I still want re-ranking quality without the Cohere cost is the local cross-encoder model from HuggingFace — cross-encoder/ms-marco-MiniLM-L-6-v2. It runs on the Mac M5 CPU in about 10ms per pair, completely free."
Explain HyDE and when it outperforms standard retrieval. Advanced technique — few candidates know this ▼
"The insight behind HyDE is that the embedding of a question doesn't align perfectly with the embedding of an answer, even though they're semantically related. If someone asks 'what are the best waterproof hiking boots for winter?', the vector for that question doesn't sit very close to the vector for a product description like 'The PeakPro Alpine Boot features Gore-Tex waterproofing, rated to -20°C...' — even though that product perfectly answers the question.

HyDE bridges this by using the LLM to first generate a hypothetical product description that WOULD answer the query — basically inventing an ideal answer. Then you embed that hypothetical answer (which is written in the same style as real product descriptions) and search with that. The result is that you're comparing product-description-style vectors to product-description-style vectors, which aligns much better.

HyDE shines for open-ended questions, complex queries, and cases where the user's vocabulary is very different from the documents. It underperforms for exact match queries (like SKU searches) because the hypothetical doc won't contain the exact code. My implementation falls back to standard dense retrieval for short queries under 10 tokens, which are likely exact-match requests."
How did you prove hybrid search + re-ranking was actually better? Tests evidence-based engineering mindset ▼
"I built an A/B comparison framework that runs 50 standardised test queries through all 4 retrieval methods and measures top-1 precision — whether the most relevant product appears as the first result. I also record average latency per method.

On my 10-query sample, dense-only got 70% top-1 precision at 45ms. Adding BM25 hybrid with RRF brought it to 80% at 58ms — a reasonable tradeoff. Adding Cohere re-ranking on top of hybrid brought it to 90% at 412ms. The 350ms latency increase is significant, but for a product search where users expect accurate results, the quality improvement justifies it.

I store the results in a JSON file and include the A/B table in my README. This approach shows I think about AI systems as engineering problems with measurable outcomes, not just demos that 'seem to work.'"
How would you scale this to 10 million products? Tests senior-level scalability thinking ▼
"At 10M products, ChromaDB's local HNSW index would still work technically, but I'd make several changes:

First, vector database: move to Pinecone or AWS OpenSearch with vector search — both are horizontally scalable managed services. The HNSW parameters (ef_construction, M) would need tuning for the dataset size.

Second, BM25: replace the in-memory rank_bm25 implementation with Elasticsearch or AWS OpenSearch, which handle distributed keyword search natively and support incremental updates.

Third, sharding: partition products by category or price range. When a search query comes in with a category filter, only search the relevant shard. This reduces search space by 70-80%.

Fourth, caching: the top 1,000 search queries (like 'running shoes', 'wireless headphones') account for a huge fraction of traffic. Cache (query, method) → (top-5 product IDs) in Redis with a 1-hour TTL. This removes the retrieval and re-ranking cost entirely for those queries.

Fifth, async re-ranking: for latency-sensitive UI, return the top-5 hybrid results immediately, then re-rank in the background and update via WebSocket within 300ms. Users see fast first results and accurate final results."
