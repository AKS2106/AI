The rule: parallel wall time = slowest individual tool. Sequential wall time = sum of all tools. Every tool you add sequentially costs you its full latency. Every tool you add in parallel costs you only the marginal time beyond the current slowest.
The entire parallel/sequential distinction lives in a single message structure difference. The LLM emits one AIMessage with N tool_calls — your node decides how to execute them.
What the API returns for parallel calls
AIMessage( content="", # empty — model is calling tools, not speaking tool_calls=[ # ← ALL THREE in the same message {"id": "tc_001", "name": "get_stock_price", "args": {"ticker": "AAPL"}}, {"id": "tc_002", "name": "get_latest_news", "args": {"ticker": "MSFT"}}, {"id": "tc_003", "name": "get_analyst_rating", "args": {"ticker": "NVDA"}}, ] )
What you return after executing all 3
[ ToolMessage(content="AAPL: $213.18 (+1.2%)...", tool_call_id="tc_001"), ToolMessage(content="MSFT latest: Azure AI...", tool_call_id="tc_002"), ToolMessage(content="NVDA: Buy, target $1100...", tool_call_id="tc_003") ] # ↑ All 3 go back as a list — the LLM sees all results at once
Why tool_call_id matters
Each ToolMessage must carry the tool_call_id matching the original request. The LLM uses this to correlate "I asked for AAPL price (tc_001) and got $213.18 back (tc_001)". Without matching IDs, the model gets confused about which result belongs to which request — especially when tool results arrive out of order from parallel execution.
The ThreadPoolExecutor pattern (core of the implementation)
def parallel_tool_node(state): tool_calls = state["messages"][-1].tool_calls # ALL calls from one message def execute_one(tc): return tc["id"], TOOL_REGISTRY[tc["name"]].invoke(tc["args"]) results = [] with ThreadPoolExecutor(max_workers=len(tool_calls)) as executor: futures = {executor.submit(execute_one, tc): tc for tc in tool_calls} for future in as_completed(futures): # collect as each finishes call_id, output = future.result() results.append(ToolMessage( content=output, tool_call_id=call_id, # ← MUST match the request )) return {"messages": results} # all results in one state update
Parallel tool calls only help when calls are independent. Calling them in parallel when they have dependencies produces wrong results.
Use parallel when — calls are independent
Good candidates for parallelism
Fetching the same data for multiple entities: "Compare AAPL, MSFT, NVDA"
Multiple independent lookups: stock price + news + analyst rating for one ticker
Searching multiple sources: web search + internal KB + database query
Validating multiple fields: check email format + check username availability + check age
Use sequential when — calls have dependencies
Must be sequential (output of A feeds input of B)
Search → then summarize the search result
Get user ID → then query their orders with that ID
Generate code → then run it and check the output
Fetch document → then extract specific fields from it
The LLM decides. You don't need to explicitly tell the agent "call these in parallel." A well-written system prompt ("call tools for all entities simultaneously") plus good tool descriptions lets the model figure it out. If it's still calling sequentially, add a few-shot example showing parallel tool calls in your system prompt.
4 things interviewers probe on
Q: "Why use threads instead of async for parallel tools?"
"Most tool implementations (HTTP clients, database drivers) are synchronous. Wrapping them in async requires rewriting them with asyncio. ThreadPoolExecutor lets synchronous tools run in parallel without any modification — it's the pragmatic default. For tools written as async functions, use asyncio.gather() instead."
Q: "What if one parallel tool fails and others succeed?"
"Wrap each tool execution in try/except inside the thread. A failed tool returns a ToolMessage with an error string instead of raising an exception — the LLM sees the error message and can adapt (try a different tool, skip that stock, ask the user). Never let one tool failure crash the entire parallel batch."
Q: "Does the LLM always call tools in parallel when it can?"
"No — it depends on the model and prompt. Claude and GPT-4 support parallel tool calls but tend toward sequential by default. You encourage parallelism with explicit system prompt instructions: 'when you need data for multiple entities, call all tools simultaneously in a single response.' Without this nudge, the model often calls tools one at a time."
Q: "How does LangGraph's built-in ToolNode handle this vs a custom node?"
"LangGraph's prebuilt ToolNode from langgraph.prebuilt executes all tool_calls in a single AIMessage concurrently by default — you get parallel execution for free. A custom tool node gives you more control: custom error handling, timing instrumentation, result truncation, rate limiting per tool. For production I start with the prebuilt node, then replace it with a custom one when I need observability or error recovery logic."
