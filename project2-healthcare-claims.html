Business Problem

🏥
A regional health insurance payer processes 50,000+ medical claims per day across three claim types: medical, pharmacy, and dental. Their data team runs nightly batch jobs that take 8+ hours and fail frequently. Actuaries can't analyze claim denial trends or provider performance in real time. Compliance teams have no automated PHI (Protected Health Information) audit trail. You're building a robust, HIPAA-aware claims pipeline that reduces processing time to under 2 hours and provides a clean, partitioned data warehouse.

Why Healthcare is Special

🔒
HIPAA (Health Insurance Portability and Accountability Act) governs how patient data is handled. As a Data Engineer, you must understand: what counts as PHI, how to mask or de-identify it in analytics layers, and how to maintain audit trails. This makes healthcare one of the most complex industries for data engineering — and one of the highest-paying.

HIPAA Essentials for Data Engineers

What You Must Know About HIPAA

You don't need a law degree. You need to know enough to design systems that don't violate it — and to talk confidently about it in interviews.

What is PHI?

🔒
Protected Health Information (PHI) = any information that can identify a patient AND relates to their health. The 18 HIPAA identifiers include: names, dates (DOB, admission date), phone numbers, email addresses, SSN, medical record numbers, account numbers, certificate/license numbers, device identifiers, URLs, IP addresses, biometric identifiers, full-face photos, any unique identifying number.

As a Data Engineer, Your Job is:

1
PHI Masking in Analytics Layer
Never expose raw PHI in downstream tables analysts query
▼

The raw layer can hold full PHI — it's encrypted at rest in S3. The processed/analytics layer replaces PHI with masked versions or drops it entirely:

PySpark — PHI Masking Examples
# Name → first initial + last initial only
df = df.withColumn("member_name_masked",
    F.concat(F.substring(F.col("first_name"), 1, 1), F.lit("."),
             F.substring(F.col("last_name"), 1, 1), F.lit(".")))

# SSN → last 4 digits only
df = df.withColumn("ssn_masked",
    F.concat(F.lit("***-**-"), F.substring(F.col("ssn"), 8, 4)))

# DOB → year only (de-identifies for analytics)
df = df.withColumn("birth_year", F.year(F.col("date_of_birth")))
df = df.drop("date_of_birth")

# Member ID → one-way hash (can join but can't reverse)
df = df.withColumn("member_key",
    F.sha2(F.concat(F.col("member_id"), F.lit("SALT_2024_ECOM")), 256))
2
Audit Trail
Log every time PHI is accessed, modified, or exported
▼

Every ETL job writes to an audit log table with: who ran it, when, what data was touched, row counts. This is what compliance teams review. In AWS, you enable S3 server access logging and CloudTrail to automatically track all API calls to PHI buckets.

Python — Audit Log Entry
def write_audit_log(job_name, table_name, rows_processed, rows_failed, status):
    audit_entry = {
        "audit_id":        str(uuid.uuid4()),
        "job_name":        job_name,
        "table_name":      table_name,
        "run_timestamp":   datetime.utcnow().isoformat(),
        "rows_processed":  rows_processed,
        "rows_failed":     rows_failed,
        "status":          status,   # SUCCESS / FAILED / PARTIAL
        "pipeline_version": "2.0",
        "phi_accessed":    True,
    }
    # Write to S3 audit bucket (separate from data bucket)
    s3.put_object(
        Bucket="healthcare-audit-logs",
        Key=f"audit/{datetime.utcnow().strftime('%Y/%m/%d')}/{audit_entry['audit_id']}.json",
        Body=json.dumps(audit_entry)
    )
3
Encryption + Access Control
S3 encryption, separate buckets, IAM policies
▼

Architecture rules for HIPAA-compliant AWS:

  • S3 SSE-KMS encryption on raw PHI bucket (not just SSE-S3)
  • Separate S3 buckets for raw (with PHI) vs processed (masked)
  • S3 bucket policies deny non-HTTPS access
  • IAM roles with least-privilege — ETL role only reads raw, only writes processed
  • CloudTrail enabled on the account to log all API access
  • VPC endpoints so data never traverses public internet
  • 90-day retention on audit logs minimum
⚠️
Interview talking point: When asked about HIPAA, don't just say "we encrypted the data." Say: "I separated the raw PHI bucket from the analytics bucket, applied SHA-256 salted hashing to member IDs so analysts can join data without seeing PHI, replaced DOBs with birth years, and wrote an audit log for every ETL run that compliance teams can review. All this without impacting query performance because masking happens at the transformation layer."

Strict separation between PHI zone (raw), de-identified zone (processed), and serving zone (Redshift). No PHI ever leaves Zone 1 unmasked.

Z3
Zone 3: Redshift (Serving Layer)
COPY from S3 · star schema views · analyst-facing
▼

Processed Parquet from Zone 2 is loaded into Redshift using the COPY command. Redshift is used here (instead of just Athena) because healthcare actuaries run complex multi-table queries and need consistent sub-second response times that Athena can't guarantee.

Redshift COPY Command Pattern
-- Load fact_claims from S3 into Redshift
COPY healthcare.fact_claims
FROM 's3://healthcare-processed-yourname/fact_claims/'
IAM_ROLE 'arn:aws:iam::ACCOUNT_ID:role/RedshiftS3Role'
FORMAT AS PARQUET;

-- Truncate-and-load pattern for dimension tables
BEGIN;
  TRUNCATE healthcare.dim_member;
  COPY healthcare.dim_member
  FROM 's3://healthcare-processed-yourname/dim_member/'
  IAM_ROLE 'arn:aws:iam::ACCOUNT_ID:role/RedshiftS3Role'
  FORMAT AS PARQUET;
COMMIT;

Data Modeling · Snowflake Schema

Snowflake Schema Design

Unlike the star schema in Project 1, dimension tables here are normalized into sub-dimensions. The diagnosis dimension points to a diagnosis category, which points to a chapter — this is the ICD-10 hierarchy.

❄️
Why snowflake (not star) here? Healthcare has deep, standardized hierarchies. ICD-10 codes (diagnoses) have 70,000+ codes organized into chapters → blocks → categories → codes. CPT codes (procedures) have similar structure. Storing all hierarchy attributes in one dim_diagnosis table would be enormous and redundant. Snowflake schema normalizes these into separate tables, saving storage and making hierarchy queries cleaner.

Synthetic Healthcare Data Generator

Generates realistic HIPAA-style data: medical claims (CSV), pharmacy claims (JSON), dental claims (CSV), member eligibility (CSV), provider NPI registry (CSV). All synthetic — no real patient data.

🔒
This script generates synthetic data only. Never use real patient data for development or testing. In real healthcare companies, they use de-identified datasets approved by the privacy team, or tools like Synthea (open-source synthetic patient generator).
scripts/generate_data.py
"""
Synthetic healthcare data generator.
Produces: medical_claims.csv, pharmacy_claims.json, dental_claims.csv,
          member_eligibility.csv, provider_npi.csv
"""
import csv, json, random, uuid
from datetime import datetime, timedelta
from faker import Faker

fake = Faker()
random.seed(99)

# ── Config ───────────────────────────────────────────────
NUM_MEMBERS   = 2000
NUM_PROVIDERS = 500
NUM_MEDICAL   = 30000
NUM_PHARMACY  = 15000
NUM_DENTAL    = 5000
START_DATE    = datetime(2024, 1, 1)
END_DATE      = datetime(2024, 12, 31)

# ICD-10 sample codes (real codes, synthetic context)
ICD10_CODES = {
    "E11.9":  "Type 2 diabetes mellitus without complications",
    "I10":    "Essential hypertension",
    "J06.9":  "Acute upper respiratory infection",
    "M54.5":  "Low back pain",
    "F32.9":  "Major depressive disorder, single episode",
    "Z00.00": "Encounter for general adult medical examination",
    "K21.0":  "Gastro-esophageal reflux disease with esophagitis",
    "J45.909":"Unspecified asthma, uncomplicated",
    "N39.0":  "Urinary tract infection, site not specified",
    "S72.001A":"Fracture of unspecified part of femoral neck",
}

# CPT procedure codes
CPT_CODES = {
    "99213": ("Office visit, established patient, low complexity", "Office Visit"),
    "99214": ("Office visit, established patient, moderate complexity", "Office Visit"),
    "93000": ("Electrocardiogram, routine ECG", "Diagnostic"),
    "85025": ("Complete blood count with differential", "Lab"),
    "71046": ("Chest X-ray, 2 views", "Imaging"),
    "27447": ("Replacement of knee joint", "Surgery"),
    "43239": ("Upper GI endoscopy with biopsy", "Procedure"),
    "99283": ("Emergency department visit, moderate severity", "Emergency"),
}

SPECIALTIES = ["Primary Care", "Cardiology", "Orthopedics", "Gastroenterology",
               "Psychiatry", "Emergency Medicine", "Radiology", "Pathology"]
PLAN_TYPES  = ["HMO", "PPO", "EPO", "HDHP"]
STATES      = ["CA", "TX", "FL", "NY", "IL", "PA", "OH", "GA"]
DENIAL_CODES= ["CO97", "CO4", "CO96", "PR1", "CO45", None, None, None]

def rand_date(start=START_DATE, end=END_DATE):
    return start + timedelta(days=random.randint(0, (end - start).days))

def rand_date_str(start=START_DATE, end=END_DATE):
    return rand_date(start, end).strftime("%Y-%m-%d")

# ── Members (with PHI — raw zone) ────────────────────────
print("Generating members...")
members = []
for _ in range(NUM_MEMBERS):
    dob = rand_date(datetime(1940,1,1), datetime(2006,1,1))
    members.append({
        "member_id":        f"MBR-{str(uuid.uuid4())[:8].upper()}",
        "first_name":       fake.first_name(),
        "last_name":        fake.last_name(),

        "plan_type":        random.choice(PLAN_TYPES),

member_ids = [m["member_id"] for m in members]
with open("data/raw/members/member_eligibility_2024.csv", "w", newline="") as f:

        "cpt_code":          random.choice(cpt_codes),
        "place_of_service":  random.choice(["Office","Outpatient","Inpatient","ER"]),
        "billed_amount":     billed,

        "paid_amount":       paid,

        "claim_status":      status,
        "denial_reason_code":denial,
        "claim_type":        "medical",
    })
with open("data/raw/claims/medical/medical_claims_2024.csv", "w", newline="") as f:
    writer = csv.DictWriter(f, fieldnames=med_claims[0].keys())
    writer.writeheader(); writer.writerows(med_claims)

        "days_supply":    qty,

        "related_diagnosis": diag,

Step 4 · PySpark ETL + SCD Type 2

Claims ETL with SCD Type 2

The most complex transformation in this project. Reads all three claim types, applies PHI masking, builds dimension tables with SCD Type 2 for members, and writes to the processed zone.

Understanding SCD Type 2 First

📚
SCD = Slowly Changing Dimension.

Type 1: Overwrite the old value. No history. "Member moved from PPO to HMO → update the record." Problem: you lose the fact that their old claims were under a PPO plan.

Type 2: Insert a new row, keep the old one. Each row has effective_start_date, effective_end_date (9999-12-31 = current), and is_current flag. When a member changes plans: close the old row (set effective_end_date = today), insert a new row. Now historical claims correctly join to the plan that was active when the claim was filed.

abc123_v1   | MBR-001   | PPO       | 2024-01-01      | 2024-06-30    | FALSE      | 1
abc123_v2   | MBR-001   | HMO       | 2024-07-01      | 9999-12-31    | TRUE       | 2

-- A claim from March 2024 joins to version 1 (PPO).
-- A claim from August 2024 joins to version 2 (HMO).
-- This is the power of SCD Type 2!

def mask_phi(df: DataFrame) -> DataFrame:
    """
    Apply PHI masking before writing to processed zone.
    Rule: Never let raw PHI escape Zone 1.
    """
    # Member ID → salted SHA-256 hash (one-way, but joinable)
    df = df.withColumn("member_id_hash",
        F.sha2(F.concat(F.col("member_id"), F.lit(PHI_SALT)), 256))

    # Name → initials only
    if "first_name" in df.columns:
        df = df.withColumn("member_name_masked",
            F.concat(F.substring("first_name", 1, 1), F.lit("."),
                     F.substring("last_name", 1, 1), F.lit(".")))
        df = df.drop("first_name", "last_name")

    # DOB → birth year only
    if "date_of_birth" in df.columns:
        df = df.withColumn("birth_year", F.year(F.to_date("date_of_birth")))
        df = df.drop("date_of_birth")

    # SSN → drop entirely (not needed for analytics)
    if "ssn" in df.columns:
        df = df.drop("ssn")

    # Drop raw member_id (use hash instead)
    df = df.drop("member_id")
    return df

# ── SCD TYPE 2 — BUILD DIM_MEMBER ────────────────────────
def build_dim_member_scd2(members_df: DataFrame) -> DataFrame:
    """
    Build SCD Type 2 dim_member.
    - Each unique combination of member + plan = one version
    - effective_start_date and effective_end_date track when each version was active
    - is_current = True means this is the latest plan for this member
    - version number increments for each plan change
    
    In a real incremental run, you'd compare against the existing dim_member
    to detect changes. For the initial full load, all records are version 1.
    """
    log.info("Building dim_member with SCD Type 2...")

    masked = mask_phi(members_df)

    # For initial load: all current records get version 1
    dim = (masked
        # Create surrogate key: hash of member_id_hash + plan_id + version
        .withColumn("version", F.lit(1).cast(IntegerType()))
        .withColumn("member_key",
            F.sha2(F.concat(
                F.col("member_id_hash"), F.lit("_"),
                F.col("plan_id"), F.lit("_"),
                F.col("version").cast(StringType())
            ), 256))
        # SCD2 date tracking columns
        .withColumn("effective_start_date",
            F.to_date(F.col("enrollment_date")))
        .withColumn("effective_end_date",
            F.to_date(F.lit("9999-12-31")))    # 9999 = current record
        .withColumn("is_current", F.lit(True))
        .withColumn("dw_inserted_at", F.current_timestamp())
        .select(
            "member_key", "member_id_hash", "member_name_masked",
            "birth_year", "gender", "state",
            "plan_id", "plan_type", "group_id",
            "effective_start_date", "effective_end_date",
            "is_current", "version", "dw_inserted_at"
        )
    )
    log.info(f"  → {dim.count()} member dimension rows (all version 1)")
    return dim

# ── SCD TYPE 2 — INCREMENTAL UPDATE (for subsequent runs) ─
def apply_scd2_changes(existing_dim: DataFrame, new_members: DataFrame) -> DataFrame:
    """
    Apply SCD Type 2 logic for incremental member plan changes.
    Called on Day 2+ runs when new member data arrives.
    
    Logic:
    1. Hash new member records
    2. Find members where plan_type changed (join on member_id_hash)
    3. Close old records (set effective_end_date = yesterday, is_current = False)
    4. Insert new version records
    5. Union with unchanged records
    """
    log.info("Applying SCD2 incremental changes...")
    today = datetime.utcnow().strftime("%Y-%m-%d")
    yesterday = (datetime.utcnow() - __import__('datetime').timedelta(days=1)).strftime("%Y-%m-%d")

    new_masked = mask_phi(new_members)

    # Find changed records: members where plan_type differs from current dim
    current_dim = existing_dim.filter(F.col("is_current") == True)

    changed = (new_masked
        .join(current_dim.select("member_id_hash", "plan_type".alias(...)))
        # simplified — in real code: filter where plan_type != existing plan_type
    )

    # Step 1: Close old records for changed members
    closed_old = (existing_dim
        .join(F.broadcast(changed.select("member_id_hash")), "member_id_hash", "left_semi")
        .withColumn("effective_end_date", F.to_date(F.lit(yesterday)))
        .withColumn("is_current", F.lit(False))
    )

    # Step 2: Get max version per member to increment
    max_versions = (existing_dim
        .groupBy("member_id_hash")
        .agg(F.max("version").alias("max_version")))

    # Step 3: Insert new version rows
    new_versions = (new_masked
        .join(max_versions, "member_id_hash", "left")
        .withColumn("version", F.coalesce(F.col("max_version") + 1, F.lit(1)))
        .withColumn("effective_start_date", F.to_date(F.lit(today)))
        .withColumn("effective_end_date",   F.to_date(F.lit("9999-12-31")))
        .withColumn("is_current", F.lit(True))
    )

    # Step 4: Union all parts
    unchanged = existing_dim.join(
        changed.select("member_id_hash"), "member_id_hash", "left_anti")

    return unchanged.unionByName(closed_old).unionByName(new_versions)

def build_fact_claims(medical_df, pharmacy_df, dental_df, dim_member):
    """
    Combine all 3 claim types into a unified fact table.
    Join to dim_member using SCD2 logic: match on member_id_hash
    AND service_date BETWEEN effective_start_date AND effective_end_date.
    """
    log.info("Building fact_claims...")

    # Normalize medical claims
    medical_norm = (medical_df
        .withColumn("member_id_hash",
            F.sha2(F.concat(F.col("member_id"), F.lit(PHI_SALT)), 256))
        .withColumn("claim_key",     F.sha2(F.col("claim_id"), 256))
        .withColumn("service_date",  F.to_date("service_date"))
        .withColumn("billed_amount", F.col("billed_amount").cast(DoubleType()))
        .withColumn("allowed_amount",F.col("allowed_amount").cast(DoubleType()))
        .withColumn("paid_amount",   F.col("paid_amount").cast(DoubleType()))
        .withColumnRenamed("rendering_npi", "provider_npi")
        .select("claim_key", "claim_id", "member_id_hash", "provider_npi",
                "service_date", "billed_amount", "allowed_amount", "paid_amount",
                "member_cost_share", "claim_status", "denial_reason_code",
                "primary_diagnosis", "cpt_code", "place_of_service", "claim_type")
    )

    # Normalize pharmacy claims (different schema)
    pharmacy_norm = (pharmacy_df
        .withColumn("member_id_hash",
            F.sha2(F.concat(F.col("member_id"), F.lit(PHI_SALT)), 256))
        .withColumn("claim_key",     F.sha2(F.col("claim_id"), 256))
        .withColumn("service_date",  F.to_date("fill_date"))
        .withColumn("billed_amount", F.col("drug_cost").cast(DoubleType()))
        .withColumn("allowed_amount",F.col("drug_cost").cast(DoubleType()))
        .withColumn("paid_amount",   F.col("plan_paid").cast(DoubleType()))
        .withColumn("member_cost_share", F.col("member_copay").cast(DoubleType()))
        .withColumn("denial_reason_code", F.lit(None).cast(StringType()))
        .withColumn("place_of_service", F.lit("Pharmacy"))
        .withColumnRenamed("pharmacy_npi", "provider_npi")
        .withColumnRenamed("related_diagnosis", "primary_diagnosis")
        .withColumnRenamed("ndc_code", "cpt_code")  # reuse column, different code type
        .select("claim_key", "claim_id", "member_id_hash", "provider_npi",
                "service_date", "billed_amount", "allowed_amount", "paid_amount",
                "member_cost_share", "claim_status", "denial_reason_code",
                "primary_diagnosis", "cpt_code", "place_of_service", "claim_type")
    )

    # Union all claim types
    all_claims = medical_norm.union(pharmacy_norm)

    # SCD2 join: find the correct member dim version for each claim's service date
    fact = (all_claims
        .join(
            F.broadcast(dim_member.select(
                "member_key", "member_id_hash",
                "effective_start_date", "effective_end_date"
            )),
            on=[
                all_claims.member_id_hash == dim_member.member_id_hash,
                all_claims.service_date >= dim_member.effective_start_date,
                all_claims.service_date <= dim_member.effective_end_date,
            ],
            how="left"
        )
        .withColumn("year",             F.year("service_date"))
        .withColumn("month",            F.month("service_date"))
        .withColumn("claim_type_part",  F.col("claim_type"))
        .withColumn("date_key",
            F.date_format("service_date", "yyyyMMdd").cast(IntegerType()))
        .withColumn("etl_inserted_at", F.current_timestamp())
        .drop("effective_start_date", "effective_end_date")
    )
    log.info(f"  → {fact.count()} combined fact rows")
    return fact

Step 5 · Incremental Load

Incremental Load with Watermark

Instead of reprocessing all 50,000 claims every day, only process claims that arrived since the last successful run. This is what reduces runtime from 8 hours to under 2 hours.

💡
Watermark pattern: After each successful run, save the timestamp of the latest record processed. Next run starts from that timestamp. If a run fails, the watermark isn't updated — so the next run reprocesses the same window safely (idempotent).
scripts/incremental_load.py
"""
Incremental load logic using watermark pattern.
Tracks the last successful run timestamp.
Only processes records newer than the watermark.
"""
import json, os
from datetime import datetime
from pyspark.sql import SparkSession, DataFrame
from pyspark.sql import functions as F

WATERMARK_FILE = "config/watermark.json"

def read_watermark() -> str:
    """Read last successful run timestamp. Default to 30 days ago."""
    if os.path.exists(WATERMARK_FILE):
        with open(WATERMARK_FILE) as f:
            wm = json.load(f)
        print(f"Watermark loaded: {wm['last_success_ts']}")
        return wm["last_success_ts"]
    else:
        # First run: process last 30 days
        default = "2024-01-01 00:00:00"
        print(f"No watermark found. Starting from: {default}")
        return default

def save_watermark(ts: str):
    """Save watermark ONLY after successful processing."""
    os.makedirs("config", exist_ok=True)
    with open(WATERMARK_FILE, "w") as f:
        json.dump({
            "last_success_ts": ts,
            "saved_at":        datetime.utcnow().isoformat(),
        }, f, indent=2)
    print(f"Watermark saved: {ts}")

def filter_incremental(df: DataFrame, date_col: str, watermark_ts: str) -> DataFrame:
    """
    Filter DataFrame to only rows where date_col > watermark.
    This is the core incremental logic.
    """
    filtered = df.filter(F.col(date_col) > F.lit(watermark_ts))
    new_count = filtered.count()
    total     = df.count()
    print(f"  Incremental: {new_count}/{total} rows are new (watermark={watermark_ts})")
    return filtered

def get_max_timestamp(df: DataFrame, date_col: str) -> str:
    """Get the maximum timestamp from processed records to use as next watermark."""
    max_ts = df.agg(F.max(date_col).cast("string")).collect()[0][0]
    return max_ts if max_ts else datetime.utcnow().isoformat()

SQL — Claim Denial Rate Analysis with Running Total
WITH denial_summary AS (
    SELECT
        p.specialty,
        f.denial_reason_code,
        f.claim_type,
        COUNT(*)                    AS total_claims,
        SUM(CASE WHEN f.claim_status = 'DENIED' THEN 1 ELSE 0 END) AS denied_claims,
        SUM(f.billed_amount)        AS total_billed,
        SUM(CASE WHEN f.claim_status = 'DENIED'
                 THEN f.billed_amount ELSE 0 END) AS denied_billed_amount
    FROM fact_claims f
    JOIN dim_provider p ON f.provider_key = p.provider_key
    WHERE f.year = 2024
    GROUP BY p.specialty, f.denial_reason_code, f.claim_type
),
denial_with_rate AS (
    SELECT
        specialty, denial_reason_code, claim_type,
        total_claims, denied_claims, denied_billed_amount,
        ROUND(denied_claims * 100.0 / NULLIF(total_claims, 0), 2) AS denial_rate_pct,
        -- Rank specialties by denial rate within each claim type
        DENSE_RANK() OVER (
            PARTITION BY claim_type
            ORDER BY denied_claims * 100.0 / NULLIF(total_claims, 0) DESC
        ) AS denial_rank,
        -- Running total of denied amount
        SUM(denied_billed_amount) OVER (
            PARTITION BY claim_type
            ORDER BY denied_claims DESC
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) AS running_denied_total
    FROM denial_summary
)
SELECT * FROM denial_with_rate
WHERE denial_rate_pct > 10   -- Flag specialties with >10% denial rate
ORDER BY claim_type, denial_rank;

WITH provider_metrics AS (
    SELECT
        p.provider_key, p.provider_name, p.specialty,
        p.network_status,

    FROM fact_claims f
    JOIN dim_provider p ON f.provider_key = p.provider_key

),
specialty_benchmarks AS (
    SELECT
        specialty,
        PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY avg_paid_per_claim) AS p50_cost,
        PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY avg_paid_per_claim) AS p75_cost,
        PERCENTILE_CONT(0.90) WITHIN GROUP (ORDER BY avg_paid_per_claim) AS p90_cost
    FROM provider_metrics
    GROUP BY specialty
)
SELECT
    pm.provider_name, pm.specialty, pm.network_status,
    pm.claim_count, pm.avg_paid_per_claim,
    sb.p50_cost AS specialty_median_cost,
    ROUND((pm.avg_paid_per_claim - sb.p50_cost) / NULLIF(sb.p50_cost,0) * 100, 1)
        AS pct_above_specialty_median,
    CASE
        WHEN pm.avg_paid_per_claim > sb.p90_cost THEN 'High Cost Outlier'
        WHEN pm.avg_paid_per_claim > sb.p75_cost THEN 'Above Average'
        WHEN pm.avg_paid_per_claim < sb.p50_cost THEN 'Below Average'

FROM provider_metrics pm

SQL — Member Risk Stratification (80/20 Rule)
WITH member_utilization AS (
    SELECT
        f.member_key,
        m.plan_type,
        m.state,
        COUNT(DISTINCT f.claim_key)          AS total_claims,
        COUNT(DISTINCT f.date_key)            AS service_days,
        SUM(f.paid_amount)                    AS total_paid,
        SUM(f.member_cost_share)              AS total_member_cost,
        COUNT(DISTINCT f.diagnosis_key)       AS unique_diagnoses,
        COUNT(CASE WHEN f.place_of_service = 'ER' THEN 1 END) AS er_visits,
        COUNT(CASE WHEN f.place_of_service = 'Inpatient' THEN 1 END) AS inpatient_admits
    FROM fact_claims f
    JOIN dim_member m ON f.member_key = m.member_key
        AND m.is_current = TRUE    -- Join only to current member record
    WHERE f.year = 2024
      AND f.claim_status = 'PAID'
    GROUP BY f.member_key, m.plan_type, m.state
),
population_stats AS (
    SELECT
        SUM(total_paid) AS pop_total_paid,
        COUNT(*)        AS total_members
    FROM member_utilization
)
SELECT
    mu.member_key, mu.plan_type, mu.state,
    mu.total_claims, mu.er_visits, mu.inpatient_admits,
    mu.unique_diagnoses,
    ROUND(mu.total_paid, 2) AS total_paid,
    -- What % of total population spend does this member represent?
    ROUND(mu.total_paid / ps.pop_total_paid * 100, 3) AS pct_of_total_spend,
    -- Cumulative % to find top 20% who drive 80% of cost
    ROUND(SUM(mu.total_paid) OVER (
        ORDER BY mu.total_paid DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) / ps.pop_total_paid * 100, 2) AS cumulative_spend_pct,
    CASE
        WHEN mu.er_visits >= 3 OR mu.inpatient_admits >= 2  THEN 'Tier 1 - Complex'
        WHEN mu.total_paid > 10000 OR mu.unique_diagnoses >= 5 THEN 'Tier 2 - High Risk'
        WHEN mu.total_paid > 3000                              THEN 'Tier 3 - Moderate'
        ELSE                                                          'Tier 4 - Low Risk'
    END AS risk_tier
FROM member_utilization mu, population_stats ps
ORDER BY mu.total_paid DESC;

WITH plan_monthly AS (
    SELECT
        m.plan_type,
        d.year, d.month, d.quarter,
        COUNT(DISTINCT f.member_key)        AS unique_members,
        COUNT(f.claim_key)                  AS total_claims,
        SUM(f.paid_amount)                  AS total_paid,
        SUM(f.paid_amount)
            / NULLIF(COUNT(DISTINCT f.member_key),0)
                                            AS pmpm,  -- Per Member Per Month (key actuarial metric)
        AVG(f.paid_amount)                  AS avg_claim_cost,
        SUM(CASE WHEN f.place_of_service = 'ER' THEN 1 ELSE 0 END)
            * 1000.0 / NULLIF(COUNT(DISTINCT f.member_key),0)
                                            AS er_visits_per_1000  -- Standard utilization metric
    FROM fact_claims f
    JOIN dim_member m ON f.member_key = m.member_key AND m.is_current = TRUE
    JOIN dim_date d   ON f.date_key = d.date_key
    WHERE f.claim_status = 'PAID' AND f.year = 2024
    GROUP BY m.plan_type, d.year, d.month, d.quarter
)
SELECT
    plan_type, year, month,
    unique_members, total_claims,
    ROUND(pmpm, 2)              AS pmpm,
    ROUND(avg_claim_cost, 2)   AS avg_claim_cost,
    ROUND(er_visits_per_1000,1) AS er_per_1000,
    -- Month over month PMPM change per plan
    ROUND(pmpm - LAG(pmpm) OVER (
        PARTITION BY plan_type ORDER BY year, month
    ), 2) AS pmpm_mom_change
FROM plan_monthly
ORDER BY plan_type, year, month;

"""Load processed Parquet from S3 into Redshift using COPY command."""
import psycopg2
import os

# Store credentials in environment variables — never hardcode
REDSHIFT_CONFIG = {
    "host":     os.environ.get("REDSHIFT_HOST"),
    "port":     5439,
    "database": "healthcare",
    "user":     os.environ.get("REDSHIFT_USER"),
    "password": os.environ.get("REDSHIFT_PASSWORD"),
}

IAM_ROLE       = "arn:aws:iam::YOUR_ACCOUNT:role/RedshiftS3ReadRole"
PROC_BUCKET    = "healthcare-processed-yourname"

def load_table(cursor, table_name, s3_prefix):
    """Truncate and reload a table from S3 Parquet files."""
    print(f"Loading {table_name}...")
    cursor.execute(f"TRUNCATE healthcare.{table_name};")
    cursor.execute(f"""
        COPY healthcare.{table_name}
        FROM 's3://{PROC_BUCKET}/{s3_prefix}/'
        IAM_ROLE '{IAM_ROLE}'
        FORMAT AS PARQUET;
    """)
    print(f"  ✓ {table_name} loaded")

def main():
    conn = psycopg2.connect(**REDSHIFT_CONFIG)
    conn.autocommit = False
    cursor = conn.cursor()
    try:
        load_table(cursor, "dim_member",   "dim_member")
        load_table(cursor, "dim_provider", "dim_provider")
        load_table(cursor, "dim_date",     "dim_date")
        load_table(cursor, "fact_claims",  "fact_claims")
        conn.commit()
        print("✅ All tables loaded successfully")
    except Exception as e:
        conn.rollback()
        print(f"❌ Load failed: {e}")
        raise
    finally:
        cursor.close(); conn.close()

if __name__ == "__main__":
    main()

Q1HIPAA What is PHI and how did you handle it in your pipeline?
Model Answer

PHI — Protected Health Information — is any data that can identify a patient and relates to their health condition, treatment, or payment. The 18 HIPAA identifiers include names, dates, SSNs, and member IDs. In my pipeline I applied a three-layer strategy: First, the raw bucket is KMS-encrypted and access-restricted — only the ETL role can read it. Second, during transformation I apply SHA-256 salted hashing to member IDs so analysts can join data without seeing the raw ID, replace names with initials, convert full DOBs to birth year only, and drop SSNs entirely. Third, I write an audit log for every ETL run recording which tables were accessed, how many rows, and when — which is a HIPAA audit trail requirement. The processed bucket contains zero raw PHI — it's safe for analysts to query.

Q2Architecture Why did you use snowflake schema instead of star schema for this project?
Model Answer

Healthcare has deep, standardized code hierarchies that make star schema impractical. ICD-10 diagnosis codes have 70,000+ individual codes organized into chapters, blocks, and categories. If I stored all hierarchy attributes in one dim_diagnosis table, I'd repeat the chapter name and category description 70,000 times. Snowflake schema normalizes these into a dim_diagnosis_category sub-dimension, which is more storage-efficient and makes hierarchy-level queries cleaner. The tradeoff is more JOINs in queries — but healthcare analysts are used to this, and the schema matches how the medical coding systems are actually structured. I'd use star schema if the dimensions were flat, like in the e-commerce project where product attributes don't have meaningful hierarchies.

Q3Architecture Explain SCD Type 1 vs Type 2. Why did you choose Type 2 for dim_member?
Model Answer

SCD stands for Slowly Changing Dimension — a dimension whose attributes change over time. Type 1 overwrites the old value — no history is kept. If a member switches from PPO to HMO, the record just shows HMO. The problem: historical claims that were filed under the PPO plan now incorrectly appear as HMO claims. Type 2 inserts a new row and closes the old one using effective_start_date, effective_end_date, and an is_current flag. The PPO row gets effective_end_date = day before the change. The new HMO row gets effective_start_date = change date and effective_end_date = 9999-12-31. This means a March 2024 claim — filed under PPO — still correctly joins to the PPO member record. I chose Type 2 because insurance plan attribution is critical for actuarial analysis — the plan type directly affects payment rates, deductibles, and network restrictions. Getting historical plan wrong corrupts every financial calculation.

Q4Incremental Walk me through your incremental load design. What happens if the pipeline fails?
Model Answer

My incremental load uses a watermark pattern. After each successful run, I save the maximum service_date (or claim_received_date) of the processed batch to a watermark file. The next run reads that watermark and only processes records newer than that timestamp. If the pipeline fails mid-run, I don't update the watermark — the failure short-circuits before the save step. So the next run reprocesses the same window from the same watermark. This makes it safe to re-run without creating duplicates, as long as the write uses partition overwrite or a MERGE/upsert pattern — which mine does. The key insight is that watermark advancement is the very last step, after all processing and writing succeeds. This is the same principle as a two-phase commit in distributed systems — don't mark something as done until it's actually done.

Q5Incremental What is idempotency and why does it matter for data pipelines?
Model Answer

An idempotent operation produces the same result no matter how many times you run it. For data pipelines this means: running the pipeline twice for the same input data produces the same output — no duplicates, no data corruption. It matters because pipelines fail and need to be retried. If re-running creates duplicate records, you've corrupted your data warehouse. I achieve idempotency in three ways: First, watermark-based incremental loads with partition overwrite — re-processing the same date partition writes the same data over itself. Second, using SHA-256 hashes as surrogate keys — the same source record always generates the same key, so you can safely upsert. Third, the audit log allows detecting and skipping already-processed files. Without idempotency, every failure becomes a crisis because you have to manually figure out what got written and what didn't.

Q6PySpark You joined three different claim types with different schemas. How did you handle that?
Model Answer

Each claim type has a slightly different schema — pharmacy claims have a fill_date instead of service_date, drug_cost instead of billed_amount, and no CPT code (they use NDC codes instead). I built a normalization function for each claim type that maps these source-specific columns to a common schema. For example, pharmacy's fill_date becomes service_date, drug_cost becomes billed_amount, and NDC code is stored in the CPT code column with a note that it's a different coding system. After normalization, all three claim types have identical schemas and can be safely unioned with Spark's unionByName. In production I'd document this mapping clearly and add a source_claim_type column so downstream users know that the cpt_code column has different semantics for pharmacy vs medical claims. This is a common data engineering challenge — harmonizing heterogeneous sources into a single analytical schema.

Q7PySpark Explain your SCD Type 2 join in PySpark. Why is it different from a normal join?
Model Answer

A normal dimension join matches on a single key: fact.member_key = dim_member.member_key. But SCD Type 2 has multiple rows per member — one for each plan version. If I joined just on member_key, a claim could match multiple member records. The SCD2 join adds a date range condition: claim's service_date must fall between the dim record's effective_start_date and effective_end_date. In PySpark, this means three join conditions: member_id_hash equality, service_date >= effective_start_date, and service_date <= effective_end_date. This is more expensive than a simple equi-join because it can't use hash join — Spark typically falls back to sort-merge join or a broadcast join if one side is small. I broadcast the dim_member table since it's much smaller than fact_claims, which avoids a full shuffle on the large claim table.

Q8Architecture Why did you use Redshift instead of Athena for the serving layer?
Model Answer

Athena is serverless and great for ad-hoc queries, but it has limitations for a high-traffic healthcare analytics environment. Each Athena query scans S3 and incurs variable latency — sometimes 5 seconds, sometimes 30 depending on data volume. Redshift provides consistent, sub-second query performance because data is stored locally in columnar format on the cluster's SSDs, not read from S3 each time. For actuaries running the same PMPM and denial rate queries daily with slight filter variations, the predictable performance matters. Redshift also supports DISTKEY and SORTKEY — I distributed fact_claims on member_key so member-centric queries can be processed node-locally without network shuffles, and sorted on service_date for fast range scans. The tradeoff is cost — Redshift charges for the cluster even when idle — while Athena charges only for data scanned. For workloads with predictable, frequent queries, Redshift wins on total cost. For truly ad-hoc one-off analysis, Athena wins.

Q9SQL What is PMPM and how did you calculate it?
Model Answer

PMPM stands for Per Member Per Month — the standard unit of cost measurement in health insurance actuarial work. It answers: "How much does the payer spend per enrolled member per month?" It's calculated as total paid claims divided by the number of unique enrolled members for that month. It normalizes for membership size changes — a plan with 10,000 members spending $2M and one with 5,000 members spending $1M both have the same $200 PMPM. In my query I calculated it as SUM(paid_amount) / COUNT(DISTINCT member_key) grouped by plan_type, year, and month. I also added month-over-month PMPM change using the LAG window function, which lets actuaries quickly see cost trends. PMPM is the single most important metric for setting insurance premiums and evaluating plan performance — if your PMPM is rising faster than you built into your premium rates, the plan loses money.

Q10HIPAA Why did you use SHA-256 salted hashing for member IDs instead of just MD5?
Model Answer

Two reasons. First, SHA-256 is more collision-resistant than MD5 — MD5 has known collision vulnerabilities that make it unsuitable for security-sensitive applications. For PHI, you need a cryptographically secure hash. Second, the salt — a secret string appended before hashing — prevents rainbow table attacks. A rainbow table is a pre-computed lookup of common values and their hashes. Without a salt, someone with access to your hashed member IDs could compare them against a rainbow table and reverse-engineer the original IDs. The salt makes every hash in your dataset unique to your pipeline, so even if someone has common member IDs to try, they'd need your specific salt to match them. The salt is stored in AWS Secrets Manager, not in code. The benefit is that analysts can still JOIN across tables using the hashed ID — the hash is deterministic, so the same member_id always generates the same hash — but they can't see or reverse the original PHI.

Q11Architecture How does your Redshift DISTKEY improve query performance?
Model Answer

Redshift distributes data across nodes. Without a DISTKEY, Redshift uses round-robin distribution — rows go to nodes randomly. When a query joins fact_claims to dim_member on member_key, Redshift has to redistribute data across the network to bring matching rows together — this is expensive. With DISTKEY(member_key) on both fact_claims and dim_member, Redshift ensures rows with the same member_key always land on the same node. The join can now happen locally on each node with no network I/O. This is called a co-located join. For healthcare analytics where almost every query joins claims to member demographics, this DISTKEY choice can reduce join time by 5-10x on large datasets. The SORTKEY(service_date) means rows are physically sorted by date on each node's storage blocks — when you filter WHERE service_date BETWEEN X AND Y, Redshift can skip entire blocks that fall outside the range, using zone maps.

Q12Incremental How would you handle late-arriving claims? (Claims filed 60 days after service date)
Model Answer

Late-arriving data is a critical problem in healthcare — claims can be filed 60-90+ days after the service date. If my watermark is based on service_date, I might miss claims that arrive today for services from 90 days ago. I'd handle this with a two-date strategy: filter on claim_received_date for the watermark (what's new in the system today), but partition on service_date (when the service happened). This means late-arriving claims for old service dates would re-trigger a re-load of those older service_date partitions. To make this idempotent, I'd use partition overwrite — not append — so re-running a service_date partition replaces the data rather than duplicating it. I'd also add a reconciliation job that runs monthly to catch any claims that slipped through by comparing claim totals in the source system against the warehouse. In production, I'd also use an event-driven trigger via S3 notifications + Lambda rather than polling, so I process new files as soon as they land.

Q13SQL Explain PERCENTILE_CONT in your provider performance query.
Model Answer

PERCENTILE_CONT is an ordered-set aggregate function that calculates the value at a specific percentile of a distribution, interpolating if needed. PERCENTILE_CONT(0.50) gives the median — the midpoint value. PERCENTILE_CONT(0.75) gives the 75th percentile — 75% of values are below this. I used it to benchmark each provider's average claim cost against their specialty's distribution. A provider at the 90th percentile is a high-cost outlier — they're more expensive than 90% of similar providers. This is more meaningful than comparing to the mean, which gets distorted by extreme outliers. For example, if one hospital does $500K surgeries, it pulls the mean up and makes all other providers look cheap. Median is robust to these outliers. PERCENTILE_DISC is the discrete version — it returns an actual value from the dataset. PERCENTILE_CONT interpolates, which gives a smoother curve. For cost analysis, PERCENTILE_CONT is typically preferred.

Q14Production How would you set up monitoring and alerting for this pipeline?
Model Answer

Four layers of monitoring. First, pipeline health metrics in CloudWatch: job run duration, row counts per table, quarantine rate percentage. I'd alert if the quarantine rate exceeds 2% — that signals an upstream data quality problem. Key Concepts

Deep Dive Concepts

Master these before your next interview. One concept per day in Week 4.

❄️
Star vs Snowflake Schema — Full Comparison
▼

Star: Fact table + denormalized dimensions. 1 JOIN per dimension. Faster queries, simpler SQL, uses more storage. Use when: dimensions are relatively flat, BI tool performance matters, analyst ease of use is priority.

Snowflake: Fact table + normalized dimensions + sub-dimensions. Multiple JOINs. Slower queries, more complex SQL, less storage. Use when: dimensions have deep hierarchies (ICD codes, org charts), dimension tables are very large, storage cost matters more than query speed.

Real world: Most production warehouses are a hybrid — some dimensions are star-like, others snowflake-like. The key is to make the decision consciously based on query patterns and dimension structure, not default to one approach.

🔄
SCD Types 1, 2, and 3
▼

Type 1 — Overwrite: Simple, no history. Use when: the old value was wrong (data correction), history doesn't matter for analysis, dimension table is huge and history would multiply storage.

Type 2 — New row: Full history with effective dates. Use when: the change itself is analytically important (member plan change, customer address change), you need point-in-time reporting, regulatory requirements need full history.

Type 3 — Add column: Add a "previous_value" column. e.g., plan_type + previous_plan_type. Limited history (only 1 change back). Use when: you only need to compare current vs previous, not full history. Rare in practice.

Type 4 — History table: Main dim has current values. Separate history table has all changes. A variant of Type 2, used when the main dim needs to stay clean for operational queries.

📈
Incremental vs Full Load Patterns
▼

Full load: Read everything, write everything. Simple. Slow. Use for: small dimensions, initial loads, when you can't trust the source to properly mark new records.

Incremental — Timestamp watermark: Track last processed timestamp, filter on it. Works when source has a reliable updated_at column. Most common pattern.

Incremental — CDC (Change Data Capture): Capture every INSERT/UPDATE/DELETE from the source database's transaction log (MySQL binlog, Postgres WAL). Most reliable, requires DB access. Tools: Debezium, AWS DMS.

Incremental — File-based: Track which files have been processed. Works for file-based sources like our healthcare CSV drops. Store processed filenames in a manifest file.

Critical rule: Whatever pattern you use, always have a backfill mechanism for when you need to reprocess historical data.

🏗
Redshift Architecture Essentials
▼

Columnar storage: Like Parquet — stores data by column, not row. Dramatically reduces I/O for analytical queries that only need a few columns from wide tables.

DISTKEY: How rows are distributed across compute nodes. Rule: set DISTKEY to the column most commonly used in JOINs. Tables joined on the same DISTKEY will have matching rows on the same node (co-located join) — no network shuffle needed.

SORTKEY: How rows are physically sorted on each node's storage. Rule: set SORTKEY to columns used in WHERE clauses (especially date ranges). Redshift uses zone maps to skip entire 1MB blocks that don't match the filter.

COPY command: The only efficient way to bulk-load data into Redshift. Never INSERT row-by-row. COPY reads from S3 in parallel across all compute nodes simultaneously.

VACUUM + ANALYZE: After large loads, run VACUUM to reclaim space from deleted rows and re-sort data, then ANALYZE to update table statistics for the query planner.
