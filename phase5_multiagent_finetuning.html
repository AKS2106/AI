Phase 5 complete guide — multi-agent systems, supervisor patterns, agent communication, fine-tuning, LoRA, PEFT

There are 4 orchestration patterns. Each solves a different coordination problem. Knowing when to use each is what separates senior candidates from junior ones.
In practice: most production systems use the supervisor pattern. It's the most debuggable and the most common in LangGraph codebases. If an interviewer asks you to design a multi-agent system, default to supervisor unless there's a specific reason not to.
How agents pass information between each other is the most underspecified part of most tutorials. There are three mechanisms, each with different tradeoffs.
Mechanism 1 — shared state (LangGraph default)
All agents read from and write to a single AgentState object. The supervisor and all workers see the same data. LangGraph's reducers handle concurrent writes safely.
class SharedState(TypedDict): messages: Annotated[list, add_messages] # append-only research_output: str # researcher writes, writer reads draft: str # writer writes, reviewer reads review_notes: list[str] # reviewer writes, writer re-reads final_output: str # completed artifact next_agent: str # supervisor writes who goes next
Best for: sequential pipelines where output of one agent is input to next. Simplest to reason about — one source of truth.
Mechanism 2 — message passing (human-readable handoffs)
Agents communicate by adding structured messages to the shared message list. Each agent reads the conversation history and extracts what it needs. More flexible but harder to parse reliably.
class AgentHandoff(BaseModel): target_agent: str # "researcher" | "writer" | "reviewer" task: str # what to do context: str # relevant background deliverable: str # exact output expected # Supervisor emits this as a ToolMessage # Target agent sees it as part of message history
Best for: dynamic routing where the supervisor decides mid-run who to involve next.
Mechanism 3 — subgraph invocation (strongest isolation)
Each agent is a compiled subgraph. The supervisor calls a subgraph like a function, passing input state and receiving output state. Agent internals are fully hidden from the supervisor.
researcher_graph = build_researcher().compile() writer_graph = build_writer().compile() def supervisor_node(state): # Call researcher as a black box research = researcher_graph.invoke({"task": state["task"]}) # Pass its output to the writer draft = writer_graph.invoke({ "research": research["output"], "style": state["style_guide"] }) return {"draft": draft["output"]}
Best for: reusable agent components, independent testing of each agent, large teams where different engineers own different agents.
Interview answer for "how do agents communicate in your system?": "I use shared state as the primary mechanism — it's a single TypedDict everyone reads and writes with LangGraph reducers handling conflicts. For complex handoffs I add structured Pydantic models to the state so the compiler catches type errors at build time, not runtime."
Fine-tuning is a continuum, not a binary choice. From lightest to heaviest intervention — each level adds quality and removes flexibility.
The fine-tuning spectrum
1. Prompt engineering zero training cost
No weight updates. Instructions in system prompt. Deploy in hours. Loses effect if model changes.
2. LoRA / PEFT fine-tuning low cost, days
Train small adapter layers only. 100–10k examples. Single GPU viable. The industry standard for open-source models.
3. Full fine-tuning high cost, weeks
Update all weights. Needs thousands of examples, multi-GPU clusters. Risk of catastrophic forgetting. Used sparingly.
4. RLHF / DPO alignment very high cost
Train a reward model on human preference data, then optimize the LLM against it. How Claude and ChatGPT are aligned. Not for application engineers.
Training data format (what your JSONL file looks like)
{"messages": [ {"role": "system", "content": "You are a SQL expert..."}, {"role": "user", "content": "Get all users created after 2024-01-01"}, {"role": "assistant", "content": "SELECT * FROM users WHERE created_at > '2024-01-01'"} ]} {"messages": [ {"role": "system", "content": "You are a SQL expert..."}, {"role": "user", "content": "Count orders by status"}, {"role": "assistant", "content": "SELECT status, COUNT(*) as count FROM orders GROUP BY status"} ]}
Catastrophic forgetting — when you fine-tune aggressively on a narrow task, the model partially "forgets" general capabilities. Example: fine-tune a coding assistant on Python snippets and it starts struggling with English explanations. LoRA avoids this because general weights stay frozen. For full fine-tuning, use a low learning rate and mix in ~10% general data.
LoRA is the most important fine-tuning technique for application engineers. It makes fine-tuning feasible on a single GPU by training only a tiny fraction of the total parameters.
The core idea — low-rank decomposition
A weight matrix W in a transformer layer might be 4096×4096 = 16.7M parameters. Instead of updating all of them, LoRA adds two small matrices A (4096×r) and B (r×4096) where r is the "rank" — typically 4–64. The update becomes W + AB, where A and B together have only 2×4096×r parameters.

PEFT techniques — the family LoRA belongs to
LoRA — Low-Rank Adaptation. Adds trainable rank decomposition matrices to attention layers. The standard choice for most tasks.
QLoRA — Quantized LoRA. First quantizes the base model to 4-bit (halving memory), then applies LoRA. Enables fine-tuning 70B models on a single 48GB GPU. The default for resource-constrained environments.
Prefix tuning — Prepends learnable "soft tokens" to every layer. No weight changes at all. Faster training, weaker adaptation. Good for very small datasets (<100 examples).
IA³ — Infused Adapter by Inhibiting and Amplifying Inner Activations. Trains even fewer parameters than LoRA by scaling activations. Extreme efficiency for very large models.
Key hyperparameters (you'll be asked to explain these)
r (rank) — higher = more expressiveness = more params. Start at 8–16. Double if results are poor. Rarely need above 64.
alpha (α) — scaling factor for the LoRA update. Usually set to 2×r or equal to r. Controls how strongly the adapter influences the base model.
target_modules — which layers to adapt. Default: q_proj and v_proj (query + value in attention). Adding more layers (k_proj, o_proj, mlp) increases quality and cost.
dropout — regularization on LoRA layers. Set 0.05–0.1 to prevent overfitting on small datasets. Set 0 for large datasets (>10k examples).
Evaluation is what makes multi-agent systems and fine-tuned models trustworthy in production. Without evals, you're deploying blind.
Multi-agent system evaluation
Task completion rate — did the agent system complete the task without hitting max_iterations or erroring out? Track this per task type and per agent.
Step efficiency — how many LLM calls did it take? A task that should take 3 steps but averages 8 has a planning or tool-use problem.
Handoff accuracy — did the supervisor route to the right sub-agent? Track cases where the wrong agent was invoked. This reveals flaws in routing prompts.
Error cascade rate — when one agent fails, how often does the whole system fail vs. recover? Good systems have explicit fallback paths.
Fine-tuning evaluation metrics
Task-specific accuracy — does it do the thing you fine-tuned it for? For SQL generation: does the query run and return correct results? For classification: precision/recall/F1.
Format compliance rate — does it always output valid JSON / follow the schema? Should be close to 100% after fine-tuning — if not, add more format examples.
General capability retention — run MMLU or HellaSwag benchmarks on your fine-tuned model vs. the base. Catastrophic forgetting shows up as drops here. Flag anything >5% degradation.
Held-out eval set — 10–20% of your labeled data never used in training. This is the only honest measure. If train accuracy is high but eval accuracy is low, you've overfit.
The evaluation cadence that works in production: run evals on every PR that changes a prompt, agent graph, or training dataset. Gate deployments on eval score ≥ threshold. Store eval results in a time-series DB so you can see regressions over time. LangSmith and Braintrust are the standard tools for this.
The 7 hardest questions at senior level
Q: "How do you prevent error cascades in a multi-agent system?"
"Three layers: (1) Each agent returns a structured result with a success flag and error field — the supervisor reads these instead of assuming success. (2) Explicit retry logic in the supervisor: if a worker fails, retry once with a rephrased task before escalating. (3) Fallback paths: for critical steps, the supervisor has an alternative agent or a simplified version of the task. The goal is that any single agent failure degrades performance, not crashes the system."
Q: "How do you debug a multi-agent system when something goes wrong?"
"Full trace logging is non-negotiable. Every supervisor decision, every handoff, every agent's Thought/Action/Observation chain gets logged with timestamps and token counts. I use LangSmith for this — it captures the entire execution tree so I can replay any run, see exactly which agent produced the wrong output, and trace it back to the prompt or tool that caused it. Without this you're debugging distributed systems blind."
Q: "What's the difference between LoRA rank 8 and rank 64?"
"Higher rank = more trainable parameters = more expressive adaptation but more risk of overfitting on small datasets. Rank 8 is ~0.4% of a large attention layer's parameters — enough to learn format and domain-specific vocabulary. Rank 64 is ~3% — better for learning complex reasoning patterns or code generation on a specific framework. I start at r=8, evaluate on held-out data, and double if the model clearly hasn't converged. Going above r=64 rarely helps for instruction-following tasks."
Q: "When would you use a multi-agent system vs a single agent with many tools?"
"Single agent when: the task is bounded, tools are independent, and context fits in one window. Multi-agent when: subtasks are long enough to fill a context window on their own, subtasks need different expertise (researcher vs. writer vs. code reviewer), or subtasks can run in parallel to reduce latency. The coordination overhead of multi-agent only pays off when the task genuinely can't fit in one context window or when parallelism provides meaningful speedup."
Q: "What is QLoRA and why is it important?"
"QLoRA = quantized LoRA. It first compresses the base model to 4-bit precision using NF4 quantization — roughly halving memory usage with minimal quality loss — then applies LoRA adapters on top. The practical result: you can fine-tune a 70B parameter model on a single 48GB A100 GPU that would otherwise require 8×80GB GPUs. This is the technique that made fine-tuning large open-source models accessible to small teams. The paper (Dettmers et al., 2023) is worth reading."
Q: "How do you handle long-running multi-agent tasks that might take minutes or hours?"
"Three things: (1) Async execution — run the graph in a background task, return a job ID immediately, poll for completion. (2) Checkpointing — LangGraph's persistence layer saves state after every node execution. If the process crashes, it resumes from the last checkpoint, not from scratch. (3) Human-in-the-loop breakpoints — for tasks over 5 minutes, add interrupt_before checkpoints so a human can review intermediate outputs and redirect before wasted computation."
Q: "What data do you need to fine-tune a model, and how do you get it?"
"You need (input, ideal_output) pairs in the task format. Sources: (1) Existing labeled data in your system — support tickets with resolutions, code reviews with comments. (2) LLM-generated data — use GPT-4 to generate 500 examples, human-review 10% for quality. (3) Few-shot bootstrapping — build a working few-shot system first, capture its outputs + human corrections as training data. The minimum viable dataset is 100–200 high-quality examples for format learning, 500+ for behavioral change, 5k+ for reliable knowledge injection."
The answer that closes the interview
"The pattern I've seen work in production is: start with a single ReAct agent and good prompts. Add multi-agent structure only when you hit a concrete scaling limit — context window, latency, or expertise isolation. Fine-tune only when you've validated the task with prompting and need consistency at scale. Each layer of complexity has a real cost: multi-agent adds coordination failures and debugging surface area; fine-tuning adds a training pipeline, eval infrastructure, and redeployment cycles. The best AI engineers I know are conservative about adding these layers — they're not inherently better, just better for specific problems."
