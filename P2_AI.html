What you're building
Business Problem & Goal
Enterprise knowledge bases contain thousands of documents — policies, FAQs, guides, runbooks. Employees waste hours searching for answers. A RAG system indexes every document and lets anyone ask questions in plain English, with cited sources.
📄

What it ingests

PDF, DOCX, and TXT files. Loads them, splits into chunks, embeds each chunk, stores in ChromaDB with metadata (filename, page, chunk index).

🔍

What it returns

A natural language answer plus the exact source chunks it used — filename, page number, and the raw text. No hallucinations, grounded answers only.

📊

How you prove it works

RAGAS evaluation: faithfulness, answer relevance, context precision — real scores you can quote in interviews. Target: faithfulness > 0.85.

Key Concepts You Must Know Cold

ConceptWhat It IsWhy Interviewers Ask
ChunkingSplitting large documents into smaller pieces (e.g. 512 tokens). Each chunk is embedded and stored separately.Chunk size is the #1 RAG tuning parameter. You must explain the tradeoff: small chunks = precise retrieval but missing context. Large chunks = more context but noisier retrieval.
Chunk OverlapEach chunk shares N tokens with the next. e.g. 50-token overlap prevents answers from being cut exactly at a chunk boundary.Without overlap, a sentence that spans two chunks may never be retrieved correctly. Shows you understand edge cases.
EmbeddingsDense vector representation of text (~384 or 1536 floats). Similar meaning = similar vectors. The all-MiniLM-L6-v2 model runs FREE on your Mac M5.Interviewers want to know: which model you chose, why, and the cost vs quality tradeoff vs OpenAI embeddings.
Cosine SimilarityHow we measure if two embeddings are similar. Returns -1 to 1. Above 0.8 = very similar. This is how ChromaDB finds relevant chunks.Be ready to explain this without code: "It measures the angle between two vectors in high-dimensional space."
Top-K RetrievalReturn the K most similar chunks to the query. K=5 is a common default. Too low = miss relevant info. Too high = add noise.Shows you understand the precision-recall tradeoff in retrieval.
GroundingInstructing the LLM to ONLY use the provided context to answer, never its training data. Prevents hallucination.This is the entire point of RAG. If your prompt doesn't enforce grounding, the LLM will ignore the context and hallucinate.
RAGASRAG Assessment framework. 4 automated metrics: Faithfulness, Answer Relevance, Context Precision, Context Recall.You must be able to define each metric without looking it up. Most candidates cannot.

RAGAS Metrics — Know These by Heart

MetricSimple DefinitionTarget ScoreHow It Fails
FaithfulnessDoes the answer ONLY use facts from the retrieved chunks? (no hallucination)> 0.85LLM adds facts from training data that aren't in the context
Answer RelevanceDoes the answer actually address what the user asked?> 0.80Technically correct but doesn't answer the specific question
Context PrecisionAre the retrieved chunks actually useful for answering the question?> 0.75Retriever returns tangentially related chunks that don't help
Context RecallDid you retrieve ALL the information needed to answer fully?> 0.70The answer exists in the KB but wasn't in the retrieved top-K

Implementation Step 2
Ingestion Pipeline
The pipeline that reads your documents, splits them into chunks, embeds each chunk, and stores everything in ChromaDB. Run once when documents change.
ingestion/document_loader.py
"""Load documents from a folder. Supports TXT, PDF, DOCX."""
import os
from pathlib import Path
from langchain_community.document_loaders import (
    TextLoader, PyPDFLoader, Docx2txtLoader
)
from langchain.schema import Document

LOADERS = {
    ".txt": TextLoader,
    ".pdf": PyPDFLoader,
    ".docx": Docx2txtLoader,
}

def load_documents(folder_path: str) -> list[Document]:
    """Load all supported documents from a folder."""
    docs = []
    folder = Path(folder_path)

    if not folder.exists():
        raise FileNotFoundError(f"Folder not found: {folder_path}")

    for file_path in folder.iterdir():
        suffix = file_path.suffix.lower()
        if suffix not in LOADERS:
            continue
        try:
            loader = LOADERS[suffix](str(file_path))
            loaded = loader.load()
            # Inject source filename into metadata
            for doc in loaded:
                doc.metadata["source_file"] = file_path.name
                doc.metadata["file_type"] = suffix
            docs.extend(loaded)
            print(f"  ✅ Loaded: {file_path.name} ({len(loaded)} pages)")
        except Exception as e:
            print(f"  ❌ Failed to load {file_path.name}: {e}")

    print(f"\nTotal documents loaded: {len(docs)}")
    return docs
ingestion/chunker.py
"""
Split documents into chunks. This is the most critical tuning
parameter in any RAG system. Run the chunk_experiment() to find
your optimal chunk_size before building the full pipeline.
"""
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain.schema import Document

def chunk_documents(
    documents: list[Document],
    chunk_size: int = 512,
    chunk_overlap: int = 50,
) -> list[Document]:
    """Split documents into chunks and add chunk metadata."""
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,
        chunk_overlap=chunk_overlap,
        # Split on these separators, in order of preference
        separators=["\n\n", "\n", ". ", " ", ""],
        length_function=len,
    )

    chunks = splitter.split_documents(documents)

    # Add chunk index to metadata — useful for debugging retrieval
    source_counts = {}
    for chunk in chunks:
        source = chunk.metadata.get("source_file", "unknown")
        source_counts[source] = source_counts.get(source, 0) + 1
        chunk.metadata["chunk_index"] = source_counts[source]
        chunk.metadata["chunk_size"] = chunk_size

    print(f"Created {len(chunks)} chunks from {len(documents)} documents")
    print(f"Avg chunk length: {sum(len(c.page_content) for c in chunks)//len(chunks)} chars")
    return chunks


def chunk_experiment(documents: list[Document]):
    """
    Run experiment with 3 chunk sizes.
    Call this before building your full pipeline —
    screenshot the output for your portfolio/interviews.
    """
    print("\n🧪 CHUNK SIZE EXPERIMENT")
    print("=" * 50)
    for size in [256, 512, 1024]:
        chunks = chunk_documents(documents, chunk_size=size, chunk_overlap=size//10)
        avg_len = sum(len(c.page_content) for c in chunks) // len(chunks)
        print(f"chunk_size={size:4d} | chunks={len(chunks):3d} | avg_chars={avg_len:4d}")
    print("=" * 50)
    print("Now run RAGAS eval with each size and pick the best scorer.\n")
ingestion/embedder.py
"""
Embedding model wrapper. Uses sentence-transformers which runs
100% locally on your Mac M5 — completely free, no API key needed.
"""
import os
from langchain_huggingface import HuggingFaceEmbeddings
from dotenv import load_dotenv

load_dotenv()

def get_embedding_model(model_name: str | None = None) -> HuggingFaceEmbeddings:
    """
    Returns the embedding model. Downloads on first call (~90MB),
    then cached locally. Subsequent calls are instant.

    Benchmark on Mac M5:
    - all-MiniLM-L6-v2:  384 dims, ~5ms per chunk, great quality
    - all-mpnet-base-v2: 768 dims, ~15ms per chunk, better quality
    """
    model = model_name or os.getenv("EMBEDDING_MODEL", "all-MiniLM-L6-v2")
    print(f"Loading embedding model: {model} (first run downloads ~90MB)")

    return HuggingFaceEmbeddings(
        model_name=model,
        model_kwargs={"device": "cpu"},  # Mac M5 uses CPU (MPS also works)
        encode_kwargs={"normalize_embeddings": True},
    )
ingestion/ingest.py
"""
Main ingestion script. Runs the full pipeline:
  Load documents → Chunk → Embed → Store in ChromaDB
Run: python ingestion/ingest.py
"""
import os, shutil
from langchain_community.vectorstores import Chroma
from dotenv import load_dotenv
from ingestion.document_loader import load_documents
from ingestion.chunker import chunk_documents
from ingestion.embedder import get_embedding_model

load_dotenv()

DOCS_FOLDER = "data/sample_docs"
CHROMA_PATH = os.getenv("CHROMA_DB_PATH", "./chroma_db")
COLLECTION_NAME = "knowledge_base"

def run_ingestion(reset: bool = False):
    """
    Run the full ingestion pipeline.
    reset=True clears the database first (use when re-indexing changed docs).
    """
    if reset and os.path.exists(CHROMA_PATH):
        shutil.rmtree(CHROMA_PATH)
        print("🗑️  Cleared existing ChromaDB")

    print("\n📥 Step 1: Loading documents...")
    documents = load_documents(DOCS_FOLDER)

    print("\n✂️  Step 2: Chunking...")
    chunks = chunk_documents(
        documents,
        chunk_size=int(os.getenv("CHUNK_SIZE", 512)),
        chunk_overlap=int(os.getenv("CHUNK_OVERLAP", 50)),
    )

    print("\n🔢 Step 3: Loading embedding model...")
    embeddings = get_embedding_model()

    print("\n🗄️  Step 4: Storing in ChromaDB...")
    vectordb = Chroma.from_documents(
        documents=chunks,
        embedding=embeddings,
        persist_directory=CHROMA_PATH,
        collection_name=COLLECTION_NAME,
    )

    count = vectordb._collection.count()
    print(f"\n✅ Ingestion complete! {count} chunks stored in ChromaDB")
    print(f"   Database path: {CHROMA_PATH}")
    return vectordb


if __name__ == "__main__":
    import sys
    reset = "--reset" in sys.argv
    run_ingestion(reset=reset)

Implementation Step 3
Retriever + RAG Chain
The core query engine — retrieves relevant chunks from ChromaDB then generates a grounded answer via Groq.
retrieval/retriever.py
"""ChromaDB retrieval wrapper with metadata-enriched results."""
import os
from langchain_community.vectorstores import Chroma
from langchain.schema import Document
from ingestion.embedder import get_embedding_model
from dotenv import load_dotenv

load_dotenv()

class KBRetriever:
    def __init__(self):
        self.embeddings = get_embedding_model()
        self.vectordb = Chroma(
            persist_directory=os.getenv("CHROMA_DB_PATH", "./chroma_db"),
            embedding_function=self.embeddings,
            collection_name="knowledge_base",
        )
        self.top_k = int(os.getenv("TOP_K_RESULTS", 5))

    def retrieve(self, query: str, k: int | None = None) -> list[Document]:
        """Retrieve top-K most relevant chunks for a query."""
        k = k or self.top_k
        results = self.vectordb.similarity_search_with_score(query, k=k)

        # Add similarity score to metadata (useful for debugging)
        enriched = []
        for doc, score in results:
            doc.metadata["similarity_score"] = round(1 - score, 3)  # convert distance → similarity
            enriched.append(doc)
        return enriched

    def get_collection_stats(self) -> dict:
        count = self.vectordb._collection.count()
        return {"total_chunks": count, "collection": "knowledge_base"}
generation/prompts.py
"""
RAG system prompts. The grounding instruction is the most critical part.
Without it, the LLM will hallucinate answers from its training data.
"""

RAG_SYSTEM_PROMPT = """You are a helpful knowledge base assistant for TechCorp.

CRITICAL RULES:
1. ONLY use information from the provided context to answer questions.
2. If the context does not contain the answer, say EXACTLY:
   "I don't have information about that in the knowledge base. Please contact support@techcorp.com for assistance."
3. Do NOT use your training data or general knowledge to fill gaps.
4. Always cite which document(s) your answer comes from.
5. Keep answers clear and concise — use bullet points for multi-step processes.

RESPONSE FORMAT:
- Answer the question directly
- If listing steps, use numbered format
- End with: "Source: [document name]" on a new line

TONE: Professional, helpful, and precise."""

def build_rag_prompt(context_chunks: list, query: str) -> str:
    """Build the user message with retrieved context injected."""
    context_parts = []
    for i, chunk in enumerate(context_chunks, 1):
        source = chunk.metadata.get("source_file", "unknown")
        score = chunk.metadata.get("similarity_score", "N/A")
        context_parts.append(
            f"[Context {i} | Source: {source} | Relevance: {score}]\n{chunk.page_content}"
        )

    context_text = "\n\n".join(context_parts)

    return f"""Use the following context to answer the question.

CONTEXT:
{context_text}

QUESTION: {query}

Remember: Only answer using the context above. If the answer isn't there, say so."""
generation/rag_chain.py
"""Full RAG chain: retrieve relevant chunks → generate grounded answer."""
import os, time
from groq import Groq
from dotenv import load_dotenv
from retrieval.retriever import KBRetriever
from generation.prompts import RAG_SYSTEM_PROMPT, build_rag_prompt

load_dotenv()

class RAGChain:
    def __init__(self):
        self.retriever = KBRetriever()
        self.llm = Groq(api_key=os.getenv("GROQ_API_KEY"))
        self.model = "llama3-70b-8192"

    def query(self, question: str, top_k: int = 5) -> dict:
        """
        Full RAG query pipeline.
        Returns: answer, source chunks, timing breakdown.
        """
        timings = {}
        start_total = time.time()

        # Step 1: Retrieve relevant chunks
        t0 = time.time()
        chunks = self.retriever.retrieve(question, k=top_k)
        timings["retrieval_ms"] = int((time.time() - t0) * 1000)

        # Step 2: Build the context prompt
        user_message = build_rag_prompt(chunks, question)

        # Step 3: Generate answer from LLM
        t0 = time.time()
        response = self.llm.chat.completions.create(
            model=self.model,
            messages=[
                {"role": "system", "content": RAG_SYSTEM_PROMPT},
                {"role": "user", "content": user_message},
            ],
            temperature=0.1,  # Low temp — we want grounded factual answers
            max_tokens=600,
        )
        timings["generation_ms"] = int((time.time() - t0) * 1000)
        timings["total_ms"] = int((time.time() - start_total) * 1000)

        answer = response.choices[0].message.content

        return {
            "answer": answer,
            "source_chunks": [
                {
                    "content": c.page_content,
                    "source_file": c.metadata.get("source_file"),
                    "similarity_score": c.metadata.get("similarity_score"),
                    "chunk_index": c.metadata.get("chunk_index"),
                }
                for c in chunks
            ],
            "tokens_used": response.usage.total_tokens,
            "timings": timings,
        }

def run_ragas_eval():
    print("\n🧪 RAGAS EVALUATION")
    print("Loading test questions...")

    with open("evaluation/test_questions.json") as f:
        test_cases = json.load(f)

    rag = RAGChain()

    # Build RAGAS dataset structure
    questions, answers, contexts, ground_truths = [], [], [], []

    print(f"Running {len(test_cases)} queries...")
    for i, case in enumerate(test_cases, 1):
        print(f"  [{i}/{len(test_cases)}] {case['question'][:50]}...")
        result = rag.query(case["question"])

        questions.append(case["question"])
        answers.append(result["answer"])
        contexts.append([c["content"] for c in result["source_chunks"]])
        ground_truths.append(case["ground_truth"])

    dataset = Dataset.from_dict({
        "question": questions,
        "answer": answers,
        "contexts": contexts,
        "ground_truth": ground_truths,
    })

    # Set RAGAS to use Groq (free) instead of OpenAI
    llm = ChatGroq(model="llama3-70b-8192")
    embeddings = HuggingFaceEmbeddings(model_name="all-MiniLM-L6-v2")

    print("\n📊 Running RAGAS metrics...")
    scores = evaluate(
        dataset=dataset,
        metrics=[faithfulness, answer_relevancy, context_precision, context_recall],
        llm=llm,
        embeddings=embeddings,
    )

    print(f"""

📊 RAGAS EVALUATION RESULTS

Project 2 Interview Guide
RAG is the most common AI engineering interview topic. These are the exact questions you will be asked. Expand each one for a full model answer.
🎯
The three things interviewers are really testing(1) Do you understand WHY RAG works — the grounding mechanism, not just the code? (2) Do you have real evaluation metrics to back up your claims? (3) Can you explain where RAG fails and what you'd do about it? Nail these three and you'll stand out from 90% of candidates.
Explain RAG from scratch to someone who doesn't know what it is. Most likely first question ▼
Model answer:

"The core problem RAG solves is that LLMs have a knowledge cutoff and can hallucinate when asked about private company data they've never seen. The solution is: instead of asking the LLM from memory, you first retrieve the relevant documents, then ask the LLM to answer based only on those documents.

Concretely, there are two phases. First, indexing — you take all your company documents, split them into small chunks of about 500 tokens each, convert each chunk into a numerical vector called an embedding that captures its meaning, and store those vectors in a vector database like ChromaDB. This runs once when documents are created or updated.

Second, querying — when a user asks a question, you convert that question into the same kind of vector, find the 5 chunks in your database whose vectors are most similar to the question vector, and pass those chunks to the LLM as context. Crucially, you instruct the LLM: 'only answer using the information I've given you — if it's not in here, say you don't know.' This eliminates hallucination because the LLM is just synthesising and formatting information you've already verified."
What chunk size did you choose and how did you decide? Very common — must have real numbers ▼
"I treated this as an experiment. I ran my RAGAS evaluation with three chunk sizes — 256, 512, and 1024 tokens — keeping everything else constant, and measured context precision and faithfulness for each.

256 tokens gave high precision — the retrieved chunks were very targeted — but hurt recall because answers sometimes span more context than 256 tokens provide. 1024 tokens had the opposite problem: more context per chunk but lower precision, so I was retrieving chunks that were only tangentially related. 512 tokens with 50-token overlap gave the best balance on my test set, scoring 0.87 faithfulness and 0.83 context precision.

The overlap is important — without it, an answer that happens to fall on a chunk boundary gets split and may not be retrieved correctly. 50 tokens is roughly one paragraph, which is usually enough to bridge any boundary."
What is the difference between faithfulness and answer relevance in RAGAS? Tests evaluation depth — many candidates can't answer this ▼
"They measure completely different failure modes. Faithfulness asks: does the answer only contain information that appears in the retrieved context? A faithfulness score of 1.0 means every fact in the answer is traceable to a retrieved chunk — no hallucination. A low faithfulness score means the LLM is adding information from its training data that wasn't in the context you provided.

Answer relevance asks: does the answer actually address the question that was asked? You can have a perfectly faithful answer — one that only uses the retrieved context — but that doesn't actually answer the user's question. For example, if someone asks 'how long does a refund take?' and the system retrieves chunks about the return process and gives a faithful answer about how to ship items back — that's high faithfulness but low answer relevance.

Both need to be high for a production RAG system to work. Faithfulness catches hallucination; answer relevance catches retrieval quality issues."
When does RAG fail and how do you fix it? Tests production thinking ▼
"Five failure modes, each with a different fix:

1. Bad chunking — the answer spans two chunks and neither chunk alone has enough to answer. Fix: increase chunk size or overlap, or use parent-child chunking (which I implement in Project 3).

2. Vocabulary mismatch — the user says 'broken' but the document says 'malfunction'. Semantic search should handle this, but sometimes doesn't. Fix: add BM25 keyword search alongside semantic search — hybrid retrieval, which is exactly what Project 3 is about.

3. The answer isn't in the knowledge base at all — the LLM hallucinates to fill the gap. Fix: the system prompt must explicitly say 'if the answer is not in the context, say you don't know' — and you must test this case in your evaluation set.

4. Top-K too low — the relevant chunk exists but ranks 6th or 7th. Fix: retrieve more candidates (K=20), then use a re-ranker to select the best 5 — this is the re-ranking technique in Project 3.

5. Stale knowledge base — documents updated but the index wasn't rebuilt. Fix: trigger re-ingestion via API when documents change, and track document version hashes to only re-embed changed files."
Why did you use sentence-transformers instead of OpenAI embeddings? Tests cost and architecture thinking ▼
"Three reasons. First, cost: at $0.0001 per 1K tokens, embedding a knowledge base of 10,000 document chunks would cost about $0.50 once, and every user query would add embedding cost. sentence-transformers runs locally for free — the only cost is compute time, which on a Mac M5 is about 5ms per chunk.

Second, data privacy: using OpenAI embeddings means every document chunk in your knowledge base is sent to OpenAI's servers. For enterprise customers with confidential internal documents, this can be a compliance blocker. Local embeddings keep everything on-premises.

Third, quality tradeoff is acceptable: all-MiniLM-L6-v2 is about 90% as good as OpenAI's text-embedding-3-small on most semantic similarity benchmarks, and it's significantly better for domain-specific fine-tuning if you want to go there later. For most enterprise RAG use cases, the quality difference isn't worth the cost and privacy tradeoffs."
How would you scale this to 100,000 documents? Tests scalability thinking ▼
"ChromaDB would still work at that scale — it handles millions of vectors efficiently — but I'd make several architectural changes.

First, asynchronous ingestion: instead of blocking while documents are processed, I'd use a message queue like AWS SQS. When a document is uploaded to S3, an event triggers a Lambda function that runs the ingestion pipeline. This decouples document upload from indexing latency.

Second, incremental re-indexing: track a content hash for each document. When a document changes, only re-embed that document, not the entire corpus. At 100K documents, full re-indexing could take hours.

Third, vector database upgrade: for production scale with concurrent users, I'd migrate from local ChromaDB to Pinecone or AWS OpenSearch with vector support. These are horizontally scalable and have built-in replication.

Fourth, query caching: the top 20% of questions account for 80% of traffic in enterprise KB systems. I'd cache the (question_embedding, answer) pairs in Redis with a 1-hour TTL to avoid redundant retrieval and LLM calls."
