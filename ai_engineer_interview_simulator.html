What separates a hire from a no-hire at senior level
Start with clarifying questions, not solutions
Before touching a design: ask about scale (100 users? 10M?), latency requirements (real-time? batch?), accuracy vs. cost trade-off, and whether there's existing infrastructure. Juniors jump to "I'd use RAG with Pinecone." Seniors ask "what's the query volume and what does a failed answer cost the business?"
Name failure modes before being asked
For every component you propose, say what breaks it. "I'd use a ReAct agent — the failure mode is infinite loops, which I'd handle with a max_iterations guard and soft-limit prompt injection." Interviewers probe failure modes anyway. Getting there first shows experience.
Quantify everything you can
"RAG reduces hallucinations" is weak. "RAG on a 10k-document knowledge base with reranking reduces hallucination rate from ~30% (vanilla LLM) to ~8% in our evals, at a cost of ~200ms additional latency per query" is strong. Even rough estimates show you think in production terms.
Propose the simple thing first, then iterate
Start with the MVP architecture, then say "I'd evolve this if X happens." This signals engineering judgment — you know when a complex solution is premature. A candidate who jumps straight to a 5-agent hierarchical system with custom fine-tuning for a simple chatbot fails the judgment test.
Evaluation is always part of the answer
No AI system design is complete without saying how you'd measure it. "I'd set up an eval pipeline with 200 labeled examples, measuring retrieval recall@3 and answer faithfulness using RAGAS, gated to CI so any prompt change runs the eval before deployment." If you don't mention evals, the interviewer will ask — and now you look reactive instead of proactive.
The one thing that ends interviews early: confidently stating something technically wrong without hedging. "I'm not 100% sure on the exact numbers but my understanding is..." is far better than a confident wrong answer. Interviewers know you can look things up — they're testing how you reason under uncertainty.
